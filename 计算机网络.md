
# 一条http请求，发生了什么
### 第一阶段：应用层的准备与解析

这是浏览器在发起网络请求前的准备工作。

1. **浏览器缓存检查 (闪光点)**:
    
    - 浏览器首先会检查自身的 **HTTP 缓存**。它会根据请求头中的 `Cache-Control` 和 `Expires` 字段判断资源是否命中了**强缓存**。如果命中，浏览器会直接从本地磁盘或内存中读取资源，不会发出任何网络请求，这是最快的情况。        
    - 如果强缓存未命中，浏览器会检查是否有 `ETag` 或 `Last-Modified` 字段，发起**协商缓存**请求。服务器会根据这些字段判断资源是否有更新，如果未更新则返回 `304 Not Found` 状态码，浏览器同样从本地加载资源。
        
2. **URL 解析**:    
    - 如果缓存未命中，浏览器开始解析 URL，提取出协议（HTTP/HTTPS）、域名、端口、路径等信息。        
    - 根据这些信息，浏览器开始构建一个 HTTP 请求报文，包括请求行、请求头（包含 `User-Agent`、`Accept` 等信息以及缓存相关的字段）和空的请求体（对于GET请求）。
        
3. **DNS 解析 (域名到IP的翻译)**:
    
    - 为了将域名转换为服务器的 IP 地址，浏览器会按照以下顺序查找：**浏览器DNS缓存 → 操作系统DNS缓存 → 本地`hosts`文件 → 路由器缓存**。        
    - 如果本地都找不到，浏览器会向本地DNS服务器发起递归查询请求。  
    - **CDN (闪光点)**: 对于大型网站，DNS 服务器通常不会直接返回源服务器 IP，而是通过 CNAME 记录返回一个 **CDN 网络的域名**。CDN 的 DNS 服务器会根据用户的地理位置，返回一个离用户最近的 **CDN 边缘节点的 IP 地址**，后续的静态资源请求将直接发往这个节点，大大提高了访问速度。        
    - 最终，客户端获取到目标服务器的 IP 地址。
        

### 第二阶段：网络层的连接与传输

这是数据包在网络中的核心旅程。

4. **建立 TCP 连接**:
    
    - 浏览器利用获取到的 IP 地址和端口号，通过操作系统的 Socket 库，向服务器发起 TCP 连接请求。
        
    - 这个过程会进行经典的 **TCP 三次握手**（SYN -> SYN/ACK -> ACK），以确保双方都具备可靠的收发能力。
        
5. **TLS/SSL 握手 (闪光点 - 针对HTTPS)**:
    
    - 在 TCP 连接建立后，如果 URL 是 HTTPS 协议，还需要进行 **TLS/SSL 的四次握手**来建立安全信道。
        
    - 这个过程包括：交换协议版本、协商加密套件、服务器发送数字证书、客户端验证证书、交换生成本次会话的**对称密钥**。
        
    - 在此之后，所有传输的 HTTP 数据都会被这个对称密钥加密。
        
6. **数据包封装与发送 (OSI模型)**:    
    - **应用层**: 加密后的 HTTP 报文准备就绪。        
    - **传输层**: HTTP 报文被分割并封装成 **TCP 段 (Segment)**，加上源/目标端口号等信息。        
    - **网络层**: TCP 段被封装成 **IP 包 (Packet)**，加上源/目标 IP 地址。        
    - **数据链路层**: IP 包被封装成 **MAC 帧 (Frame)**，加上源/目标 MAC 地址。目标 MAC 地址通过 **ARP 协议**（地址解析协议）在子网内广播查询得到。        
    - **物理层**: 网卡将数字化的 MAC 帧转换为电信号（或光信号），通过网线、光纤等物理介质发送出去。
        
7. **数据包路由**:    
    - **交换机 (二层)**: 在子网内，交换机根据 MAC 地址表，将数据帧精确地转发到目标设备的端口，或者连接到路由器的端口。        
    - **路由器 (三层)**: 数据包每经过一个路由器，路由器会查询其**路由表**，选择最佳路径，将数据包转发到下一个路由器，这个过程不断重复，最终将数据包送达目标服务器所在的网络。        

### 第三阶段：服务器端的处理与响应

数据包到达目的地后，服务器开始工作。

8. **服务器接收与解封装**:    
    - 服务器的网卡接收到电信号，将其转换回 MAC 帧。        
    - 然后数据被层层解封装：**MAC 帧 → IP 包 → TCP 段 → HTTP 报文**。每一层都会进行校验（如 MAC 地址、IP 地址、端口号、TCP 序列号等）。        
9. **请求处理**:    
    - **负载均衡器 (闪光点)**: 在大型应用中，请求首先到达的可能是一个**负载均衡器（如 Nginx）**。它会根据配置的策略（如轮询、最少连接），将请求转发给后端众多应用服务器中的一台。        
    - Web 服务器（如 Tomcat, Node.js）接收到 HTTP 请求后，会进行相应的业务逻辑处理，这可能包括查询数据库、调用其他微服务、读取文件等。        
    - 处理完成后，服务器会生成一个 HTTP 响应报文，包含状态码（如 `200 OK`）、响应头和响应体（通常是 HTML 内容）。
        
10. **服务器发回响应**:    
    - 响应报文会经历与请求时**完全相同的封装过程**（应用层 → ... → 物理层），然后通过网络路由返回给客户端。        
### 第四阶段：客户端的渲染与展示

浏览器收到响应后的最后一步。
11. **浏览器接收并渲染页面 (闪光点)**:    
    - 浏览器接收到服务器返回的 HTML 响应后，开始执行 **关键渲染路径 (Critical Rendering Path)**：        
        1. **解析HTML生成 DOM 树**: 浏览器自上而下解析 HTML 文档，构建出 DOM（文档对象模型）树。            
        2. **解析CSS生成 CSSOM 树**: 在解析过程中如果遇到 CSS 链接或样式块，会异步下载并解析，构建出 CSSOM（CSS 对象模型）树。            
        3. **构建渲染树 (Render Tree)**: 将 DOM 树和 CSSOM 树结合起来，只包含需要显示的节点及其样式信息，生成渲染树。            
        4. **布局 (Layout/Reflow)**: 根据渲染树，计算出每个节点在屏幕上的精确位置和尺寸。            
        5. **绘制 (Paint)**: 将布局计算好的节点，转换为实际的像素，绘制到屏幕上。            
    - **JS 的影响**: 如果解析过程中遇到 `<script>` 标签（且没有 `async`或`defer`属性），HTML 解析会**暂停**，等待脚本下载并执行完毕后才会继续。如果 JS 操作了 DOM，可能会导致**重排 (Reflow)** 和 **重绘 (Repaint)**，影响页面性能。

# HTTP

解决“如何在客户端和服务器之间更高效、更安全地传输超文本（及其他资源）”这个问题

## HTTP 1.0 

- 核心特性：
	- **无连接**：每发送一个请求，客户端和服务器就会建立一个新的 TCP 连接，请求完成后立即关闭。
	- **无状态**：服务器不保留与客户端通信的任何状态信息。

- 主要问题：
	- **性能瓶颈**：由于每次请求都需要新建和关闭 TCP 连接，而 TCP 连接的建立（三次握手）和关闭（四次挥手）都有开销，这导致了大量的延迟（尤其是对加载包含许多小资源的网页）。

## HTTP 1.1

- 核心改进：
	- **持久连接 (Persistent Connections / Keep-Alive)**：默认开启，客户端和服务器可以<mark class="hltr-red">复用</mark>同一个 TCP 连接来发送和接收多个请求/响应。大大减少了 TCP 连接的建立和关闭开销。
	- **管道化 (Pipelining)**：允许客户端在收到前一个响应之前，发送多个请求。但由于实现复杂性和潜在的乱序问题，实际应用中较少广泛使用。
	- **增加 Host 头字段**：允许在同一个 IP 地址和端口上运行多个虚拟主机，解决了 Web 服务器部署的扩展性问题。
	- **缓存机制 (Cache-Control, ETag)**：引入更丰富的缓存控制策略，减少不必要的数据传输。
	- **断点续传 (Range 请求)**：支持客户端从中断的地方继续下载文件。

- 遗留问题：
	- **队头阻塞 (Head-of-Line Blocking, HOL Blocking)**：虽然持久连接解决了每次建立 TCP 连接的开销，但在一个 TCP 连接中，HTTP/1.1 的请求/响应仍然是串行处理的。如果前面的请求处理较慢或响应较大，会导致后面的请求被阻塞，即使它们可以独立处理。
	- **请求头部冗余**：每次请求都会发送大量的重复头部信息。

## HTTP 2
- 核心改进：
	- **二进制分帧 (Binary Framing Layer)**：HTTP/2 不再是文本协议，而是将所有传输的信息分割成更小的二进制帧，并对这些帧进行多路复用。这使得解析更高效、更健壮。
	- **多路复用 (Multiplexing)**：这是最核心的改进。在**单个 TCP 连接**上，可以同时发送多个请求和接收多个响应，并且各个请求/响应之间互不影响。这彻底解决了 HTTP/1.1 的应用层队头阻塞问题。
	- **服务器推送 (Server Push)**：服务器可以在浏览器请求某个 HTML 页面时，主动推送该页面可能需要的 CSS、JavaScript 或图片等资源，而无需等待浏览器解析 HTML 后再发起请求。
	- **头部压缩 (Header Compression)**：使用 HPACK 算法对 HTTP 头部进行压缩，减少每次请求和响应的数据量。
	- **请求优先级**：允许客户端为不同的流设置优先级，服务器可以根据优先级来决定资源的发送顺序。

- **底层协议：TCP**

- 遗留问题：
	- **TCP 队头阻塞**：虽然 HTTP/2 解决了应用层队头阻塞，但由于它仍然使用 TCP，如果底层的 TCP 连接中某个数据包丢失，TCP 的重传机制会阻塞整个 TCP 连接上的所有数据流，直到该丢失的包被重传成功。这在网络丢包率较高的环境下仍然是性能瓶颈。

## HTTP 3

HTTP/3 旨在解决 HTTP/2 遗留的 TCP 队头阻塞问题，并进一步提升性能和安全性，尤其是在移动网络等不稳定环境中。

它基于 Google 的 **QUIC  (Quick UDP Internet Connections)协议**。

- 核心改进：
	- **基于 QUIC 协议**：这是 HTTP/3 与前两代最本质的区别。HTTP/3 不再使用 TCP 作为传输层协议，而是使用了基于 **UDP** 的 QUIC 协议。
	- **彻底解决 TCP 队头阻塞**：QUIC 协议在 UDP 之上实现了自己的可靠传输机制和流控制。它同样支持多路复用，但其流是独立的，即使某个流的底层 UDP 数据包丢失，也只影响这一个流，不会阻塞其他流的数据传输。
	- **更快的连接建立**：QUIC 结合了 TCP 和 TLS 握手，通常可以在 1-RTT (一次往返时间) 完成连接建立和加密协商。如果客户端之前连接过服务器，甚至可以实现 0-RTT 恢复会话，进一步减少延迟。
	- **更好的连接迁移**：QUIC 连接不再依赖 IP 地址和端口号，而是通过一个 Connection ID 来标识。这意味着客户端可以在 Wi-Fi 和蜂窝网络之间无缝切换，而无需中断或重新建立连接。
	- **内置 TLS 1.3 加密**：QUIC 从设计之初就包含了完整的 TLS 1.3 加密，默认提供了传输安全性，且加密握手更快。

# 常见HTTP状态码

- 5xx 服务器错误
	- **`500 Internal Server Error`**：服务器内部错误。
	    - **作用**：最通用的服务器端错误，表示服务器在执行请求时遇到了一个无法预料的错误，可能是代码逻辑错误、数据库连接问题等。
	- **`502 Bad Gateway`**：网关错误。
	    - **作用**：作为网关或代理的服务器从上游服务器（如应用服务器）接收到无效的响应。通常出现在反向代理（Nginx/Apache）与后端服务通信出现问题时。
	- **`503 Service Unavailable`**：服务不可用。
	    - **作用**：服务器暂时无法处理请求，通常是由于服务器过载或停机维护。
	- **`504 Gateway Timeout`**：网关超时。
	    - **作用**：作为网关或代理的服务器在规定时间内没有从上游服务器得到响应。通常表示后端服务处理时间过长或已崩溃。

# HTTP 请求包含的内容

## 请求行

- 请求方法
	- `GET`：请求获取指定资源
	- `POST`：创建
	- `PUT`：更新
	- `DELETE`：请求删除指定资源
- 请求URL
- HTTP版本号

例如：
`GET /index.html HTTP/1.1`
`POST /api/users HTTP/1.1`

## 请求头

包含各种键值对
- `Host`：请求来源。请求的目标服务器的域名和端口号。**HTTP/1.1 协议中这是必不可少的头部。**
- `User-Agent`：用户代理。发送请求的客户端（如浏览器及其版本、操作系统）的信息。
- `Accept`：接受的内容类型。客户端希望接收的响应数据类型（MIME Type），如 `application/json`, `text/html`。
- `Authorization`：包含客户端的认证凭证（如 Token），用于身份验证。

## 请求体

- **含义**：可选部分，包含客户端要发送给服务器的实际数据。
- **何时使用**：主要用于 `POST`, `PUT`, `PATCH` 等方法，这些方法需要向服务器提交数据以创建、更新或部分修改资源。`GET` 和 `HEAD` 方法通常没有请求体。
- **内容类型**：请求体的格式由 `Content-Type` 头部指定，可以是 JSON、XML、表单数据 (form-urlencoded)、文件等。
```JSON
{
    "username": "john.doe",
    "password": "securepassword123",
    "email": "john.doe@example.com"
}
```

## **HTTP 请求的完整结构示意**
```TEXT
<请求行>
<请求头1>: <值1>
<请求头2>: <值2>
...
<空行>
<请求体> (可选)
```

# HTTP 和 HTTPS

HTTPS 并不是一个新的协议，而是 **HTTP 协议与 SSL/TLS (Secure Sockets Layer / Transport Layer Security) 协议的组合**。

SSL/TLS 协议在 HTTP 和 TCP 层之间提供了一个安全层，对 HTTP 传输的数据进行加密、认证和完整性保护。


| 特性           | HTTP                                      | HTTPS                                     |
| :------------- | :---------------------------------------- | :---------------------------------------- |
| **安全性**     | **不安全**：数据明文传输，易被窃听、篡改。 | **安全**：通过 SSL/TLS 进行数据加密、身份认证、数据完整性保护。 |
| **加密机制**   | 无                                        | **数据加密**：通过对称加密算法对传输数据加密。<br>**身份验证**：通过数字证书验证服务器身份。<br>**数据完整性**：通过摘要算法（如 SHA256）验证数据未被篡改。 |
| **默认端口**   | `80`                                      | `443`                                     |
| **连接过程**   | 直接与服务器建立 TCP 连接后发送 HTTP 请求。 | 先与服务器进行 **SSL/TLS 握手**建立安全连接，再发送 HTTP 请求。 |
| **证书需求**   | 不需要                                    | **需要数字证书**（通常由权威的证书颁发机构 CA 签发）。 |
| **资源消耗**   | 较少（无加密解密开销）                    | 较多（加密解密、握手过程带来计算开销和网络延迟），但现代硬件和协议优化已大幅降低。 |
| **SEO 影响**   | 无正面影响，甚至可能被搜索引擎降权。      | **有积极影响**：Google 等搜索引擎会优先索引和排名 HTTPS 网站。 |

## HTTPS 的安全机制详解（SSL/TLS 的核心作用）

SSL/TLS 协议主要通过以下三个方面来保证通信安全：

1. **数据加密 (Encryption)**：
    <mark class="hltr-green">保密性</mark>
    - **对称加密**：用于实际传输数据。加密和解密使用同一个密钥。效率高，但密钥分发是挑战。
    - **非对称加密**：用于密钥协商（交换对称加密的密钥）。加密和解密使用一对公钥和私钥。公钥加密的数据只能用私钥解密，私钥加密的数据只能用公钥解密。安全性高，但效率低。
    - **HTTPS 结合使用**：在 SSL/TLS 握手阶段，客户端和服务器通过**非对称加密**来安全地协商出用于后续数据传输的**对称加密密钥**。一旦对称密钥确定，后续的 HTTP 数据传输就全部使用**对称加密**进行，兼顾了安全性和效率。
	    - 详解：
			客户端和服务器在握手阶段，会利用非对称加密算法来交换对称密钥。服务器会发送自己的公钥给客户端，客户端生成一个随机的对称密钥，然后用服务器的公钥加密这个对称密钥，再发送回服务器。服务器收到后，用自己的私钥解密，从而安全地获取到对称密钥。客户端和服务器在非对称加密协商好对称密钥后，后续的所有应用层数据传输都将使用这个对称密钥进行加密和解密。

2. **身份认证 (Authentication)**：

    - **数字证书 (Digital Certificate)**：服务器会向客户端出示其数字证书。证书包含了服务器的公钥、服务器的身份信息（如域名）、以及证书颁发机构（CA）的签名。
    - **CA (Certificate Authority)**：是一个权威的第三方机构。它用自己的私钥对服务器的证书进行签名，证明这个服务器的身份是可信的。
    - **客户端验证**：客户端收到证书后，会使用操作系统内置的 CA 公钥来验证证书的合法性、颁发机构、有效期等。如果验证失败，浏览器会发出安全警告。这可以防止“中间人攻击”（Man-in-the-Middle Attack），确保客户端连接的是真正的服务器。

3. **数据完整性 (Data Integrity)**：
    
    - **哈希算法 (Hash Algorithm)**：在发送数据之前，发送方会对数据计算一个哈希值（消息摘要），然后连同数据一起发送。
    - **接收方验证**：接收方收到数据后，会用相同的哈希算法重新计算一遍哈希值，并与收到的哈希值进行比对。如果两者一致，则表明数据在传输过程中未被篡改。

## SSL/TLS 握手

三次握手解决的是**连接问题**，SSL/TLS 握手解决的是**安全问题**

### 三次握手

<mark class="hltr-green">传输层</mark>

主要目标是**建立可靠的传输连接**。它确保了通信双方（客户端和服务器）的收发能力都是正常的。

- **SYN** (同步) ：客户端发送同步报文，询问服务器是否可以建立连接。
	确认客户端的发送正常；
- **SYN + ACK** (同步 + 确认) ：服务器收到后，发送确认报文，并告知自己也想建立连接。
	确认服务器的接收、发送功能正常；
- **ACK** (确认) ：客户端收到后，发送确认报文，表示连接正式建立。
	确认客户端接收功能正常。

1. 客户端向服务器发送 SYN 包，表示请求建立连接，此时客户端进入 SYN_SENT 状态。
2. 服务器接收到 SYN 包后，回应一个 SYN-ACK 包，表示同意建立连接，此时服务器进入 SYN-RECEIVED 状态。
3. 客户端接收到 SYN-ACK 包后，向服务器回应一个 ACK 包，表示确认建立连接，此时客户端进入 ESTABLISHED 状态。服务器收到 ACK 包后也进入 ESTABLISHED 状态。

### SSL/TLS握手

<mark class="hltr-green">应用层</mark>

主要目标是**建立安全的加密通道**。它通过一系列复杂的步骤，完成身份认证、协商加密算法和交换会话密钥等任务。

- **Finish**：双方都发送一个握手结束的消息，表示密钥协商完成，可以开始加密通信。


### 四次挥手

TCP的四次挥手是用于安全关闭一个已建立的连接过程，它确保双方都能完成数据传输并安全地释放连接资源。
 
1. **第一次挥手** (FIN → ACK) : 客户端主动关闭连接, 发送 FIN 包, 进入 FIN_WAIT_1 状态。服务器收到 FIN 后, 表示不再接收数据, 但仍可能继续发送数据。
2. **第二次挥手** (ACK) : 服务器发送 ACK 包, 确认已收到 FIN, 此时服务器进入 CLOSE_WAIT 状态, 客户端进入 FIN_WAIT_2 状态。
3. **第三次挥手** (FIN → ACK) : 服务器完成所有数据传输后, 发送 FIN 包, 进入 LAST_ACK 状态。客户端收到 FIN 后, 准备关闭连接。
4.  **第四次挥手** (ACK) : 客户端发送最后一个 ACK 包, 进入 TIME_WAIT 状态, 等待可能迟到的 FIN 包。服务器收到 ACK 后, 关闭连接, 进入 CLOSED 状态。客户端在 TIME_WAIT 计时结束后 (2MSL), 正式关闭连接。

- 为什么需要四次？
	确保数据完整性。tcp是全双工，所以一个 FIN-ACK 对，就代表一方结束数据传输，一共需要两对。
- 三次可以吗？
	可以，如果客户端要结束的时候，服务器也没有数据发送了，FIN和ACK就可以一起发。
- 为什么要有 TIME_WAIT 状态？
	1. 确保最后一个 ACK 报文能被服务器接收。如果ACK丢失，另一方重发FIN。就会在TIME_WAIT 状态保持一段时间，重发。
	2. 防止已失效的、旧连接的报文段，出现在后续的新连接中，从而避免数据混淆。

	在正常情况下，服务器收到客户端的确认报文后会立即关闭连接，但是由于网络的不稳定性，可能导致最后一个 ACK 报文丢失。如果客户端立即关闭连接，那么服务器无法收到确认，会认为客户端并没有收到服务器的 FIN 报文，从而会重新发送 FIN 报文。而客户端此时已经关闭连接，无法接收到服务器的重传 FIN 报文，导致服务器一直等待连接关闭，造成资源浪费。

	为了避免这种情况，客户端需要等待 2 倍的 MSL 时间，确保服务器收到了确认报文并关闭了连接。这样即使服务器的重传 FIN 报文到达客户端，客户端也可以忽略它，因为此时客户端已经关闭连接。等待2倍的 MSL 时间可以确保网络中的所有报文都已经被丢弃，避免了潜在的连接问题。

- 为什么TIME_WAIT 是 2MSL 的时间？
	如果服务器没有收到最后一个ACK，会触发超时重传FIN，当客户端接收到FIN后，会重发ACK给服务器，一来一2MSL。MSL(报文最大生存时间)

- 除了四次挥手，还可以使用RST（Rest）报文断开连接
	用于在异常情况下，强制终止连接
# TCP

## 可靠性传输

- **序列号（Sequence Number）**：每个 TCP 数据包（段，segment）都有一个唯一的序列号，用来标识数据段在整个数据流中的位置。接收方可以根据序列号对乱序到达的数据进行重新排序。
- **确认应答（Acknowledgement Number）**：接收方收到数据后，会发送一个确认应答（ACK）报文，其中的确认号告诉发送方，它希望接收下一个数据段的序列号。这表示这个序列号之前的所有数据都已成功接收。
- **超时重传（Retransmission）**：发送方在发送数据后会启动一个计时器。如果在设定的时间内没有收到对应的 ACK，就认为数据丢失，会重新发送。
	- 比如，发送1，2，3，4， ACK5就表示前四个收到了。
	- 如果只收到了1，2，4 ，那只能ACK3。
	- 超时重传时间RTO，要根据RTT（Round Trip Time）也就是来回一趟的时间来设定。先采样一些RTT，再加权平均

- **校验和（Checksum）**：数据包中包含一个校验和，接收方通过计算并对比校验和来检测数据在传输过程中是否损坏。

## 流量控制

TCP 利用 **滑动窗口（Sliding Window）** 机制来实现流量控制。

- **工作原理**：发送方和接收方各自维护一个窗口。接收方在 ACK 报文中，会通过**窗口大小（Window Size）** 字段告知发送方自己当前还能接收多少字节的数据。
    
- **动态调整**：发送方会根据这个窗口大小来调整自己的发送速率，确保发送的数据量不会超过接收方的处理能力。这样可以防止接收方的缓冲区溢出，导致丢包。

- TCP 的滑动窗口大小是由**接收方**决定的，并包含在每个 ACK 报文的 **“窗口大小”** 字段中。

- **零窗口**：如果接收方的缓冲区满了，它会把窗口大小设置为 **0**，以此来告诉发送方“请停止发送数据”。

## 拥塞控制

是为了防止**整个网络**的拥塞，它与流量控制的区别在于，后者关注的是点对点（发送方和接收方）的通信，而前者关注的是整个网络。TCP 通过以下四种算法协同工作：

1. **慢启动（Slow Start）**：连接建立之初，发送方以一个非常小的拥塞窗口开始，指数级地增加发送速率，直到达到某个阈值。
2. **拥塞避免（Congestion Avoidance）**：当发送速率达到慢启动阈值后，开始线性地增加拥塞窗口，以更平缓的方式探测网络的承载能力。
3. **快重传（Fast Retransmit）**：发送方如果连续收到**三次**相同的 ACK（即“三连 ACK”），就认为数据包丢失，不等超时就立即重传。
4. **快恢复（Fast Recovery）**：在快重传之后，TCP 并不立即进入慢启动，而是以更谨慎的方式线性增加拥塞窗口，避免网络拥塞加剧。

- IP 是网络层的核心协议
	它解决了两个最基础、最核心的问题：**寻址（Addressing）** 和 **路由（Routing）**
	1. **寻址：** IP 协议通过为每个设备分配一个唯一的 IP 地址
	2. **路由：** 当数据包需要从一个网络发送到另一个网络时，IP 协议负责决定最佳的路径

- **IP 地址**是逻辑地址，用于在网络层进行寻址和路由。

- **ARP（Address Resolution Protocol）** 将 IP 地址转换为 MAC 地址，它工作在 **网络层** 和 **数据链路层** 之间，主要用于在局域网中确定一个特定 IP 地址对应的物理地址（MAC 地址）。因为最终需要找到 MAC 地址才能跟具体的设备通信。

- **RARP（Reverse Address Resolution Protocol）** 用于将 MAC 地址转换为 IP 地址，比如一些设备启动的时候，需要根据 RARP 来得知分配给它的 ip 是什么。


## Cookie、Session、Token 之间有什么区别？

## Cookie

- HTTP 协议是无状态的，但我们平时访问网站时却感觉是有状态的，比如登录后不需要再次登录。这是为什么？请从网络分层的角度来解释一下。

HTTP 协议本身确实是**无状态的（Stateless）**，这意味着每个 HTTP 请求都是独立的，服务器不会保留关于之前请求的任何信息。例如，当你在浏览器里连续两次请求同一个网页，服务器会把这两个请求视为完全不相关的，即使它们来自同一个用户。

因为**应用层**引入了额外的机制来弥补 HTTP 的无状态性。最常见的技术就是 **Cookie 和 Session**。

Cookie 是存储在用户浏览器端的一个小型数据文件，用于跟踪和保存用户的状态信息。

主要用于保持用户登录状态、跟踪用户行为、存储用户偏好等。

浏览器接收到Cookies后会保存起来，之后，你的浏览器向这个网站发送的**每一个 HTTP 请求**，都会自动将 Cookies 放在 **HTTP 请求头**中，一起发送给服务器。

**服务器**收到请求后，会读取请求头里的会话 ID。然后，它根据这个 ID 去查找自己**内部存储的会话（Session）信息**。这个 Session 里包含了用户的登录状态、购物车内容等数据。

# 什么是 SSE？为什么 AI 应用接口选择 SSE 而不是 WebSocket？

SSE 服务器发送事件（Server-Sent Events）是一种用于从服务器到客户端的 单向 数据传输技术，基于 HTTP 协议实现。连接一旦建立，服务器就可以持续地将新的信息流式传输给客户端，不需要客户端重复发起请求，非常适合需要服务器主动更新数据的场景，比如我们 AI 应用中逐步显示大模型生成的回复，实现类似打字机的效果。

不选择 WebSocket 的原因：

实现成本：SSE 基于标准的 HTTP 协议，实现相对简单，前端使用 JavaScript 的 EventSource API 即可轻松接收。WebSocket 则是一个独立的、更复杂的双向通信协议，需要专门的连接管理和握手过程。
通信模式：AI 应用的流式响应通常是服务器向客户端的单向数据推送，跟 SSE 的单向通信模型完美契合。WebSocket 提供的是全双工双向通信，对于 AI 流式输出这类场景，双向能力并不必要，反而增加了复杂性。
内置机制：SSE 规范包含自动重连和事件 ID（lastEventId）机制。如果网络连接意外中断，浏览器可以自动尝试重连，还可以通过上次接收到的事件 ID 告诉服务器从什么地方继续，对于保证数据流的连续性很有用。这些在 WebSocket 需要开发者自行实现。
资源消耗：一般来说，SSE 比 WebSocket 更轻量，WebSocket 需要建立和维护一个持久的 TCP 连接。
WebSocket 在需要真正实时双向交互（比如在线聊天室、协同编辑工具）的场景下是更优的选择，但对于 AI 应用的场景，SSE 更加简单、高效，比 WebSocket 更合适。

# RPC

**RPC（Remote Procedure Call 是一种 远程调用协议**，允许一台计算机通过网络调用另一台计算机上的服务或方法。它可以让开发人员像调用本地方法一样调用远程方法，将网络通信细节封装起来，提高了分布式系统中各个模块之间的耦合性。

目前常见的 **RPC 框架**有：

1. Dubbo：阿里巴巴开源的分布式 **RPC 框架**，支持多种协议和负载均衡策略。
2. gRPC：Google 开源的高性能 **RPC 框架**，支持多种语言。
3. Thrift：Facebook 开源的跨语言 **RPC 框架**，支持多种传输协议和数据编解码方式。
4. Spring Cloud Netflix：Spring Cloud 的子项目之一，提供了基于 Netflix OSS 开源组件的微服务解决方案，包括服务发现、负载均衡、熔断器等功能。
