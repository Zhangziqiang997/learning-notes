# 线程的生命周期
- New（初始状态）：线程对象创建后，但未调用 start() 方法。
- Runnable（可运行状态）：调用 start() 方法后，线程进入就绪状态，等待 CPU 调度。
- Blocked（阻塞状态）：线程试图获取一个对象锁而被阻塞。
- Waiting（等待状态）：线程进入等待状态，需要被显式唤醒才能继续执行。
- Timed Waiting（含等待时间的等待状态）：线程进入等待状态，但指定了等待时间，超时后会被唤醒。
- Terminated（终止状态）：线程执行完成或因异常退出。

# wait() 和 sleep() 的区别？

`wait()` 和 `sleep()` 都是用于暂停线程的操作，但它们有明显的区别（先说面试官最关心的）：
1）**使用要求不同**：
- `wait()` 方法必须在synchronized **同步块或同步方法内调用**，否则会抛出 `IllegalMonitorStateException`。这是因为 `wait()` 依赖于对象锁来管理线程的等待和唤醒机制。调用后，当前线程会释放它持有的对象锁，并进入等待状态。
	- `wait()`的职责是 **“临时释放你正持有的那把锁，并去休息室等待”**，所以前提是你必须得先持有这把锁。
- `sleep()` 方法**可以在任何上下文中调用，不需要获取对象锁**。调用后，线程会进入休眠状态，但不会释放它持有的任何锁。

2）**方法所属类不同**：
- **`wait()`**：属于 `Object` 类。
- **`sleep()`**：属于 `Thread` 类。

3）**恢复方式不同**：
- **`wait()`**：需要被其他线程通过 `notify()` 或 `notifyAll()` 显式唤醒，或被 `wait(long timeout)` 的超时参数唤醒。
- **`sleep()`**：在指定时间后自动恢复运行，或通过抛出 `InterruptedException` 恢复。

4）**用途不同**：
- **`wait()`**：通常用于线程间通信，配合 `notify()` 或 `notifyAll()` 来实现线程的协调工作。
- **`sleep()`**：用于让线程暂停执行一段时间，通常用于控制线程的执行频率或模拟延时。

**常见错误**：
- **误用 `sleep()`**：有时开发者会错误地使用 `sleep()` 进行线程间通信，但是 `sleep()` **不释放锁**，可能会导致其他线程无法进入同步块，造成线程饥饿或死锁。
- **忽略中断**：`sleep()` 可能抛出 `InterruptedException`，如果不正确处理中断信号，可能会导致线程提前退出或错误行为。

<mark class="hltr-red">如果 `wait()` 不在 synchronized 中调用，会出现“丢失的唤醒” (Lost Wake-up)</mark>
```java
// 这是一个错误！！！的伪代码
// 消费者线程
if (queue.isEmpty()) {
    // 危险时刻：线程可能在这里被切换走！
    queue.wait(); // 准备等待
}
// ...取出元素

// 生产者线程
queue.add(element);
queue.notify(); // 唤醒可能在等待的消费者
```
**灾难性的执行顺序可能如下：**
1. **消费者**检查`queue.isEmpty()`，发现队列是空的，判断自己需要等待。    
2. **CPU时间片切换！** 就在消费者准备调用`wait()`**之前**，操作系统暂停了消费者线程。    
3. **生产者**开始执行。它成功地向队列中添加了一个元素 (`queue.add`)。    
4. 生产者心想：“我刚加了东西，可能有人在等，我得发个通知。” 于是它调用了`queue.notify()`。    
5. **唤醒信号丢失了！** 因为此时消费者线程**还没有**进入`wait()`状态，所以这个`notify()`信号就像是对空谷呐喊，没有任何线程会收到它。    
6. **CPU时间片切换回来！** 消费者线程从上次暂停的地方继续执行，它执行了`queue.wait()`方法，进入了等待状态。    
**最终结果：** 消费者永远地等待下去，因为它在等待一个**已经发生过并且不会再来**的唤醒信号。而生产者可能也因为队列已满而等待消费者消费，最终导致整个系统**死锁**。
**`synchronized`如何解决这个问题？**
通过将条件检查和`wait()`调用包裹在同一个同步块中，我们保证了它们是一个**原子操作**。
```java
// 正确的写法
synchronized (queue) {
    while (queue.isEmpty()) { // 使用while防止“假唤醒”
        queue.wait();
    }
    // ...取出元素
}
```


# notify() 和 notifyAll()
都是`Object`类的成员方法
都必须与`synchronized`关键字配合使用
- **`.notify()` 作用** ：从对象的“等待队列”中，**随机唤醒一个**正在等待的线程。
- **`.notifyAll()` 作用** ：唤醒对象的“等待队列”中**所有**正在等待的线程。

<mark class="hltr-red"> `notify()` vs `notifyAll()`：为何强烈推荐 `notifyAll()`？</mark>

虽然`notifyAll()`会唤醒所有线程，看起来效率可能更低，但它**远比`notify()`安全**。

**强烈推荐使用`notifyAll()`的原因，是为了避免“信号丢失”或“死锁”的风险。**
**设想一个复杂的场景：**
- 有一个缓冲区，我们有多个**生产者线程**和多个**消费者线程**。    
- 所有线程都在同一个`lock`对象的等待队列中。    
    - 生产者在等待的条件是 `!isFull` (缓冲区不满)。        
    - 消费者在等待的条件是 `!isEmpty` (缓冲区不空)。        

**如果使用`notify()`，可能会发生什么？**
1. 一个**消费者**成功消费了一个元素，现在缓冲区从“满”变成了“不满”。    
2. 它调用`lock.notify()`，希望能唤醒一个**生产者**。    
3. 但`notify()`是随机的，它非常不凑巧地唤醒了另一个**消费者**！    
4. 这个被唤醒的消费者检查`while(!isEmpty)`，发现队列还是空的，于是它又继续`wait()`睡了过去。    
5. **唤醒信号被“偷走”并浪费了！** 真正该被唤醒的生产者线程没有收到信号，如果这种情况持续发生，可能导致所有生产者都在等待，而消费者也在互相唤醒，最终整个系统停止工作。    
**`notifyAll()`如何解决这个问题？** 当消费者调用`lock.notifyAll()`时，它会唤醒**所有**等待的线程（包括所有生产者和所有消费者）。
- 被错误唤醒的消费者们，会检查`while(!isEmpty)`，发现条件不满足，于是它们会自觉地回去继续`wait()`。  
- 需要被唤醒的生产者们，会检查`while(!isFull)`，发现条件满足了，于是它们会跳出循环，开始生产。 虽然带来了一些额外的线程上下文切换开销，但它保证了**正确的信号一定能被正确的线程接收到**，程序的**正确性**和**健壮性**得到了保障。
**那么，什么时候可以用`notify()`？** 只有在满足以下所有严苛条件时，才可以使用`notify()`作为一种性能优化：
1. **所有等待者是同质的**：即所有等待线程的类型和等待的条件都完全相同。
2. **单进单出**：任何一个线程调用`notify()`，最多只能唤醒一个线程来满足它的条件。

# Thread.sleep(0) 的作用?
它的作用和它的名字`sleep`（睡眠）几乎没有关系，甚至可以说是完全相反。
`Thread.sleep(0)`的真正意图是**触发一次操作系统的重新调度，主动请求当前线程“放弃”CPU时间片**。
是一个 **“让位”** 的动作。

现代操作系统都是**抢占式的多任务系统**。CPU会被划分成一个个极短的 **“时间片”**。调度器会把这些时间片轮流分配给各个“就绪”状态的线程。一个线程通常会一直运行，直到它的时间片用完，然后被操作系统强制“抢占”，换下一个线程运行。

**`Thread.sleep(100)`**：你拿到麦克风，但你对主持人说：“我需要准备100毫秒的材料，这期间别叫我”。于是主持人让你**回到座位上休息（进入等待状态）**，100毫秒内都不会再把麦克风给你。

**`Thread.sleep(0)`**：你拿到麦克风，只讲了30秒，你觉得自己这一小段讲完了，想听听别人的意见。于是你对主持人说：“**我这一轮的发言可以先到这里了，你看看有没有其他人想讲**”。你这是在**主动地、自愿地提前结束你自己的发言时间**。

<mark class="hltr-green">工作原理</mark>
1. **线程状态**：一个线程在调用`Thread.sleep(0)`时，它必须处于“运行中”状态。    
2. **系统调用**：这个Java方法会触发一个底层的系统调用，通知操作系统内核。    
3. **调度器行为**：操作系统调度器收到这个“睡眠0毫秒”的请求后，会将其理解为：“当前线程自愿放弃剩余的时间片”。于是，调度器会：    
    - 将该线程从“运行中”状态，立刻变回“就绪”状态。        
    - 将它放入对应优先级的“就绪队列”的**队尾**。        
    - 触发一次**重新调度**，从所有就绪的线程中，根据优先级等策略，选择下一个要运行的线程。
<mark class="hltr-grey">现在几乎不使用</mark>
1. 现代OS的线程调度器已经非常成熟
2. 行为不确定且不可移植，它完全依赖于底层操作系统的实现。在某些OS上它可能效果明显，在另一些OS上它可能约等于一个空操作。
3. 有更好的替代方案。阻塞队列、wait/notify等，这些工具能让你清晰地表达“等待某个条件”的意图，而不是用`sleep(0)`这种模糊的“让一下”的把戏。

<mark class="hltr-green">和 Thread.yield() 的区别？</mark>
`Thread.sleep(0)` 和 `Thread.yield()` 都是**低级别的、依赖于调度器的API**，它们的共同意图是告诉线程调度器：“我愿意让出当前CPU的执行权”。但在**提示的强度**和**行为的确定性**上，它们存在着关键的区别。
简单来说：**`Thread.sleep(0)` 的“让步”意愿更强，而 `Thread.yield()` 更像一个“无所谓的建议”**。

| 特性           | `Thread.sleep(0)`           | `Thread.yield()`            |
| ------------ | --------------------------- | --------------------------- |
| **调度器提示强度**  | **较强**。通常会强制操作系统进行一次重新调度。   | **非常弱**。仅仅是一个提示，调度器完全可以忽略它。 |
| **对同优先级线程**  | **大概率会让出**。当前线程会被放到就绪队列的队尾。 | **不一定会让出**。可能只让给更高优先级的线程。   |
| **JLS 规范保证** | 无严格保证，但暗示一次线程状态的转换。         | **完全没有保证**，规范明确说明这是一个提示。    |
| **平台依赖性**    | **高**                       | **极高**                      |

# 线程池
线程池是一种池化技术，用于预先创建并管理一组线程，避免频繁创建和销毁线程的开销，提高性能和响应速度。

它几个关键的配置包括：核心线程数、最大线程数、空闲存活时间、工作队列、拒绝策略。
![[Pasted image 20250910105313.png]]
主要工作原理如下：

1. 默认情况下线程不会预创建，任务提交之后才会创建线程（不过设置 prestartAllCoreThreads 可以预创建核心线程）。
2. 当核心线程满了之后不会新建线程，而是把任务堆积到工作队列中。
3. 如果工作队列放不下了，然后才会新增线程，直至达到最大线程数。
4. 如果工作队列满了，然后也已经达到最大线程数了，这时候来任务会执行拒绝策略。
5. 如果线程空闲时间超过空闲存活时间，并且当前线程数大于核心线程数的则会销毁线程，直到线程数等于核心线程数（设置 allowCoreThreadTimeOut 为 true 可以回收核心线程，默认为 false）。

# 为什么线程池要先使用阻塞队列，而不是直接增加线程？
因为每创建一个线程都会占用一定的系统资源（如栈空间、线程调度开销等），直接增加线程会迅速消耗系统资源，导致性能下降。

使用阻塞队列可以将任务暂存，避免线程数量无限增长，确保资源利用率更高。

如果阻塞队列都满了，说明此时系统负载很大，再去增加线程到最大线程数去消化任务即可。

举个例子：老板现在手下有 10 个人在干活（核心线程数），突然活变多了，每个人干不过来了，此时老板不会立马招人，它会让这些活积累一下（放到阻塞队列中），看看过段时间能不能消化掉。如果老板发现这个活积累的实在太多了（队列满了），他才会继续招人（达到最大线程数）。这就是所谓的人员（线程）有成本。

# Java 创建线程池有哪些方式？ 
1）使用 Executors 工厂类，例如`Executors.newFixedThreadPool(10);`

2）使用 ThreadPoolExecutor 直接创建线程池
```java
ExecutorService threadPool = new ThreadPoolExecutor(
    5, // corePoolSize
    10, // maximumPoolSize
    60, // keepAliveTime
    TimeUnit.SECONDS, // TimeUnit
    new LinkedBlockingQueue<Runnable>(100) // BlockingQueue
);
```

3）通过 ForkJoinPool 创建并行任务线程池。
```java
ForkJoinPool forkJoinPool = new ForkJoinPool();
forkJoinPool.submit(() -> {
    // Task
});
```

# Java 并发库中提供了哪些线程池实现？它们有什么区别？

Java的并发库（JUC, `java.util.concurrent`）通过`Executors`工厂类，为我们预先配置了5种最常见的线程池实现。
1）**FixedThreadPool (固定大小线程池)**：创建一个固定数量的线程池。
- 拥有固定数量的核心线程，最大线程数也与之相等。    
- 线程处理完任务后不会被销毁，会一直复用。    
- 当所有线程都在忙时，新任务会进入一个无边界的队列中等待。
**适用**：
- 适合负载稳定的场景，任务数量确定且不需要动态调整线程数。
- 需要严格控制并发线程数量的场景，比如你需要管理有限的资源（数据库连接、CPU密集型任务）。
**风险**：
- **OOM (OutOfMemoryError) 风险**！由于其等待队列`LinkedBlockingQueue`是无界的，如果任务提交的速度远远超过线程处理的速度，队列会无限增长，最终耗尽内存。**这是它最主要的风险点**

2）**CachedThreadPoo(可缓存线程池)l**：一个可以根据需要创建新线程的线程池。
- 核心线程数为0，没有“常驻”线程。    
- 最大线程数几乎是无限的。    
- 使用`SynchronousQueue`，这意味着它不会缓存任务。每当一个任务提交时，如果没有空闲线程，就会**立刻创建一个新线程**来处理它。    
- 空闲线程超过60秒会被回收。
**适用场景**：
- **大量并发、短暂**的异步任务。    
- 负载压力经常变化，需要弹性伸缩的场景。
**风险**：
- **OOM (OutOfMemoryError) 风险**！由于其最大线程数是`Integer.MAX_VALUE`，如果瞬间涌入大量任务，可能会创建成千上万的线程，每个线程都需要占用一定的栈内存，最终会耗尽系统资源导致OOM。**这是它最主要的风险点**。

3）**SingleThreadExecuto (单线程线程池)r**：创建一个只有单个线程的线程池。
- 永远只有一个工作线程（最大线程数1，核心线程数1）。
- 保证所有提交的任务按照**FIFO (先进先出)** 的顺序串行执行。
- **适用场景**：    
    - 需要保证任务顺序执行的场景。例如，多个需要写入同一个文件的数据，可以依次提交给这个线程池，保证写入顺序。        
- **风险**：    
    - 与`FixedThreadPool`相同，**无界队列可能导致OOM**。        
    - 由于是单点执行，性能吞吐量有限。

4）**ScheduledThreadPool (可调度线程池)**：支持定时任务和周期性任务的线程池。
- **特点**：    
    - 核心线程数固定，但最大线程数是无界的（`Integer.MAX_VALUE`）。        
    - 专门用于执行需要延迟或周期性执行的任务。        
    - 提供了`schedule()` (延迟执行), `scheduleAtFixedRate()` (固定频率执行), `scheduleWithFixedDelay()` (固定延迟执行)等方法。        
- **适用场景**：    
    - 需要执行定时任务、周期性任务的场景，如心跳检测、定时清理缓存、定时发送报表等。     
- **风险**：    
    - `maximumPoolSize`是无界的，虽然不像`CachedThreadPool`那样容易触发，但仍有风险。其内部的`DelayedWorkQueue`也是无界的，如果提交了大量延迟非常长的任务，也可能造成OOM。
5）**WorkStealingPool**：基于任务窃取算法的线程池。
线程池中的每个线程维护一个双端队列（deque），线程可以从自己的队列中取任务执行。如果线程的任务队列为空，它可以从其他线程的队列中“窃取”任务来执行，达到负载均衡的效果。
适合大量小任务并行执行，特别是递归算法或大任务分解成小任务的场景。

<mark class="hltr-green">总结：</mark>
在阿里巴巴的《Java开发手册》中，明确规定**禁止使用`Executors`去创建线程池**。
**原因**：从上表可以看出，`Executors`创建的线程池，<mark class="hltr-cyan">要么使用了无界队列，要么最大线程数是无界</mark>的，这都隐藏了**资源耗尽的风险**。

**推荐的做法**：**通过`ThreadPoolExecutor`的构造函数，手动创建线程池。**

# 如何设计线程池?
### 1. 线程池的核心设计目标
- **资源利用率最大化：** 确保 CPU 不会因为线程频繁创建销毁而浪费资源，同时让 CPU 保持繁忙。 
- **系统稳定性：** 防止因创建过多线程而导致内存溢出（OOM）或系统卡顿。    
- **任务可控性：** 能够管理、监控和处理提交给线程池的任务。
### 2. 参数的选择与依据
- **`corePoolSize` 和 `maximumPoolSize` (核心与最大线程数)：** 这是线程池设计最关键的部分。我会根据任务的类型来决定：    
    - **CPU 密集型任务：** 这种任务大部分时间都在进行计算，很少等待。为了不让线程上下文切换的开销太大，我会将核心线程数设置为 `CPU核心数 + 1`，最大线程数可以设为核心线程数。        
	    - **避免因页缺失（Page Fault）导致的阻塞**：在一些极端情况下，即使是 CPU 密集型任务，也可能因为**缺页中断**而导致短暂的阻塞。
		    - **什么是缺页中断？** 当程序需要访问的内存页不在物理内存中时，会触发一个**缺页中断**。    此时，操作系统需要从磁盘将该页数据加载到内存，这个过程涉及到**磁盘 I/O**，是**非常慢的**。- 这会导致当前线程暂停执行，等待数据加载完成。
    - **I/O 密集型任务：** 这种任务大部分时间都在等待 I/O（如数据库查询、文件读写、网络请求）。在等待时，线程不会占用 CPU。我会将核心线程数设置为一个较大的值，公式通常是 `CPU核心数 * (1 + I/O耗时/CPU耗时)`。最大线程数可以设得更高一些，以应对突发流量。        
- **`workQueue` (任务队列)：**    
    - **有界队列（如 `ArrayBlockingQueue`）：** 如果我希望控制任务总量，防止内存溢出，我会选择有界队列。它能提供一种**反压（Backpressure）** 机制，当任务过多时，会触发拒绝策略，保护系统。        
    - **无界队列（如 `LinkedBlockingQueue`）：** 如果任务提交速度稳定，且单个任务执行时间较短，可以使用无界队列。但需要警惕任务堆积导致的内存问题。
	    - 如何防止内存问题：1. 即使队列是无界的，你仍然可以设置一个合理的 `corePoolSize`
	    - 2. **引入外部监控：**  **队列深度监控：** 这是最重要的指标。你需要实时监控 `LinkedBlockingQueue` 的大小。如果队列长度持续增长并达到某个阈值（例如几千或几万），就应该触发告警，提醒开发者。
			- **JVM 内存监控：** 监控 JVM 的堆内存使用情况。如果堆内存持续增长，且 `Full GC` 频繁，很可能就是任务队列堆积导致的。
- **`RejectedExecutionHandler` (拒绝策略)：** 这是系统的最后一道防线，当线程池和队列都满了，我需要决定如何处理新任务。    
    - **`AbortPolicy`：** 抛出异常，让调用方感知到系统过载。这是最直接的策略。        
    - **`CallerRunsPolicy`：** 让提交任务的线程自己执行这个任务。这可以有效地降低任务提交速率，是一种很好的反压机制。        
    - **自定义策略：** 在批量导出等特定场景下，我可能会自定义拒绝策略，将任务信息持久化到数据库或消息队列，待系统空闲时再异步处理。
### 3. 设计的额外考量点
除了核心参数，一个健壮的线程池设计还应考虑：
- **线程命名：** 通过自定义 `ThreadFactory` 为线程赋予有意义的名称（例如 `export-task-thread-1`），这在排查问题和分析日志时非常有用。    
- **异常处理：** 异步任务中的异常如果未被捕获，可能会被默默吞掉。我会在任务的 `Runnable` 或 `Callable` 中捕获异常，并记录日志，或者通过 `CompletableFuture` 的异常处理机制来管理    
- **监控与调优：** 我会暴露线程池的运行指标，例如任务数、队列大小、活跃线程数等，以便实时监控和进行动态调优。

# 线程同步
线程同步是指在多线程环境下，为了避免多个线程对共享资源进行同时访问，从而引发数据不一致或其他问题的一种机制。它通过对关键代码段加锁，使得同一时刻只有一个线程能够访问共享资源。

当多个线程共享同一资源（如变量、对象或文件）时，若没有同步机制，可能会导致**竞态条件**，即线程对共享资源的操作是非原子性的，多个线程之间可能会同时修改数据，导致结果不符合预期。

常见同步方式
#### synchronized
Java 提供的加锁关键字，用于在方法或代码块上加锁，以确保同一时刻只有一个线程能够执行被同步的方法或代码块。
在 synchronized 可以使用 wait()、notify() 和 notifyAll() 实现条件等待通知。
- wait()：当前线程进入等待状态，直到被其他线程唤醒。必须在同步块或同步方法中调用。
- notify()：唤醒一个等待的线程。如果有多个线程在等待，同一时刻只能唤醒一个。
- notifyAll()：唤醒所有等待的线程。
例如在 synchronized 块或方法中，可以使用 wait() 方法使线程等待某个条件满足，可使用 notify() 或 notifyAll() 方法唤醒等待的线程。

#### ReentrantLock
是 JUC （java.util.concurrent）提供的可重入锁，相比 synchronized 它更加灵活，
ReentrantLock 使用 Condition 对象来提供了更灵活的等待/通知机制。每个 ReentrantLock 可以创建一个或多个 Condition 对象，通过 newCondition() 方法创建。
- await()：使当前线程等待，直到收到信号或被中断。
- signal()：唤醒一个等待线程。
- signalAll()：唤醒所有等待线程。
相比于 synchronized，ReentrantLock 还提供了公平锁和非公平锁机制。

# 核心线程 和 非核心线程

|对比维度|核心线程 (Core Thread)|非核心线程 (Non-core Thread)|
|---|---|---|
|**创建时机**|**优先创建**。当任务到来，只要当前线程数少于`corePoolSize`，就会被创建，直到达到`corePoolSize`。|**滞后创建**。只有当核心线程都在忙，并且工作队列(`workQueue`)也满了之后，才会作为“救急”方案被创建。|
|**生命周期**|**默认是“永生”的**。一旦被创建，它们会一直存活在线程池中，即使长时间没有任务执行也不会被销毁。|**有“生命时限”的**。如果一个非核心线程在`keepAliveTime`这段时间内一直没接到新任务，它就会被销-毁回收。|
|**核心作用**|**承担常规负载**。它们是线程池稳定处理能力的基石，保证了线程池能随时响应任务。|**处理突发负载**。它们为线程池提供了弹性伸缩的能力，用于在短时间内消化掉激增的任务，任务洪峰过后就自动消失，释放资源。|


# 线程池参数
Java线程池`ThreadPoolExecutor`的构造函数通常有7个参数，其中5个是核心：

1. **`int corePoolSize` (核心线程数)**    
    - 线程池中保持活动状态的最小线程数，即使它们处于空闲状态。这是线程池的“常备军”。
        
2. **`int maximumPoolSize` (最大线程数)**    
    - 线程池能够容纳同时执行的最大线程数。这个值必须大于等于1，并且大于等于`corePoolSize`。这是线程池的总兵力（常备军+临时工）。
        
3. **`long keepAliveTime` (线程空闲存活时间)**    
    - 当线程池中的线程数量超过`corePoolSize`时，如果一个线程的空闲时间达到`keepAliveTime`，它将被终止。这主要用于管理**非核心线程**。
        
4. **`TimeUnit unit` (存活时间的单位)**    
    - `keepAliveTime`参数的时间单位，如`TimeUnit.SECONDS`（秒）、`TimeUnit.MILLISECONDS`（毫秒）等。
        
5. **`BlockingQueue<Runnable> workQueue` (工作队列)**    
    - 用于保存在被提交但尚未被执行的任务。它是线程池缓冲机制的核心。常见的队列有：        
        - `ArrayBlockingQueue`：一个由数组结构组成的有界阻塞队列。必须指定容量。            
        - `LinkedBlockingQueue`：一个由链表结构组成的阻塞队列。可以是无界的（默认容量`Integer.MAX_VALUE`），**使用时要特别小心，可能导致内存溢出**。            
        - `SynchronousQueue`：一个不存储元素的阻塞队列。每个插入操作必须等待一个相应的删除操作，反之亦然。它相当于任务的“直接交接”。
            
6. `ThreadFactory threadFactory` (线程工厂)    
    - 用于创建新线程。可以通过它给线程自定义命名，方便问题排查。
        
7. `RejectedExecutionHandler handler` (拒绝策略)    
    - 当队列已满且线程数达到`maximumPoolSize`后，用于处理新提交任务的策略。常见的策略有：        
        - `AbortPolicy` (默认)：直接抛出`RejectedExecutionException`异常。            
        - `CallerRunsPolicy`：由提交任务的那个线程自己来执行这个任务。这是一种有效的“反压”机制。            
        - `DiscardPolicy`：直接悄悄地丢弃这个任务，不报任何错。            
        - `DiscardOldestPolicy`：丢弃队列最前面的任务，然后重新提交当前任务。

# 一个新任务的流程
当一个新任务通过`execute()`方法提交给线程池时，它会严格按照以下**四步判断流程**来处理：
1. **判断核心线程数**：检查当前运行的线程数是否小于`corePoolSize`。    
    - **是**：则**立刻创建一个新的核心线程**来执行该任务，即使其他核心线程当前是空闲的。        
    - **否**：进入下一步。
        
2. **判断工作队列**：尝试将任务添加到`workQueue`中。    
    - **成功**：任务被放入队列，等待空闲线程来取出并执行。        
    - **失败**（通常因为队列已满）：进入下一步。
        
3. **判断最大线程数**：检查当前运行的线程数是否小于`maximumPoolSize`。    
    - **是**：则**创建一个新的非核心线程**来执行该任务。        
    - **否**（说明线程池已达到最大容量）：进入下一步。
        
4. **执行拒绝策略**：线程池已经完全饱和，无法再处理新任务。此时，调用`RejectedExecutionHandler`来处理这个任务。

# 线程池调优
**核心思想：在“性能”与“资源”之间找到最佳平衡点**
调优的本质，是在**吞吐量/响应时间**和**CPU/内存消耗**这两个对立的目标之间寻找一个最优点。
- **线程太少**：无法充分利用CPU资源，导致系统吞吐量低下，任务处理慢，响应延迟高。    
- **线程太多**：线程本身占用内存，大量的线程切换会消耗CPU时间（称为上下文切换开销），反而可能导致系统性能下降。

一个科学的调优过程，应该遵循“**评估 -> 压测 -> 监控 -> 调整 -> 验证**”的闭环。

#### 第一步：任务性质评估（最重要的前提）
在开始调优前，你必须清晰地知道线程池要执行的是什么类型的任务。这决定了调优的大方向。
1. **CPU密集型 (CPU-Bound)**    
    - **特点**：任务需要进行大量的计算，几乎没有等待（IO、网络）时间。例如：复杂的算法、视频编码、正则表达式计算。        
    - **调优目标**：用最少的线程数，让CPU的每个核心都饱和运转，最大化利用CPU，同时最小化线程上下文切换的开销。        
    - **理论公式**：`线程数 = CPU核心数 + 1` (这个 `+1` 是为了防止某个线程因偶尔的页错误或其他原因阻塞时，CPU能有替补线程)。        
2. **IO密集型 (I/O-Bound)**    
    - **特点**：任务的大部分时间都在等待IO操作的完成。例如：数据库查询、文件读写、调用外部API。        
    - **调优目标**：创建足够多的线程，以确保在部分线程等待IO时，CPU能够被其他“就绪”状态的线程利用起来，不至于空闲。        
    - **理论公式**：`线程数 = CPU核心数 * (1 + 任务的平均等待时间 / 任务的平均计算时间)`
        - 这个公式偏理论，在实践中，一个更经验性的初始值可以是 `线程数 = CPU核心数 * 2`。
#### 第二步：设定基准，进行压力测试
基于第一步的评估，为线程池设置一个初始的、合理的参数。然后，使用压力测试工具（如 JMeter, Gatling, nGrinder）模拟真实世界的负载，来观察系统的表现。
#### 第三步：核心监控指标
在压测过程中，你需要像医生一样，通过监控数据来给系统“听诊把脉”。关键的监控指标有：
- **线程池自身指标 (通过JMX或Prometheus暴露)**：    
    - `poolSize`：当前池中线程数。观察它是否能达到`maximumPoolSize`。        
    - `activeCount`：当前正在执行任务的线程数。        
    - `queueSize`：等待队列中的任务数。**这是最重要的指标之一**。如果它持续增长，说明消费能力跟不上生产能力。        
    - `completedTaskCount`：已完成的任务总数，可以用来计算吞吐量。        
    - `rejectedTaskCount`：被拒绝的任务数。        
- **系统级指标**：    
    - **CPU使用率**：如果是CPU密集型任务，理想值应该在90%以上。如果是IO密集型，这个值可能不高，是正常的。        
    - **内存使用和GC活动**：观察线程数的增减是否导致内存压力和频繁的GC。        
    - **平均响应时间 (Latency)** 和 **吞吐量 (Throughput/QPS)**：这是最终的业务指标，也是调优成功与否的衡量标准。       
#### 第四步：分析与调整
根据监控到的数据，进行分析和调整。这是一个迭代的过程。
- **场景一：CPU使用率很低，但队列长度持续增长，响应时间变长。**    
    - **分析**：典型的IO密集型任务，但线程数不足。线程都在等待IO，CPU闲置，任务在队列里堆积。        
    - **调整**：逐步**增加`corePoolSize`和`maximumPoolSize`**，观察吞吐量和响应时间的变化。        
- **场景二：CPU使用率很高（甚至100%），但吞吐量却上不去。**    
    - **分析**：典型的CPU密集型任务，但线程数过多。CPU的大量时间都耗费在了线程上下文切换上，而不是真正地执行任务。        
    - **调整**：逐步**减少线程池大小**，观察吞吐量的变化曲线，找到那个峰值点。        
- **场景三：压测时出现大量任务被拒绝。**    
    - **分析**：瞬间请求洪峰超出了系统的最大处理能力 (`maximumPoolSize` + `queueCapacity`)。   
    - **调整**：        
        1. 首先考虑**增大`workQueue`的容量**。这是最直接的缓冲方式。            
        2. 如果业务要求低延迟，不能容忍任务在队列里长时间等待，那就需要**增大`maximumPoolSize`**，让更多的“临时工”线程来快速处理突发任务。            
        3. 评估`RejectedExecutionHandler`，`CallerRunsPolicy`在这种场景下可以起到一定的缓冲和反压作用。            
- **场景四：内存使用持续增长，有OOM风险。**    
    - **分析**：最可能的原因是使用了**无界队列**（如`LinkedBlockingQueue`的默认构造），导致任务无限堆积。        
    - **调整**：**必须换成有界队列**（如`ArrayBlockingQueue`或指定容量的`LinkedBlockingQueue`），并设置一个合理的容量上限。这是保证生产环境稳定性的重要原则。
