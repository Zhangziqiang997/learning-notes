## 对比

| 特性        | B树（B-Tree）                                       | B+树（B+Tree）                                                |
| :-------- | :----------------------------------------------- | :--------------------------------------------------------- |
| **数据存储**  | **所有节点（包括非叶子节点和叶子节点）都存储键值和对应的数据**。               | **只有叶子节点存储键值和对应的数据**。非叶子节点只存储键值和子节点的指针。                    |
| **非叶子节点** | 包含键值和指向子节点及数据的指针。                                | 只包含键值和指向子节点的指针，不存储实际数据。                                    |
| **叶子节点**  | 存储键值和对应的数据。                                      | 存储键值和对应的数据。此外，**所有叶子节点之间通过链表连接**。                          |
| **查询效率**  | 每次查找都可能在非叶子节点找到数据，查询路径长度不固定，效率相对不固定。             | 所有查询都必须查找到叶子节点才能获取数据，查询路径长度固定且一致。                          |
| **范围查询**  | 对于范围查询，需要进行中序遍历，可能需要在多个节点之间跳跃，效率相对较低。            | 所有叶子节点构成有序链表，非常适合范围查询。只需找到范围的起始点，然后通过链表顺序遍历即可。             |
| **磁盘I/O** | 非叶子节点存储数据，每个节点能存储的键值数量相对较少，导致树的高度可能更高，增加磁盘I/O次数。 | 非叶子节点只存储键值和指针，能存储更多的键值，从而降低树的高度，减少磁盘I/O次数。                 |
| **内存利用率** | 非叶子节点也存储数据，有效键值数量减少，缓存命中率可能受影响。                  | 非叶子节点更“纯粹”，只包含索引信息，因此在相同内存页中可以存储更多的键值，提高了内存利用率和缓存命中率。      |
| **应用场景**  | 文件系统（如APFS、ReiserFS）中的目录索引，一些不常需要范围查询的数据库。       | 数据库系统（如MySQL的InnoDB存储引擎、PostgreSQL、Oracle）中的索引，文件系统中的文件索引。 |

B+树之所以成为数据库索引的首选，主要原因有：

1. **更少的磁盘I/O**: 非叶子节点不存储数据，使得单个节点可以存储更多的索引键值。这意味着在相同的磁盘块大小下，B+树的扇出（fan-out）更大，树的高度更低。数据库操作（如查询）通常涉及磁盘I/O，更少的树高意味着更少的磁盘块读取，从而提高查询性能。
2. **更高效的范围查询**: 数据库中经常需要进行范围查询（例如 `WHERE age BETWEEN 18 AND 30`）。B+树的叶子节点通过链表连接，使得范围查询变得非常高效。一旦找到起始点，就可以沿着链表顺序遍历，而无需回溯或重新从根节点查找。
3. **所有查询路径长度相同**: 所有的查询都必须到达叶子节点才能获取数据，这使得查询性能更加稳定。
4. **更好的缓存命中率**: 非叶子节点只存储索引键值，可以尽可能多地缓存在内存中，减少磁盘访问。

## InnoDB页的逻辑结构
1.  **File Header (文件头部)**

    - **大小**: 38字节
    - **作用**: 记录页的通用信息，如页的校验和（checksum）、页类型（Page Type）、上一页和下一页的物理地址（用于B+树叶子节点的双向链表）、页的LSN（Log Sequence Number，用于恢复和检查一致性）等。
    - **重要性**: 其中的`FIL_PAGE_PREV`和`FIL_PAGE_NEXT`字段是B+树叶子节点之间形成双向链表的基础，使得范围查询非常高效。
2. **Page Header (页头部)**
    
    - **大小**: 56字节
    - **作用**: 记录页的状态信息，如页中包含的记录数、第一个和最后一个记录的偏移量、空闲空间的起始位置、页的级别（B+树中，0代表叶子节点）、最大最小记录的ID、插入记录的方向等。
    - **重要性**: 其中的`PAGE_N_DIR_SLOTS`表示页目录中的槽数量，`PAGE_HEAP_TOP`表示空闲空间的起始位置。
3. **Infimum + Supremum Records (最小记录和最大记录)**
    
    - **作用**: 这不是用户实际插入的数据，而是两个伪记录（pseudo-records）。
        - **Infimum Record**: 比该页中所有用户记录都小，是该页的第一个记录。
        - **Supremum Record**: 比该页中所有用户记录都大，是该页的最后一个记录。
    - **重要性**: 它们的存在使得在页内对记录的查找和定位更加方便和统一，所有用户记录都位于这两个伪记录之间。
4. **User Records (用户记录区)**
    
    - **大小**: 不固定，取决于用户实际插入的数据量。
    - **作用**: 存储所有行数据。这些记录按照主键的顺序通过**单向链表**连接，从infimum记录开始，到supremum记录结束。
    - **重要性**: 这是存储实际表数据的地方。记录的物理存储顺序不一定和逻辑顺序一致，但逻辑上它们通过链表有序连接。当记录被删除时，其空间会被标记为可复用，但不会立即回收。
5. **Free Space (空闲空间)**
    
    - **大小**: 不固定。
    - **作用**: 用于填充新的记录。当新的记录插入时，会首先使用这部分空间。
    - **特点**: 这部分空间是一个链表，记录被删除后，其占用的空间会被添加到这个链表中。
6. **Page Directory (页目录 / Slots)**
    
    - **大小**: 不固定，每个槽2字节。
    - **作用**: 页目录中存放的是“槽”（Slots）。每个槽指向一组记录的最后一条记录（即组的最高记录）。这些槽是有序的。
    - **重要性**: InnoDB通过页目录来管理和加速页内记录的查找。当查找某条记录时，首先通过二分查找在页目录中定位到对应的槽，然后通过槽指向的记录，在小范围内顺序查找，大大提高了查找效率。每个槽通常负责4-8条记录。
7. **File Trailer (文件尾部)**
    
    - **大小**: 8字节
    - **作用**: 用于校验页的完整性。它包含一个校验和（checksum）和页的LSN的后4字节。
    - **重要性**: 与File Header中的校验和一起，用于在I/O过程中验证页的数据是否损坏或被篡改。

# 索引

<mark class="hltr-red">分类</mark>
从**数据结构**角度来看，MySQL 索引可以分为以下几类：
- **B+树索引**：通过树形结构存储数据，适用于范围查询（如 `BETWEEN`）和精确查询（如 `=`），支持有序数据的快速查找、排序和聚合操作。是 MySQL 默认的索引类型，常用于 **InnoDB** 和 **MyISAM** 引擎。
- **哈希索引**：基于哈希表的结构，适用于等值查询（如 `=`），查询速度非常快，但不支持范围查询（如 `>`、`<`）。哈希索引不存储数据的顺序，常用于 **Memory** 引擎。
- **倒排索引（Full-Text）**：用于全文搜索，**Elasticsearch**，将全文分词，通过存储词与文档的映射，支持模糊匹配和关键字搜索。特别适合用于大文本字段，如 `TEXT` 类型的列，用于查找包含特定词语的记录。
- **R-树索引**：专为多维空间数据（如地理坐标）设计，适用于空间查询（例如，计算地理位置的最近距离、区域查询等）。常用于存储和查询地理信息系统（GIS）中的空间数据。

从**常见的基于 InnoDB B+ 树索引角度来看**，可以分为：
- 聚簇索引（Clustered Index）
- 非聚簇索引（Non-clustered Index）

从**索引性质**的角度来看，可以分为：
- **普通索引（二级索引、辅助索引）：**一般指非主键索引且非唯一索引。
- **主键索引**
- **联合索引**
- **唯一索引**： 保证索引列中的值是唯一的，可以有效防止重复数据的插入。唯一索引允许 `NULL` 值，但一个列中可以有多个 `NULL`。
- **全文索引**：用于全文搜索，支持对长文本字段（如 `TEXT` 类型）进行关键字查找，支持自然语言处理、模糊匹配等操作。适用于需要对文本内容进行复杂搜索的场景。
- **空间索引** ：用于空间数据（如地图上的经纬度坐标等）查询。通常使用 R-树结构，适合多维数据的查询，如区域查询和最近距离查询，主要用于 **MyISAM** 和 **InnoDB** 存储引擎中的地理信息数据。

## 聚簇索引
- 在 InnoDB 中，**主键索引就是聚簇索引**（即数据按主键顺序存储）。
- 索引叶子节点存储的是数据行，可以直接访问完整数据。

## 非聚簇索引——二级索引
- 除了主键以外的其他字段建立的索引
- 叶子节点存储的是行的主键和对应的索引列，需通过主键才能访问完整的数据行---**回表**
- 而且。回表会带来**随机I/O**，因为回表查的主键id是不连续的，所以频繁回表效率很低。
- 覆盖索引：
	- 是一次查询中的所有列刚好是联合索引里面的所有列，这时候直接从走联合索引而不走主键索引
```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY,     -- 主键索引（聚簇索引）
    name VARCHAR(50),
    age INT,
    INDEX idx_name (name)       -- 二级索引
);
```

## 联合索引

覆盖索引——不用回表

<mark class="hltr-red">最左匹配原则
</mark>

联合索引在 B+ 树中的排列方式遵循“从左到右”的顺序，跳过最左侧字段会导致无法利用该索引。

以下查询条件 **符合** 最左匹配原则：
``` sql
where a=1; 
where a=1, b=2; 
where a=1, b=2, c=3;
```

以下查询条件 **不符合** 最左匹配原则：
```sql
where b=2;
where c=3;
where b=2, c=3;
```

对于这个：
```sql
where a=1, c=3;
```
"可能"用上联合索引，因为**索引下推**

### 索引下推

索引下推（Index Condition Pushdown, ICP）是一种<mark class="hltr-yellow">减少回表查询</mark>，提高查询效率的技术。

它允许 MySQL 在使用索引查找数据时，将部分查询条件下推到存储引擎层过滤，从而减少需要从表中读取的数据行，减少了 IO（本该由 Server 层做操作，交由存储引擎层因此叫做 “下推” ） 。

索引下推在 MySQL 5.6 及以后的版本支持。

假设当前 people 表有索引 `INDEX (zipcode, lastname, firstname)` ，当执行以下查询时：
```sql
SELECT * FROM people
  WHERE zipcode='95054'
  AND lastname LIKE '%etrunia%'
  AND address LIKE '%Main Street%';
```

**如果没有索引下推**，当前的联合索引只能用上 `zipcode='95054'` 这个条件，引擎层用不上 `lastname` 这个条件过滤，只能得到所有符合 `zipcode='95054'` 记录，传递给 server 层过滤。

**有了索引下推之后**，引擎层在得到符合`zipcode='95054'`的数据后，可以直接通过 lastname 条件过滤数据，不符合条件的不会返回给 server 层。

## 索引失效

1. 左或左右模糊匹配，因为 MySQL 采用最左匹配原则。
```sql
SELECT * FROM users WHERE name LIKE '%abc';
```

2. 查询条件中对索引列使用函数。
```sql
SELECT * FROM users WHERE LOWER(name) = 'abc';
```

3. 查询条件对索引列使用表达式计算。
```sql
SELECT * FROM users WHERE age + 10 = 30;
```

4. 隐式类型转换。
	如果索引列是字符串，并且条件语句中输入参数是数字，那么索引列会产生隐式类型转换，CAST 函数实现，因为等同于堆索引列使函数，导致索引失效，反之索引列是数字，输出参数是字符串，那么不会失效。
```sql
SELECT * FROM users WHERE str = 1234 ;
```

5. 在 Where 子句中，如果在 OR 前的条件列是索引列，但 OR 后面的条件列不是索引列。
```sql
SELECT * FROM users WHERE name = 'abc' OR age = 30;
```

6. 使用 Select * 语句，大概率不会走索引，因为不是每一列都加索引。

7. 使用 Not Exists 关键字，索引也会失效，本质上是 Where 查询范围过大。
```sql
SELECT * FROM users WHERE NOT EXISTS (SELECT * FROM orders WHERE orders.user_id = users.id);
```

8. 使用 Order by 注意最左匹配，并且要加 limit 或者 Where 关键字，否则索引会失效。
```sql
SELECT * FROM users ORDER BY name LIMIT 10;
```

# SQL慢查询优化

1）合理设计索引，利用联合索引进行覆盖索引的优化，避免回表的发生，减少一次查询和随机 I/O

2）避免 SELECT * ，只查询必要的字段

3）避免在 SQL 中进行函数计算等操作，使得无法命中索引

4）避免使用 %LIKE，导致全表扫描

5）注意联合索引需满足最左匹配原则

6）不要对无索引字段进行排序操作

7）连表查询需要注意不同字段的字符集是否一致，否则也会导致全表扫描


# INNER JOIN、LEFT JOIN 和 RIGHT JOIN 

**INNER JOIN**：

- 只返回两个表中匹配的行。如果没有匹配，则该行不会出现在结果集中。
- 适合用在只需要共同数据的结果集中。

**LEFT JOIN**（或 LEFT OUTER JOIN）：

- 返回左表中的所有行，即使右表中没有匹配的行。如果右表没有匹配，则结果中的右侧列会显示为NULL。
- 在需要保留主表所有数据的报表中非常有用，比如在客户（左表）和订单表（右表）中查找所有客户及其订单信息，即使有些客户没有订单。

**RIGHT JOIN**（或 RIGHT OUTER JOIN）：

- 返回右表中的所有行，即使左表中没有匹配的行。如果左表没有匹配，则结果中的左侧列会显示为NULL。
- 适用于需要保留右表所有数据的场景。

## 为什么在 MySQL 中不推荐使用多表 JOIN？ 

**性能问题**：

- 多表 JOIN 可能导致查询性能下降，尤其是在处理大数据集时，JOIN 操作的计算复杂度会显著增加，需要进行大量的数据扫描和匹配，增加了内存和CPU的消耗，导致响应时间变长。

**可读性和维护性**：

- 多表 JOIN 的查询语句较为复杂，降低了 SQL 的可读性和可维护性。复杂的语句可能会增加错误发生的概率，使得后续的调试和优化更加困难。

# ACID // MySQL 如何实现事务？

MySQL 主要是通过：**锁、Redo Log 、Undo Log、MVCC** 来实现事务。

事务基本特性：

原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）和持久性（Durability）

## 原子性（Atomicity）

**<mark class="hltr-red">由 undo log 保证</mark>**

记录了数据被修改前的信息。如果在事务执行过程中失败，MySQL 会利用 `undo` 日志将数据回滚到事务开始前的状态。

## 一致性（Consistency）

**<mark class="hltr-red">一致性是目的，由其他三大特性来保证</mark>**

一致性保证事务的执行将数据库从一个一致状态转移到另一个一致状态。就是说，无论事务是否成功完成，数据库的完整性约束（如外键约束、唯一性约束）都不会被打破。例如，在转账后，两个账户的总金额应该保持不变，而不会因为某个操作失败导致金额丢失或多出。
## 隔离性（Isolation）

**<mark class="hltr-red">由 MVCC 保证</mark>**

隔离性保证一个事务的执行不会被其他事务干扰，多个事务并发执行时，其效果等价于按某种顺序串行执行。不同的隔离级别（如读未提交、读已提交、可重复读、序列化）提供了不同级别的保护，以处理脏读、幻读和不可重复读等问题。例如，当一个事务正在读取某一行数据时，另一个事务不应该修改这一行数据。

通过**锁机制**和 **MVCC（多版本并发控制）** 来实现。锁机制保证了数据在被一个事务修改时，其他事务无法访问。MVCC 则是一种更高效的机制，它通过保存数据的多个版本，让读操作和写操作可以同时进行，大大提高了并发性能。

1、 锁
	事务在修改数据之前，需要先获得相应的锁。获得锁之后，事务便可以修改数据。该事务操作期间，这部分数据是锁定的，其他事务如果需要修改数据，需要等待当前事务提交或回滚后释放锁。

2、 MVCC

## 持久性（Durability）

**<mark class="hltr-red">由内存 + redo log 来保证</mark>**

`redo` 日志记录了事务对数据所做的所有修改。事务提交时，`redo` 日志会被持久化到磁盘上。即使系统崩溃，重启后也能通过 `redo` 日志来重做（re-do）已提交的事务，确保数据不丢失

持久性保证事务一旦提交，其结果必须永久保存在数据库中，不会因为系统故障或其他问题而丢失。
# 三种日志

`bin log` 属于 Server 层，作用是记录所有**已提交**的事务操作，用于**数据库复制**（主从同步）和**数据恢复**。

`undo log` 和 `redo log` 属于 存储引擎层，确保 **`InnoDB` 引擎**自身的数据事务安全，即保证 **ACID 特性**。

## undo log（回滚日志）

<mark class="hltr-yellow">逻辑日志</mark>

确保<mark class="hltr-red">原子性</mark>

当执行 `UPDATE` 语句时，`InnoDB` 存储引擎会在**数据真正修改前**，将**旧的数据快照**写入 `undo log`。

如果这个事务在提交前失败了（比如断电、服务器崩溃），MySQL 会利用 `undo log` 中的记录，恢复到更新前的状态，从而确保了原子性。

## redo log （重做日志）

<mark class="hltr-yellow">物理日志</mark>

确保 <mark class="hltr-red">持久性</mark>，即已提交的事务对数据库的修改是永久的，即使系统崩溃也不会丢失。

在数据页被修改后，`InnoDB` 会将这个**修改操作本身**写入 `redo log`。

它记录的是“在某个数据页的某个偏移量上做了某个修改”，比如“将 `id = 1` 的用户的 `balance` 减去了 100”。

`redo log` 的写入是**顺序的**，而且是 **<mark class="hltr-green">先于数据页写入磁盘</mark>** 的（WAL, Write-Ahead Logging）。当事务提交时，`InnoDB` 会强制将 `redo log` 刷入磁盘。即使数据页还没来得及同步到磁盘，系统突然崩溃，重启后 MySQL 也会根据 `redo log` 里的记录，重新执行一遍操作，恢复数据到崩溃前的状态。

# 隔离级别

## 解决什么问题

1. **脏读（Dirty Read）**
- **定义**：一个事务**读到了另一个未提交事务修改过的数据**。
- **比喻**：读者 A 在一本书上做了修改，但还没决定要不要提交这个修改（还没归还）。这时，读者 B 拿过这本书，看到了这个修改。如果读者 A 最终决定撤销修改（回滚事务），那么读者 B 看到的数据就是“脏”的，因为它最终并不存在。

2. **幻读（Phantom Read）**
- **定义**：一个事务**在前后两次查询同一个范围的数据时，查到的记录数量发生了变化**。
- **比喻**：读者 A 统计书架上所有《计算机网络》书籍的数量，发现有 10 本。在他还没离开这个书架时，另一个读者 B 借走了一本，并归还了一本新的《计算机网络》。当读者 A 再次统计时，发现数量还是 10 本。但在他统计的过程中，书籍的数量发生了变化，这就是“幻影”。更常见的幻读是，读者 A 再次统计时，发现多了或少了。
- **与不可重复读的区别**：
	- **不可重复读**：侧重于**同一条记录**在多次读取时，其**内容**发生了变化。
	- **幻读**：侧重于一个**范围内的记录数量**发生了变化，新增或删除了记录。

## 分类
SQL 标准定义了四种隔离级别，这四种隔离级别分别是：

- 读未提交（READ UNCOMMITTED）；
    - 这是最低的隔离级别，在该级别下，一个事务可以看到另一个事务尚未提交的数据修改。这可能会导致**脏读**问题，即读取到其他事务未提交的数据。

- 读已提交 （READ COMMITTED）；
    - 在这个级别下，一个事务只能看到已经提交的其他事务所做的修改。这可以避免脏读问题，但是可能会引发**不可重复读**问题，即在同一个事务中，相同的查询可能返回不同的结果。

- 可重复读 （REPEATABLE READ）；
	- 在这个级别下，确保在一个事务中的多个查询返回的结果是一致的。这可以避免不可重复读问题，但是可能会引发幻读问题，即在同一个事务中，多次查询可能返回不同数量的行（MySQL 默认的隔离级别）。

- 串行化 （SERIALIZABLE）。
	- 并发 SQL 事务在 SERIALIZABLE 隔离级别下的执行被保证是可串行化的。可串行化执行被定义为：并发执行的 SQL 事务的操作，其效果与这些 SQL 事务按某种顺序串行执行的效果相同。串行执行是指每个 SQL 事务在下一个 SQL 事务开始之前完成其全部操作。
	- 这是最高的隔离级别，在这个级别下，保证事务间的操作结果相当于一个按顺序执行的单线程操作。这可以避免所有的并发问题，但是会大大降低并发性能。

事务隔离是为了解决脏读、不可重复读、幻读问题，下表展示了 4 种隔离级别对这三个问题的解决程度：

| 隔离级别             | 脏读  | 不可重复读 | 幻读  | 级别  |
| ---------------- | --- | ----- | --- | --- |
| READ UNCOMMITTED | ×   | ×     | ×   | 最低  |
| READ COMMITTED   | √   | ×     | ×   |     |
| REPEATABLE READ  | √   | √     | ×   |     |
| SERIALIZABLE     | √   | √     | √   |     |
较低的隔离级别（如读未提交）提高了并发性，但可能导致数据不一致；

较高的隔离级别（如串行化）保证数据一致性，但降低了并发性。因此，在设计应用时，需要在性能和数据一致性之间找到平衡。

如何实现？
- 读未提交：天然就有不用实现
-  串行化：加锁
- 读已提交、可重复读：MVCC
## 如何解决幻读
MySQL **InnoDB 存储引擎** 在默认的 **可重复读（Repeatable Read）** 隔离级别下，通过 **MVCC** 和 **间隙锁（Gap Lock）** 的组合，彻底解决了幻读问题。

- **MVCC（多版本并发控制）**：它可以解决**快照读**（即普通的 `SELECT` 语句）下的幻读。MVCC 为每个事务创建一个数据快照，让事务在整个生命周期中都看到这个快照中的数据，即使有其他事务插入了新数据，当前事务也看不到，从而避免了幻读。
    
- **间隙锁（Gap Lock）**：它是解决**当前读**（即 `SELECT ... FOR UPDATE` 或 `INSERT`、`UPDATE`、`DELETE` 等语句）下幻读的关键。间隙锁锁定的是索引记录之间的“间隙”，防止其他事务在这个间隙中插入新的记录。
### 技术拓展：间隙锁的实现

间隙锁是 **InnoDB** 独有的锁，它只在**可重复读**隔离级别下生效。你可以把间隙锁想象成在数据库中的 “空位”上加了一把锁。

当一个事务执行一个范围查询并加锁（如 `SELECT ... FOR UPDATE`）时，**InnoDB** 会使用**间隙锁**来锁定这个范围：
```sql
-- 事务 A
SELECT * FROM products WHERE id BETWEEN 100 AND 200 FOR UPDATE;
```
如果 `products` 表中 `id` 为 105、110、195 的记录存在，那么 `InnoDB` 会对以下**间隙**加锁：
- `(100, 105)`
- `(105, 110)`
- `(110, 195)`
- `(195, 200)`
当另一个事务尝试在这些间隙中**插入**一条新的记录（比如 `id=107`）时，它就会因为无法获取间隙锁而被**阻塞**。这样就保证了在事务 A 的生命周期内，这个范围内的记录数量不会发生变化，彻底解决了幻读。


# MVCC 多版本并发控制

<mark class="hltr-grey">维护一个数据的多个版本，每个事务修改一次，就生成一个版本</mark>
<mark class="hltr-grey">让不同的事务去读不同的版本</mark>

在 MVCC 中，数据库为每个事务创建一个数据快照。每当数据被修改时，MySQL 不会立即覆盖原有数据，而是生成新版本的记录。每个记录都保留了对应的版本号或时间戳。

多版本之间串联起来就形成了一条版本链，这样不同时刻启动的事务可以**无锁**地获得不同版本的数据(普通读)。此时读(普通读)写操作不会阻塞。

- 读已：让事务读已经提交的数据版本，避免了脏读
- 可重：让事务 每次 读到同一个版本的数据，避免了不可重复读

<mark class="hltr-purple">使用 `undo log` 实现</mark>
	给表增加一个<mark class="hltr-yellow">隐藏字段-回滚指针 - roll_pointer</mark>
	指向 `undo log`，串成一个链表

<mark class="hltr-purple">事务怎么知道要查哪个版本？——每个事务分配事务 id- trx_id</mark>
	事务id自增，通过对比id大小，知道事务创建的先后，再记录事务是否提交了
	让提交的早的事务，不要看到提交的晚的事务
	因此，给表再加一个<mark class="hltr-yellow">隐藏字段-修改数据的事务id（谁修改它的）</mark>
	
<mark class="hltr-purple">readview</mark>
- creator_trx_id，创建当前 readView 的事务ID。
- m_ids，创建readView 时，活跃的事务ID集合，也就是已经启动但是还未提交的事务ID列表。
- min_trx_id，创建readview时，数据库中活跃ID之中的最小值。
- max_trx_id，创建readview时， 数据库中将分配给下一个事务的 ID 的值（事务 ID 是递增分配的，越后面申请的事务ID越大）
<mark class="hltr-blue">判断</mark>
- trx_id == creator_trx_id， 修改这条数据的事务就是当前事务，所以可见。
- trx_id < min_trx_id，修改这条事务的数据版本在当前事务生成 readView 的时候已提交，可见。
- min_trx_id < trx_id < max_trx_id，分情况
	- 如果trx_id 若在 m_ids 中，说明生成这个数据的事务还未提交，不可见；
	- 相反，不在，说明生成这个数据的事务已经提交，可见
- trx_id >= max_trx_id，创建这条数据的事务在当前事务生成 readView 的时候还未启动，所以不可见(结合事务ID递增来看)。

<mark class="hltr-purple">如何实现隔离级别</mark>
1. 读已提交
- 每次 select 的时候，都生成一个新的readview

2. 可重复读
- 会在第一次 select 的时候生成一个 readview
- 每次都会复用第一次的readview，已经固定下来了，实现了可重复读

<mark class="hltr-grey">为了避免幻读——加锁</mark>
- **快照读**：事务在查的时候，不直接读当前最新数据，只读历史版本（快照），读不到其他事务做了什么，就不会产生幻读
	- 不可能永远读不到：比如你先开启了事务，别人后开启，但别人先插入id为1的数据，你读不到，以为不存在，你就插入id为1。所以插入这种操作，必须读到新数据，才判断能不能插，这就是当前读。
- **当前读**：读最新的数据。mysql-临键next-key锁，锁定读范围内的所有记录和间隙，防止其他事务在该范围内插入新记录，就不会幻读
- `SELECT ... FOR UPDATE` (加排他锁)、`SELECT ... LOCK IN SHARE MODE` (加共享锁)、
`UPDATE`、`INSERT`、  `DELETE`——都是当前读

- <mark class="hltr-green">依然不能避免幻读</mark>：先快照读，再当前读，就会出现幻读
	- 例子： 事务A先开启事务，查id=1的数据，查不到；
	- 事务B后开启，插入id=1的数据；
	- 事务A再次用普通select查询id=1，由于是快照读，还是查不到；
	- 事务A执行`select for update` 查，是当前读，查到了——幻读。
- 解决：一个事务操作某张表时候，另外一个事务不允许新增、删除
	- 加表锁、行锁——`select for update` 

# 乐观锁和悲观锁

**悲观锁（Pessimistic Locking）**：

- 假设会发生冲突，因此在操作数据之前就**对数据加锁**，确保其他事务无法访问该数据。常见于对数据一致性要求较高的场景。
- 实现方式：使用行级锁或表级锁，例如可以使用 `SELECT ... FOR UPDATE` 或 `LOCK IN SHARE MODE` 语句来加锁。

**乐观锁（Optimistic Locking）**：
核心思想是：**“无锁定，先修改，提交时再校验”**
- 假设不会发生冲突，因此在操作数据时不加锁，而是在更新数据时**进行版本控制或校验**。如果发现数据被其他事务修改，则会拒绝当前事务的修改，需重新尝试。
- 实现方式：通常通过版本号或时间戳来实现，每次更新时检查版本号或时间戳是否一致。

1）乐观锁适合**并发冲突少，读多写少**的场景，不用通过加锁只需通过比较字段版本号（或时间戳）是否发生改变的形式，无锁操作，吞吐量较高。

数据库实现乐观锁的例子（版本号）：
```sql
-- 假设有一张用户表 users，包含 id、name 和 version 字段
-- 读取数据
SELECT id, name, version FROM users WHERE id = 1;

-- 更新数据时检查版本号
UPDATE users
SET name = 'new_name', version = version + 1
WHERE id = 1 AND version = current_version;
```
2）悲观锁适合**并发冲突多，写多读少**的场景。通过每次加锁的形式来确保数据的安全性，吞吐量较低。

数据库实现悲观锁的例子：
```sql
-- 读取数据并加锁
SELECT id, name FROM users WHERE id = 1 FOR UPDATE;

-- 执行更新操作
UPDATE users SET name = 'new_name' WHERE id = 1;
```


# 死锁

## 操作系统中的死锁

锁是指两个或多个进程在执行过程中因争夺资源而造成的一种僵局，当进程处于死锁状态时，它们将无法继续执行，而只能相互等待，直到被外部的程序干预或自行放弃。

**死锁的必要条件：**
1. 互斥条件：每个资源只能被一个线程占用。
2. 占有和等待：线程在持有至少一个资源的同时，等待获取其他资源。
3. 不可抢占：线程所获得的资源在未使用完毕之前不能被其他线程抢占。
4. 循环等待：多个线程形成一种头尾相接的循环等待资源关系。

预防和避免死锁需要采取一些措施，包括：

1. **避免资源独占**：尽量避免一个进程在获得了某些资源后再次请求其他资源，而应该将所有需要的资源一次性申请到位。
2. **避免资源持有和等待**：当一个进程占用了一些资源并等待另一些资源时，其他进程就无法使用这些资源，容易引发死锁。因此，尽可能减少资源持有和等待时间。
3. **避免资源互斥**：有些资源在同一时间只能被一个进程占用，比如打印机、磁带机等，需要采用一些技术手段来避免资源互斥的问题。
4. **引入资源剥夺策略**：当一个进程请求的资源被其他进程占用时，可以采取剥夺资源的策略，即暂停占用该资源的进程，直到该资源被释放后再恢复该进程的执行。
5. **引入进程抢占策略**：当一个进程等待时间过长时，可以采取抢占其资源的策略，即中断正在执行的进程，强制释放其占用的资源。
## MySQL中的死锁
**自动检测与回滚**：
- MySQL 自带死锁检测机制（innodb_deadlock_detect），当检测到死锁时，数据库会自动回滚其中一个事务，以解除死锁。通常会回滚事务中持有最少资源的那个。
- 也有锁等待超时的参数（innodb_lock_wait_timeout），当获取锁的等待时间超过阈值时，就释放锁进行回滚。
**手动 kill 发生死锁的语句**：
- 可以通过命令，手动快速地找出被阻塞的事务及其线程 ID，然后手动 kill 它，及时释放资源。

常见避免死锁或降低死锁的手段
**1）避免大事务**。大事务占据锁的时间长，将大事务拆分成多个小事务快速释放锁，可降低死锁产生的概率和避免冲突。

**2）调整申请锁的顺序**。在更新数据的时候要保证获得足够的锁，举个例子：先获取影响范围大的锁，比如说修改操作，先将排他锁获取到，再获取共享锁。或固定顺序访问数据，这样也能避免死锁的情况。

**3）更改数据库隔离级别**。可重复读比读已提交多了间隙锁和临键锁，利用读已提交替换之可降低死锁的情况。

**4）合理建立索引，减少加锁范围**。如果命中索引，则会锁对应的行，不然就是全表行都加锁，这样冲突大，死锁的概率就高了。

**5）开启死锁检测**，适当调整锁等待时长


# 三大范式
### 第一范式（1NF）

**定义：** 确保表中的**每一列都是不可再分的原子数据项**。

- **通俗理解：** 每一格数据都是“原子”的，不能再拆分。
    
- **例子：**
    
    - **不符合 1NF**：一个 `students` 表，有一个 `contact_info` 列，里面存储了多个值，比如“电话:12345678, 邮箱:xxx@example.com”。
        
    - **符合 1NF**：将 `contact_info` 拆分为 `phone` 和 `email` 两列。
        

---

### 第二范式（2NF）

**定义：** 在满足 1NF 的基础上，表中的**非主键列必须完全依赖于主键**。

- **通俗理解：** 每一列数据都必须**直接**与整个主键相关。如果主键是复合键，非主键列不能只依赖于主键的一部分。
    
- **例子：**
    
    - 假设有一个 `order_details` 表，主键是复合键 `(order_id, product_id)`。
        
    - **不符合 2NF**：`customer_name` 列只依赖于 `order_id`，而不依赖于整个主键 `(order_id, product_id)`。
        
    - **符合 2NF**：将 `customer_name` 拆分到另一个 `orders` 表中，只保留 `order_id`。
        

---

### 第三范式（3NF）

**定义：** 在满足 2NF 的基础上，表中的**非主键列之间不能存在传递依赖**。

- **通俗理解：** 如果 A 决定 B，B 决定 C，那么 C 不应该放在由 A 决定的表中。
    
- **例子：**
    
    - 假设有一个 `students` 表，主键是 `student_id`。`city_name` 和 `zip_code` 都是非主键列。
        
    - **不符合 3NF**：`zip_code` 依赖于 `city_name`，而 `city_name` 依赖于 `student_id`。这就形成了传递依赖：`student_id` -> `city_name` -> `zip_code`。
        
    - **符合 3NF**：将 `city_name` 和 `zip_code` 拆分到另一个 `cities` 表中，`students` 表只保留 `city_id` 作为外键。

# Elasticsearch ES

Elasticsearch 是一个开源的分布式搜索引擎，提供了强大的全文搜索和复杂查询功能。
**1）Elasticsearch**
主要的应用场景：全文搜索、日志分析
优点：搜索性能高、能够实现灵活的全文搜索、实时性强、支持分布式扩容
缺点：部署运维成本较大，且查询操作学习成本较高

**2）MySQL**
主要的应用场景：关系型数据存储、事务和数据一致性支持（比如财务系统）
优点：支持事务、成熟稳定
缺点：实时搜索性能和灵活性较差，不利于大数据量的扩展
在实际业务中，可以结合使用这两种数据库，把需要全文检索功能的数据同步到 Elasticsearch 上（比如文章数据）；而其他无需检索的关系型数据（比如用户数据）存储在 MySQL 中。

它的三大支柱：**倒排索引**、**搜索与评分机制**、**分布式架构**。
#### 1. 核心数据结构：倒排索引 (Inverted Index)
倒排索引主要由两部分组成：
- **词典 (Term Dictionary)**：记录了所有文档中出现过的**不重复的词条 (Term)**，并对它们进行排序（通常是字典序）。为了能快速找到某个词条，ES会使用像FST（Finite State Transducer，是一种高度压缩的前缀树）这样的数据结构来存储词典。    
- **倒排列表 (Posting List)**：对于词典中的每个词条，都有一个与之对应的列表。这个列表记录了**哪些文档包含了这个词条**。一个倒排列表通常包含：
1. **文档ID (Document ID)**：包含了该词条的所有文档的ID。**这些ID是严格按升序排列的**，这一点至关重要，它使得多个列表之间的合并（求交集、并集）操作可以极其高效地完成。
2. **词频 (Term Frequency, TF)**：该词条在每个文档中出现的次数。这个信息是**相关性评分**的核心依据之一。出现次数越多，通常意味着文档越相关。    
3. **位置 (Position)**：该词条在每个文档中出现的位置（比如是第几个词）。这个信息是**短语查询 (Phrase Query)** 和**邻近查询 (Proximity Query)** 的基础。例如，搜索 `"Hello World"` 这个短语，就需要保证 "World" 的位置必须紧跟在 "Hello" 之后。    
4. **偏移量 (Offset)**：该词条在文档中的起始和结束字符位置。这个信息主要用于对搜索结果进行**高亮显示**。


**文档写入流程 (Indexing)**： 当一个新文档写入ES时，它会经过一个叫做**“分词” (Analysis)** 的过程。
1. **分词 (Tokenization)**：将文档中的文本（比如 "I love data science"）切分成一个个独立的词条（"i", "love", "data", "science"）。    
2. **标准化 (Normalization)**：将切分出的词条进行处理，如转为小写、去除“的”、“是”这类无意义的停用词 (Stop Words)、提取词干（比如将 "running" 处理成 "run"）等。    
3. **建立索引**：将处理好的词条更新到词典和倒排列表中。
#### 2. 搜索与评分机制
- **搜索流程**：    
    - **单次查询**：如搜索 "java"。ES会快速在词典中定位到 "java"，然后直接返回其倒排列表中的所有文档ID。  
    - **布尔查询**：如搜索 "java" AND "expert"。ES会分别找到 "java" 和 "expert" 的倒排列表，然后对这两个列表（内部是有序的）求交集。这个交集运算在有序列表上非常快。OR就是求并集，NOT就是求差集。        
- **相关性评分 (Relevance Score)**： ES不仅仅是找出包含关键词的文档，更重要的是，它会**按相关性高低进行排序**。这个评分的计算模型很复杂（早期是TF-IDF，现在主要是BM25），但其核心思想包括：    
    - **词频 (Term Frequency, TF)**：一个词在文档中出现的次数越多，说明该文档与该词越相关。        
    - **逆文档频率 (Inverse Document Frequency, IDF)**：一个词在**所有文档**中出现的次数越少，说明这个词越稀有、越重要。比如，“的”这个词几乎所有文档都有，它的IDF就很低；而“Elasticsearch”这个词只在特定文档中出现，它的IDF就很高。        
#### 3. 分布式架构
单个服务器的容量和算力总是有限的，ES从设计之初就是分布式的。
- **节点 (Node) & 集群 (Cluster)**：一个ES实例就是一个节点，多个节点组成一个集群。    
- **索引 (Index)**：一个逻辑上的数据集合，类似于MySQL中的数据库。    
- **分片 (Shard)**：ES会将一个大的索引**水平切分**成多个小的部分，每个部分就是一个**分片**。每个分片都是一个功能完备、独立的搜索引擎（一个Lucene实例）。**这是ES能够存储海量数据的关键。**    
- **副本 (Replica)**：ES可以为每个分片创建一份或多份拷贝，这就是副本。副本有两个主要作用：    
    1. **高可用**：如果持有主分片(Primary Shard)的节点宕机，一个副本分片(Replica Shard)可以被提升为新的主分片，保证服务的连续性。        
    2. **提升读性能**：搜索请求可以同时在主分片和副本分片上并行执行，提高了查询的吞吐量。       
当一个搜索请求到达集群时，ES会把这个请求广播到所有相关的分片上，每个分片独立完成本地搜索，然后将结果返回给协调节点，由协调节点进行汇总、排序，最后返回给用户。
# 一条 SQL 语句在 MySQL 中的执行过程

我们以一条查询 SQL 为例：select * from user where id=1；

在这条查询语句中，大体可以分为两个层面，Server 层和存储引擎层。

Server 层涉及到一系列的业务的组件，而存储引擎层则是负责存储发送过来的数据、提供读写接口等功能。
具体流程如下：

1）当客户端的 SQL 发送到 MySQL 时，首先是到达 Server 层的**连接器**，连接器会对你此次发起的连接进行权限校验，以此来获取你这个账号拥有的权限。当你的账号或密码不正确时，会报 `Access denied for user` 错误，相信大家对这个错误也并不陌生。连接成功如果后续没有任何操作，那么这个连接就处于空闲状态，到达一定时间后它便会断开连接，这个时间一般是 8 小时，是由 **wait_timeout** 参数控制的。

2）查询缓存（在 MySQL 8.0 之后就被砍了）具体做法就是将一个查询语句作为 key ，将上一次请求的结果作为 value，存储在缓存组件中，当同样的语句来查询的时候即可立马返回结果，不需要经历词法、语法分析等以下的步骤。只要表有数据改动缓存就失效了，在我们常见的联机事务处理（OLTP） 场景下是个鸡肋。
	8.0后删除的原因：数据可能变，导致结果不对；缓存不容易命中

3）接下来就到了**分析器**来进行**语法分析**、**词法分析**。MySQL 会首先对你的语句进行 ”词法分析“ ，来判断你的语句是什么类型以及携带什么参数等。比如：MySQL 会将输入语句的 select 提取出来，判断出这是一条查询语句、将 from 后面的 user 提取出来作为查询的表名、把 id 提取出来作为列名等。做完这些 MySQL 将会进行 ”语法分析“ 来判断你的语句的语法是否有误、是否满足 MySQL 的语法。如果语法有问题，那这个错误相信大家都不陌生：`You have an error in your SQL syntax; check the manual......`。

4）经过分析器就到了**优化器**，它会对你的语句进行优化判断。比如你的表中有多个索引，优化器会帮你选择使用哪个索引、你使用了 join 多表连接，优化器会帮你调整表的连接顺序。我们平日里用的 explain 其实就是让 MySQL 告诉我们它的优化决定策略是怎样的。

5）最后会到达**执行器**，它先会判断你对这个 user 表是否有权限查询，如果没有权限它将会拒绝本次查询，返回错误信息。如果有权限，它将会根据表的存储引擎提供的接口进行数据查询将重复遍历表的行数据，判断 id 字段是否等于 1。直到遍历完整个表将符合条件的数据作为结果集返回给客户端。

# MySQL如何优化

## 如何监控
可以利用 MySQL 自带的 `slow_query_log` 来监控慢 SQL，它是 MySQL 提供的一个日志功能，用于记录执行时间超过特定阈值的 SQL 语句。
### MySQL 慢 SQL 配置

在 MySQL 配置文件（通常是 my.cnf 或 my.ini）中添加或修改以下配置项：
```
[mysqld]
slow_query_log = 1                    # 启用慢查询日志
slow_query_log_file = /var/log/mysql/mysql-slow.log  # 指定慢查询日志文件路径
long_query_time = 2.0                 # 设置慢查询的阈值时间（单位：秒）
log_queries_not_using_indexes = 1     # 记录未使用索引的查询
```
其中的 2.0 就是 2 秒，当一个 SQL 执行的时间超过 2 秒，就会被记录到慢日志中。


对于慢查询，再使用 `EXPLAIN` 分析执行计划，查看查询的执行顺序、使用的索引、扫描的行数等，以识别潜在的性能瓶颈。

基于 `EXPLAIN` 再进行针对性的优化，常见的优化方向有：

- 根据 EXPLAIN 的结果，检查是否有合适的索引。若缺失索引，则添加（特别是在 WHERE、JOIN 和 ORDER BY 子句中使用的列上）
- 将复杂的 JOIN 查询拆分成多个简单查询，尽量小表驱动大表
- 避免 SELECT * ，仅选择需要的字段
- 等等(更多可参考扩展中的 SQL 调优)
## SQL 优化

根据慢 SQL 日志，找出需要优化的一些语句。

常见优化方向：

1）避免 SELECT * ，只查询必要的字段

2）避免在 SQL 中进行函数计算等操作，使得无法命中索引

3）避免使用 %LIKE，导致全表扫描

4）注意联合索引需满足最左匹配原则

5）不要对无索引字段进行排序操作

6）连表查询需要注意不同字段的字符集是否一致，否则也会导致全表扫描

## 库表设计优化

1）合理的表结构：比如选择合适的数据类型，例如能用 int 的不要用 bigint，还有 varchar 和 char 的选择等等。

2）合理冗余字段：在适当的情况下进行反规范化设计，冗余部分数据，减少关联查询。

3）索引优化：根据查询频率和条件，创建合适的索引，删除不必要的索引，因为索引的维护也是需要成本的。建议使用 EXPLAIN 分析查询执行计划，确认是否用上索引，是否用对索引。

4）分库分表：对于超大规模的数据库系统，可以采用分库分表策略，将数据拆分到多个数据库或表中，提高读写性能和扩展性。
