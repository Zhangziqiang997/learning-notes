# RPC

RPC，Remote Procedure Call，即远程过程调用，是一种技术协议，它允许你**像调用本地函数一样，去调用另一台机器上的函数或服务。**

**RPC 框架**的核心原理是基于网络传输协议实现的远程方法调用。**RPC 框架**通常由服务提供者和服务消费者两部分组成，服务提供者将本地方法暴露成远程服务，服务消费者通过远程代理对象调用远程方法。

在实现远程方法调用时，需要进行序列化和反序列化操作。序列化将对象转换为二进制数据流，以便于在网络中传输；反序列化则将接收到的二进制数据流转换为对象。

为了提高性能，一些 **RPC 框架**使用了二进制协议，如 Dubbo 使用的 Hessian2 协议和 gRPC 使用的 Protocol Buffers 协议，与基于文本的协议（如 XML 和 JSON）相比，二进制协议具有更小的传输体积和更高的解析速度，能够减少网络传输的开销。

## 常见的**RPC 框架**

**gRPC**：
- **简介**：gRPC 是 Google 开发的高性能 RPC 框架，基于 **HTTP/2** 协议，可以进行双向流通信，支持负载均衡、认证、追踪等特性。
- **优点**：使用 **Protocol Buffers（Protobuf）** 进行序列化和反序列化，传输效率高，适合微服务间的高性能通信。
- **应用场景**：适合需要高效传输和复杂数据结构的微服务架构，如实时通信服务、物联网数据传输。

**Thrift**：
- **简介**：Thrift 是由 **Apache** 维护的跨语言 RPC 框架，支持多种语言（如 Java、C++、Python、PHP），通过定义接口描述文件（IDL）来生成客户端和服务端代码。
- **优点**：支持多语言环境，适合异构系统之间的高效通信。
- **应用场景**：适用于需要跨语言、高性能的 RPC 调用场景，如大数据平台中的服务调用。

**Dubbo**：
- **简介**：Dubbo 是阿里巴巴开源的 Java RPC 框架，提供了高性能的远程服务调用能力，支持负载均衡、服务注册与发现等特性。
- **优点**：与 Spring 生态深度集成，支持多种通信协议和序列化方式，适合 Java 生态下的分布式系统。
- **应用场景**：适合中国市场中基于 Java 的微服务系统，特别是在 Spring Boot 环境下的微服务调用。

## RPC 和 HTTP的区别
HTTP 和 RPC 不是一个层面的东西：
- HTTP 是一种应用层的协议，主要强调的是网络通信；
- RPC，其是一种分布式系统之间通信的方式，强调的是服务之间的远程调用。

## 让你设计一个 RPC 框架，怎么设计？

**其实**就这么几点：
1）动态代理(屏蔽底层调用细节)  
2）序列化(网络数据传输需要扁平的数据)  
3）协议(规定协议，才能识别数据)  
4）网络传输(I/O模型相关内容，一般用 Netty 作为底层通信框架即可)

<mark class="hltr-orange">1. 动态代理</mark>
按上面的例子来说，yesService 服务实现被移到了远程服务上，本地没有具体的实现只有一个接口。
那这时候我们需要调用 `yesService.get(str)` ，所要做的就是把**传入的参数和调用的接口全限定名通过网络通信告知到远程服务**那里。然后远程服务接收到参数和接口全限定名就能选中具体的实现并进行调用。业务处理完之后再通过网络返回结果。
<mark class="hltr-grey">不过 yesService 就是一个接口，没有实现的，所以这些操作是怎么来的？</mark>
**是通过动态代理来的。**

RPC 会给接口生成一个代理类，所以我们调用这个接口实际调用的是动态生成的代理类，由代理类来触发远程调用，这样我们调用远程接口就无感知了。

动态代理想必大家都比较熟悉，最常见的就是 Spring 的 AOP 了，涉及的有 JDK 动态代理和 cglib。

<mark class="hltr-orange">2. 序列化</mark>
**把对象转化成二进制数据的过程称为序列化，把二进制数据转化成对象的过程称为反序列化。**

比如采用二进制的序列化格式数据更加紧凑，采用 JSON 等文本型序列化格式可读性更佳，排查问题比较方便。XML（冗长）。

<mark class="hltr-orange">3. RPC协议</mark>
只有二进制数据才能在网络中传输。
定义一个协议，把二进制数据还原成对应的一个个请求。
一般 RPC 协议都是**采用协议头+协议体的方式。**

协议头放一些元数据，包括：魔法位、协议的版本、消息的类型、序列化方式、整体长度、头长度、扩展位等。

协议体就是放请求的数据了。

通过魔法位可以得知这是不是咱们约定的协议，比如魔法位固定叫 233 ，一看我们就知道这是 233 协议。

然后协议的版本是为了之后协议的升级。

从整体长度和头长度我们就能知道这个请求到底有多少位，前面多少位是头，剩下的都是协议体，这样就能识别出来，扩展位就是留着日后扩展备用。

<mark class="hltr-orange">4. 网络传输</mark>
组装好数据就等着发送了，这时候就涉及网络传输了。
网络通信那就离不开网络 IO 模型了。
![[Pasted image 20251011163403.png]]
网络 IO 分为这四种模型，一般而言我们用的都是 IO 多路复用，因为大部分 RPC 调用场景都是高并发调用，IO 复用可以利用较少的线程 hold 住很多请求。
一般 RPC 框架会使用已经造好的轮子来作为底层通信框架。
例如 Java 语言的都会用 Netty ，人家已经封装的很好了，也做了很多优化，拿来即用，便捷高效。

<mark class="hltr-orange">真正工业级别的 RPC</mark>
以上提到的只是 RPC 的基础流程，生产环境中的服务提供者都是集群部署的，所以有多个提供者，而且还会随着大促等流量情况动态增减机器。

**因此需要注册中心，作为服务的发现。**
调用者可以通过注册中心得知服务提供者们的 IP 地址等元信息，进行调用。
调用者也能通过注册中心得知服务提供者下线。

还需要有**路由分组策略**，调用者根据下发的路由信息选择对应的服务提供者，能实现分组调用、灰度发布、流量隔离等功能。

还需要有**负载均衡策略**，一般经过路由过滤之后还是有多个服务提供者可以选择，通过负载均衡策略来达到流量均衡。

当然还需要有**异常重试**，毕竟网络是不稳定的，而且有时候某个服务提供者也可能出点问题，所以一次调用出错进行重试，减少业务的损耗。

还需要**限流熔断**，限流是因为服务提供者不知道会接入多少调用者，也不清楚每个调用者的调用量，所以需要衡量一下自身服务的承受值来进行限流，防止服务崩溃。

而熔断是为了防止下游服务故障导致自身服务调用超时阻塞堆积而崩溃，特别是调用链很长的那种，影响很大，比如A=>B=>C=>D=>E，然后 E 出了故障，你看ABCD四个服务就傻等着，慢慢的资源就占满了就崩了，全崩。

# Dubbo
Dubbo 是一个高性能、轻量级的开源 Java **RPC 框架**，它提供了完整的 RPC 协议栈，包括**服务发布、服务引用、负载均衡、容错、服务治理和服务监控**等功能，同时提供了可扩展的 RPC 协议、数据模型、序列化和网络传输等组件，支持多语言和多协议。

- **服务提供者**（如销售部）向**总机**（注册中心）注册自己的分机号（服务地址）。
- **服务消费者**（如市场部）想联系销售部时，不需要知道具体是谁在接电话，只需向总机询问“请帮我接销售部”（服务发现），总机就会提供一个可用的分机号。同时，总机还负责监控各部门的状态（健康检查）、在电话占线时提供其他分机（负载均衡）等。

**1. 服务提供者**
- **职责**：发布自身实现的服务到注册中心。    
- **行为**：在启动时，向注册中心注册自己的服务地址（IP、端口、接口名、版本号等）。  
**2. 服务消费者**
- **职责**：从注册中心订阅所需的服务。    
- **行为**：在启动时，向注册中心拉取提供者列表并缓存到本地。发起调用时，从本地列表中选择一个提供者进行远程调用。    
**3. 注册中心**
- **职责**：服务的注册与发现。是连接提供者和消费者的桥梁。    
- **实现**：通常使用 **ZooKeeper**、**Nacos**、**Redis** 等。它负责保存服务提供者的地址列表，并在列表发生变化时通知消费者。    
- **面试精要**：这里体现了 **CP** 还是 **AP** 的权衡。早期 Dubbo 多用 ZooKeeper（CP），保证强一致性；现代微服务更倾向 Nacos（默认 AP），保证高可用。    
**4. 监控中心**
- **职责**：统计服务的调用次数、调用时间等监控信息，用于运维和优化。

**工作流程（一次远程调用）**：
1. **服务提供者** 启动，向 **注册中心** 注册自己提供的服务。    
2. **服务消费者** 启动，向 **注册中心** 订阅自己所需的服务。注册中心将提供者地址列表返回给消费者。    
3. **注册中心** 在提供者地址变化（上线/下线）时，会主动通知消费者。    
4. **服务消费者** 基于软负载均衡算法，从本地缓存的服务列表中选择一个提供者，发起远程调用。    
5. **服务消费者** 和 **服务提供者** 在内存中累计调用次数和耗时，定时异步发送统计数据到 **监控中心**。

# 什么是 API 网关？

API 网关是位于客户端与后端服务之间的一层，**充当所有客户端请求的统一入口**，负责路由请求到正确的服务，并提供认证与授权、负载均衡、限流、缓存、日志记录等功能。

**为什么需要？**
API 网关是因为微服务架构的出现而出现的，因为微服务都是单独部署的，有各自的访问地址，对客户端而言一个页面的渲染可能需要调用多个服务的接口才能完成。

**作用**

**简化客户端调用**：
- **统一入口**：API 网关提供了**单一入口**，客户端只需与网关交互，而不需要了解后端服务的具体部署位置和接口细节，从而简化了客户端的调用逻辑。
- **请求聚合**：在需要调用多个后端服务时，API 网关可以将多个请求合并为一个请求，从而减少客户端的请求次数，提高性能。

**增强系统安全性**：
- **认证与授权**：API 网关可以对所有进入的请求进行**身份验证**和**权限控制**，如验证 OAuth2 令牌、JWT 等，确保只有授权用户才能访问特定的服务。
- **隐藏内部结构**：通过 API 网关，后端服务的内部结构对外部不可见，从而降低了被攻击的风险，增加了系统的安全性。

**流量管理与优化**：
- **负载均衡**：API 网关可以对不同服务实例之间的请求进行**负载均衡**，均匀分配流量，确保服务的高可用性。
- **限流与熔断**：API 网关可以实现**限流、熔断和降级**，在流量激增时保护后端服务，避免因某个服务不可用而影响整个系统。
- **缓存**：通过 API 网关，可以缓存一些高频访问的接口响应结果，减少后端服务的压力。

# 注册中心
在传统的单体应用中，模块间的调用是函数调用，地址是固定的内存地址。但在微服务架构下，每个服务都是一个独立的进程，部署在不同的机器上，它们通过网络进行通信。这就带来了几个严重的问题：

- **地址动态性：** 服务的 IP 地址和端口不是固定的。服务可能会因为故障重启、弹性扩缩容（增加或减少实例）而改变地址。    
- **服务数量庞大：** 微服务数量可能从几十个到成千上万个，手动管理这些服务的地址列表是不可想象的。    
- **负载均衡需求：** 通常一个服务会有多个实例来分担压力，调用方需要一种机制能将请求均匀地分发到这些健康的实例上。    

**注册中心的核心使命就是解决“服务发现 (Service Discovery)”这一难题。** 它提供了一个统一的地方，让服务能够“自我报告”地址（服务注册），也让调用方能够“查询”到地址（服务发现）。

<mark class="hltr-pink">注册中心 如何工作？原理？它的工作流程主要包含三个关键部分</mark>
- **服务注册 (Service Registration):**    
    - 当一个服务实例（比如 `order-service-01`）启动时，它会主动将自己的元数据（服务名 `order-service`、IP、端口、健康状态等）发送给注册中心。        
    - 注册中心收到后，会把这些信息记录在一个“服务清单”里。

- **心跳机制 (Heartbeat):**   **健康检查** 
    - 服务实例会周期性地（比如每30秒）向注册中心发送一个“心跳包”，证明自己还活着。        
    - 如果注册中心在一定时间内（比如90秒）没有收到某个实例的心跳，就会认为该实例已经“死亡”或不可用，并将其从服务清单中剔除。这保证了服务列表的实时性和有效性。

- **服务发现 (Service Discovery):**    
	- 服务消费者在发起RPC调用前，会向注册中心查询提供者的地址列表。这有两种主要模式：	
		- **客户端发现模式:** 消费者直接从注册中心获取所有提供者列表，并在本地通过负载均衡算法（如轮询、随机、加权）选择一个实例进行调用。这是最常用的模式（如Dubbo, Eureka）。	    
		- **服务端发现模式:** 消费者通过一个负载均衡器（如Nginx, Kubernetes Service）进行调用，由负载均衡器去查询注册中心并完成路由。

<mark class="hltr-pink">常见注册中心类型</mark>
注册中心本身就是一个高可用的分布式系统，所以它也必须遵循 CAP 定理，在 C（一致性）和 A（可用性）之间做出选择。
- **CP 类型的注册中心 (e.g., ZooKeeper, Etcd):**    
    - 它们采用 Raft 或 ZAB 这类共识算法，非常注重数据的**一致性**。        
    - **优点：** 任何时刻从注册中心拉取到的服务列表都是最准确的。        
    - **缺点：** 为了保证一致性，当网络分区导致集群无法形成“多数派”时，整个注册中心可能会变得不可用（无法写入或读取），这会影响到新服务的注册和现有服务的发现，风险较高。
- **AP 类型的注册中心 (e.g., Eureka, Nacos):**    
    - 它们更注重**可用性**。Eureka 的设计哲学是“宁可返回一个稍微过期的服务列表，也比无法提供服务导致整个系统瘫痪要好”。        
    - **优点：** 即使注册中心集群节点之间通信中断，每个节点依然可以独立对外提供服务，保证了注册和发现功能的可用性。        
    - **缺点：** 在网络分区期间，你可能会从注册中心拿到一个已经下线的服务地址，导致调用失败。但通常客户端会内置重试机制来缓解这个问题。Eureka 的“自我保护模式”就是这种设计哲学的极致体现。

<mark class="hltr-pink">RPC中的注册中心</mark>
**没有注册中心时：** 服务地址（IP和端口）需要硬编码在配置文件中，难以管理，无法动态扩容和容灾。  
**有注册中心时：** RPC调用流程变为：
1. **启动阶段：** 服务提供者启动，向注册中心注册自身信息。    
2. **订阅阶段：** 服务消费者启动，向注册中心订阅它所需要的服务列表，并缓存在本地。    
3. **调用阶段：**    
    - 消费者在发起RPC调用前，通过本地缓存的列表，通过负载均衡算法选择一个提供者。        
    - 然后执行我们上一讲中提到的完整RPC调用流程（序列化 -> 网络传输 -> ...）。        
4. **治理阶段：** 注册中心持续进行健康检查。当提供者下线，注册中心通知消费者更新列表，后续调用将自动避开故障节点。

# 什么是分布式？为什么需要分布式？

分布式是指在多台计算机上协同工作的系统，这些计算机通过网络连接在一起，共同完成一个任务。

  为什么需要：

| 驱动力          | 单体系统瓶颈                                              | 分布式解决方案                                               | 现实案例                                             |
| ------------ | --------------------------------------------------- | ----------------------------------------------------- | ------------------------------------------------ |
| **高性能**      | 单台服务器的CPU、内存、磁盘I/O存在物理上限。                           | **任务并行拆分**，将大规模计算任务分发到多个节点上处理，最后汇总结果。                 | **大数据计算**（如Hadoop/Spark）、科学计算。                   |
| **可扩展性**     | 单体服务器性能固定，流量激增时只能更换更昂贵的硬件（**垂直扩展/Scale-up**），存在天花板。 | **水平扩展（Scale-out）**，通过简单地增加廉价的普通服务器来分担负载。这是分布式最核心的优势。 | **Web应用服务器**，通过负载均衡器将用户请求分发到后端的成百上千台应用服务器上。      |
| **高可用**      | 单点故障！一旦这台唯一的服务器宕机，整个服务就彻底不可用。                       | **冗余与容错**。让多个节点提供相同的服务，当一个节点失败时，请求可以立刻被其他健康的节点接管。     | **数据库主从复制**、**微服务多实例部署**。即使部分机器宕机，整个系统依然能对外提供服务。 |
| **地理分布与低延迟** | 一个位于北京的服务器，无法为洛杉矶的用户提供低延迟的访问体验。                     | **将服务节点部署在全球多个数据中心**，让用户就近访问，提升体验。                    | **CDN（内容分发网络）** 将静态资源（图片、视频）缓存到全球各地的边缘节点。        |

# CAP理论
是指在分布式系统中，**一致性**（Consistency）、**可用性**（Availability）**和分区容错性**（Partition Tolerance）这三个指标无法同时满足的问题。具体来说：

- **一致性**（Consistency）：指多个副本之间数据保持一致，即在一个副本上的写操作会立即同步到其他所有副本，所有副本的数据都是最新的，保持**强一致性**。
- **可用性**（Availability）：指系统在任何时候都能对外提供服务，即**系统随时能够响应用户请求**，不会因为节点故障或其他原因而导致服务中断。
- **分区容错性**（Partition Tolerance）：指系统在出现**网络分区**（**节点之间失去联系**）时，仍能够继续工作，保证数据的一致性和可用性。

CAP 理论指出，**一个分布式系统只能同时满足其中的两个指标，无法同时满足三个**。

例如，当出现**网络分区**时，如果要**保证一致性**，就必须**停止对外服务**，从而**失去可用性**；如果要**保证可用性**，就必须**放弃一致性**，从而可能**导致不同节点之间数据不一致**。

因此，在设计分布式系统时，需要根据具体的场景和需求来选择合适的权衡方案，比如选择 **CP（一致性和分区容错性）** 或者选择 AP**（可用性和分区容错性）**。

需要注意的是，CAP 理论只是一种理论框架，不能直接应用于实际的分布式系统设计。**在实际应用中，还需要考虑系统的具体业务需求、数据访问模式、节点规模和部署环境等因素**，综合权衡之后再选择合适的分布式架构和技术方案。

# BASE理论

BASE理论是对CAP定理中“AP”架构的实践延伸。它通过牺牲数据的强一致性，来换取分布式系统的高可用性和可扩展性。

BASE理论是对ACID事务四个特性（原子性、一致性、隔离性、持久性）的一种妥协和补充，特别是在CAP定理的框架下，为了获得**可用性(A)** 和**分区容错性(P)** 而不得不放弃强**一致性(C)** 时的解决方案。

BASE是一个缩写，由三个部分组成：
- **BA: Basically Available (基本可用)**    
    - **是什么？** 分布式系统在出现不可预知的故障时，允许损失部分功能的可用性，但核心功能要保证依然可用。        
    - **如何实现？** 这不是系统完全崩溃，而是“有损服务”。例如：        
        - **流量削峰：** 在秒杀场景下，通过排队、验证码等方式延长用户请求的处理时间，从而保证系统不崩溃。            
        - **功能降级：** 在系统压力过大时，关闭非核心服务（如商品评论、个性化推荐），确保核心交易链路（下单、支付）畅通。            
        - **体验降级：** 在高峰期，返回给用户的数据可能是稍旧的缓存数据，而不是实时的数据库数据。

- **S: Soft State (软状态)**    
    - **是什么？** 指允许系统中的数据存在中间状态，并且该中间状态的存在不会影响系统的整体可用性。        
    - **如何理解？** 与“硬状态”（数据在任何时刻都保持一致且确定的）相对。软状态承认数据在同步过程中存在延迟，各个副本的数据可以暂时不一致。这个状态是“软”的，因为它会改变，最终会趋于一致。例如，主数据库同步到从数据库的延迟期间，从库的数据就是一种“软状态”。

- **E: Eventual Consistency (最终一致性)**    
    - **是什么？** 这是BASE理论的最终目标。它保证只要没有新的数据更新，经过一段时间的同步后，所有副本的数据最终会达到完全一致的状态。        
    - **关键点：** “一段时间”是一个系统设计变量，可能是几毫秒，也可能是几分钟。它强调系统**不保证**时刻一致，但**保证**最终一致。

# 什么是网关，网关有哪些作用？
网关（Gateway）是在计算机网络中用于**连接两个独立的网络的设备**，它能够在两个不同协议的网络之间传递数据。在互联网中，网关是一个**可以连接不同协议的网络的设备**，比如说可以**连接局域网和互联网**，它可以把**局域网的内部网络地址转换成互联网上的合法地址**，从而使得局域网内的主机可以与外界通信。

在计算机系统中，网关可以用于实现负**载均衡、安全过滤、协议转换**等功能。具体来说，网关可以分为以下几种：
- **应用网关**：用于应用层协议的处理，如 HTTP、SMTP 等。  Nginx  
- **数据库网关**：用于数据库访问的控制和管理。    
- **通信网关**：用于不同通信协议之间的数据交换，如 TCP/IP、UDP/IP 等。    
- **API 网关**：用于管理和转发 API 请求，实现 API 的授权、限流、监控等功能。  Spring Cloud Gateway  

总的来说，网关可以为不同网络提供连接和通信的功能，同时也可以提供安全、性能、可靠性等方面的增强功能，是现代计算机系统中不可或缺的一部分。

# 分布式锁-数据库实现
可以利用数据库的**行锁**或**唯一索引**来实现分布式锁。
- **工作原理**：
    1. **加锁**：客户端尝试在数据库中插入一条记录（例如，一个 `lock` 表），如果插入成功，则获得锁。
    2. **利用唯一索引**：在 `lock` 表的 `lock_name` 字段上创建唯一索引。如果多个客户端同时插入同一条记录，只有一个会成功。
    3. **解锁**：执行完业务逻辑后，删除这条记录。
- **优点**：
    - **简单易实现**：利用数据库的原子操作，逻辑简单。
    - **强一致性**：依赖于数据库的事务和 ACID 特性。
- **缺点**：
    - **性能问题**：数据库的并发处理能力有限，锁的竞争可能导致数据库成为性能瓶颈。
    - **死锁问题**：如果客户端崩溃，没有释放锁，会造成死锁。需要配合定时任务来清理超时的锁。
    - **扩展性差**：单个数据库实例的并发量有限，难以扩展。
    - **不具备阻塞锁特性**，获取不到锁直接返回失败，所以需要优化获取逻辑，循环多次去获取。

# 分布式锁-Redis实现

在 Redis 中实现分布式锁的常见方法是通过`set ex nx` 命令 + lua 脚本组合使用。
确保多个客户端不会获得同一个资源锁的同时，也保证了安全解锁和意外情况下锁的自动释放。

1）加锁：`SET lock_key uniqueValue EX expire_time NX` 2）解锁：使用 lua 脚本，先通过 get 获取 key 的 value 判断锁是否是自己加的，如果是则 del。<mark class="hltr-red">【先判断key，再删除，要保证原子性->lua】</mark>

**Q:** 那为什么set命令不用放到lua中？
**A:** 因为setnx是单个命令，redis执行是单线程，天然保证原子性

lua 脚本如下：
```lua
if redis.call("GET",KEYS[1]) == ARGV[1]
then
    return redis.call("DEL",KEYS[1])
else
    return 0
end
```
首先**锁需要有过期机制**。假设某个客户端加了锁之后宕机了，锁没有设置过期机制，会使得其他客户端都无法抢到锁。

`EX expire_time` 就是设置了锁的过期，单位是秒。还一个 `PX`， 也是过期时间，单位是毫秒。

然后，在 2.6.12 版本之前只有 `SETNX` 即 SET if Not eXists，它表示如果 key 已存在，则什么都不会做，返回 0，如果不存在则会设置它的值，返回 1。

那个时候 `SETNX` 和过期时间的设置就无法保证原子性，如果客户端在发送完 `SETNX` 之后就宕机了，还没来得及设置过期时间，一样会导致锁不会被释放。

因此 2.6.12 版本之后，优化了 SET 命令，使得可以执行 `set ex px`。

最后，再解释下什么是 `uniqueValue`，翻译过来就是一个唯一的值。

之所以要设置唯一值是为了**防止被别的客户端给释放了**。
	我们来看下这个场景：
		1.  客户端 1 加锁成功，然后执行业务逻辑，但执行的时间超过了锁的过期时间
		2.  此时锁已经过期被释放了，客户端 2 加锁成功
		3.  客户端 2 执行业务逻辑
		4.  客户端 1 执行完了，执行释放锁的逻辑，即删除锁。
客户端 2 的锁被删除了，所以每个客户端加锁（客户端可能是每个线程），需要是设置一个唯一标识，比如一个 uuid，防止锁被别的客户端误释放了。

因为需要先判断锁的值和唯一标识是否一致，一致后再删除释放锁，这里就涉及到两步操作，只有使用了 lua 脚本才能保证原子性，这也是为什么释放锁需要用 lua 脚本的原因。

## 看门狗

<mark class="hltr-green">拿到锁的线程挂了怎么办 -> 给锁加过期时间，自动释放 -> 线程还没执行完，超时，锁自动释放怎么办
-> 看门狗机制 -> 看门狗线程设置为守护线程，其生命周期依赖于业务线程，挂了JVM会自动终止守护线程</mark>

Redisson 的看门狗（watchdog）主要用来避免 Redis 中的锁在超时后业务逻辑还未执行完毕，锁却被自动释放的情况。它通过定期刷新锁的过期时间来实现自动续期。

**主要原理**：

1. **定时刷新**：如果**当前分布式锁未设置过期时间**，Redisson 基于 Netty 时间轮启动一个定时任务，定期向 Redis 发送命令更新锁的过期时间，默认每 10s 发送一次请求，每次续期 30s。
2. **释放锁**：当客户端主动释放锁时，Redisson 会取消看门狗刷新操作。如果客户端宕机了，定时任务自然也就无法执行了，此时等超时时间到了，锁也会自动释放。

### 如果获取锁的客户端挂了怎么办？

续期是通过定时任务执行的，如果当前客户端宕机，那么定时任务就没了

按照默认每 10s 续期 30s 的情况来看，锁不可能无限续期，等 30s 时间一到，集群中的其他客户端就可以获取锁，不会阻碍正常业务的执行。

## 可重入

- Redisson 支持可重入锁，持有锁的线程可以多次获取同一把锁而不会被阻塞。具体是利用 Redis 中的哈希结构，哈希中的 key 为线程 ID，Field为`UUID + threadId` ，value为锁的**重入次数**。

- 如果重入则 `value +1`，如果释放则 `value -1`，减到 0 说明锁被释放了，则 `del` 锁。

	- Redisson 底层使用了 **`HSETNX`** 而不是 `SETNX`。
	- 为什么内层字段要拼接 UUID + threadId？
		- 如果是分布式（集群）环境下，线程id可能会重复，所以要拼接；
		- 同时避免锁id重复导致的误删

# 分布式锁-ZooKeeper实现

- 在 ZK 中创建一个持久节点作为“锁目录”，例如 `/locks`。    
- **获取锁：** 每个想获取锁的客户端，都在 `/locks` 目录下创建一个**临时顺序节点**，例如 `/locks/lock_0000000001`、`/locks/lock_0000000002`。    
- 创建完成后，客户端获取 `/locks` 目录下的所有子节点，并判断自己创建的节点序号是否是**最小**的。    
    - 如果是最小的，则成功获得锁。        
    - 如果不是最小的，则获取锁失败。客户端会找到比自己序号**小一位**的那个节点，并对其设置一个 **Watch (监听)**。
- **释放锁：** 持有锁的进程完成业务操作后，**关闭与 ZK 的会话**即可。因为创建的是**临时节点**，会话一关闭，该节点就会被 ZK **自动删除**。    
- **唤醒等待者：** 当持有锁的节点被删除后，之前监听它的那个客户端会收到 ZK 的通知。该客户端会再次检查自己是否是当前最小的节点，如果是，则获得锁。这个过程形成了一个公平的排队等待机制。
- **优点：**    
    - **可靠性极高**：基于 ZAB 协议保证了强一致性，锁服务非常稳健。        
    - **天然解决死锁**：临时节点机制确保客户端崩溃后锁能自动释放。        
    - **公平锁与等待队列**：顺序节点实现了先到先得的公平排队。        
    - **高可用**：ZK 本身就是集群部署。        
- **缺点：**    
    - **性能不如 Redis**：每次操作都需要在 ZK 集群内进行数据同步，性能相对较低。        
    - **架构复杂**：需要独立维护一个 ZK 集群，增加了系统复杂度。

# 分布式锁-etcd实现
是一个键值存储仓库，但功能很多，
- 简单：基于 HTTP+JSON 的 API 让你用 curl 就可以轻松使用。
- 安全：可选 SSL 客户认证机制。
- 快速：每个实例每秒支持一千次写操作。
- 可信：使用 Raft 算法充分实现了分布式。

分布式锁**核心思想：** 利用 `etcd` 的 **Lease (租约)** 机制和 **Key 的 Revision (版本号)**，结合 **Watch (监听)** 机制、**强一致性**（Raft保证）来实现。
Ectd实现分布式锁基本实现原理为：
1. 在ectd系统里创建一个key
2. 如果创建失败，key存在，则监听该key的变化事件，直到该key被删除，回到1
3. 如果创建成功，则认为我获得了锁
**1. 核心机制**
- **租约**：etcd 中可以对一个 Key-Value 绑定一个租约。租约本质上是一个有时间期限的合同。合同到期后，Key-Value 将被自动删除。这实现了 **锁的自动释放**，避免了因为客户端崩溃而导致的死锁。客户端可以不断 **续约** 来保持锁。
- **事务**：etcd 提供基于 **MVCC** 的多版本并发控制事务。事务条件 `txn.If` 和多个操作 `txn.Then` 是原子性的。这是实现 **锁的互斥性** 和 **避免惊群效应** 的关键。
**2. 加锁流程**
假设多个客户端要竞争一个名为 `/lock/mylock` 的锁。
1. **创建租约**：客户端首先创建一个租约，并设置一个 TTL（例如 10 秒）。这是锁的默认持有时间。    
2. **发起事务性 PUT 请求**：客户端发起一个 etcd 事务，这个事务包含以下核心逻辑： 
    - **条件**：检查 `/lock/mylock` 这个 Key 是否存在（`CreateRevision` 是否为 0）。        
    - **成功分支**：如果 Key **不存在**（即 `CreateRevision == 0`），则执行 `Then`：将 Key `/lock/mylock` 的值设置为客户端的唯一标识（如 `host-1`），并绑定上一步创建的租约。
    - **失败分支**：如果 Key **已经存在**（即 `CreateRevision != 0`），则执行 `Else`：获取当前 Key 的信息。
3. **处理事务结果**：    
    - **成功**：如果事务执行成功，说明客户端**成功创建了 Key**，即**抢锁成功**。客户端需要定期向租约发送 **KeepAlive** 请求来续期，防止锁过期。        
    - **失败**：如果事务执行失败，说明已经有其他客户端创建了这个 Key（即锁已被占用）。当前客户端**抢锁失败**。        
**3. 解决“惊群效应”和等待**
单纯的“失败-重试”会导致所有等待的客户端同时去抢锁，产生 **惊群效应**，给 etcd 集群带来巨大压力。etcd 官方推荐的最佳实践是使用 **前缀 + 顺序 Key** 来实现 **公平锁**。
4. **创建顺序 Key**：客户端不是直接创建 `/lock/mylock`，而是创建一个顺序 Key，例如 `/lock/mylock/lease_id_1`。
5. **列出并排序**：客户端列出 `/lock/mylock` 前缀下的所有 Key，这些 Key 会按创建顺序自动排序。
6. **判断自身位置**：
    - 如果自己创建的 Key 是序列中的**第一个**，则成功获得锁。
    - 如果不是第一个，则**监听排在它前面的那个 Key 的删除事件**。
7. **等待和监听**：当前一个 Key 被删除（意味着前一个持有者释放了锁），客户端会被通知到，然后再次执行步骤 3 的判断。这样就形成了一个有序的、逐个唤醒的等待队列，彻底避免了惊群效应。  
8. 释放锁**
- **正常释放**：客户端完成业务逻辑后，主动 **删除** 自己创建的那个 Key。
- **异常释放**：如果客户端崩溃，无法续约，租约到期后 etcd 会**自动删除**绑定的 Key，锁被释放。这是 etcd 分布式锁最核心的**安全特性**。

# ZooKeeper 有哪些使用场景？
- **分布式协调：** 这个其实就是 zk 很经典的一个用法，简单来说，就好比，你系统 A发送个请求到 mq，然后 B消费了之后处理。那 A系统如何指导 B系统的处理结果？用 zk 就可以实现分布式系统之间的协调工作。A系统发送请求之后可以在 zk 上对某个节点的值注册个监听器，一旦 B系统处理完了就修改 zk 那个节点的值，A立马就可以收到通知，完美解决。
- **分布式锁**：对某一个数据联系发出两个修改操作，两台机器同时收到请求，但是只能一台机器先执行另外一个机器再执行，那么此时就可以使用 zk 分布式锁，一个机器接收到了请求之后先获取 zk 上的一把分布式锁，就是可以去创建一个 znode，接着执行操作，然后另外一个机器也尝试去创建那个 znode，结果发现自己创建不了，因为被别人创建了，那只能等着，等等一个机器执行完了自己再执行。
- **配置信息管理**：zk 可以用作很多系统的配置信息的管理，比如 kafka，storm 等等很多分布式系统都会选用 zk 来做一些元数据，配置信息的管理，包括 dubbo 注册中心不也支持 zk 么。
- HA高可用性：这个应该是很常见的，比如 hdfs，yarn 等很多大数据系统，都选择基于 zk 来开发 HA高可用机制，就是一个重要进程一般会主备两个，主进程挂了立马通过 zk 感知到切换到备份进程

# 分布式Session

**<mark class="hltr-green">什么是 Session？</mark>**
首先，HTTP 协议本身是无状态 (Stateless) 的。这意味着服务器默认不会记得你上一次请求做了什么。为了让网站能“记住”用户的登录状态、购物车内容等信息，Session 机制应运而生。
**单机工作流程：**
1. 用户第一次访问网站并登录。    
2. 服务器创建一个 `Session` 对象，里面存储了该用户的信息（如 `userId: 123`），并把这个 `Session` 保存在自己的**内存**里。    
3. 服务器生成一个唯一的 `SessionID`，通过 Cookie 发送给用户的浏览器。    
4. 用户再次访问时，浏览器会自动带上这个 `SessionID` Cookie。    
5. 服务器根据 `SessionID`，在自己的内存中找到对应的 `Session` 对象，从而知道“哦，是老朋友 `userId: 123` 回来了”。

**<mark class="hltr-green">为什么它在分布式环境下会出问题？</mark>**
变成了多机，并在前面放了一个负载均衡器
- 用户的第一次请求（登录）被负载均衡器发往了 **服务器 A**。服务器 A 在自己的内存里创建了 Session。    
- 用户的第二次请求（比如访问购物车），被负载均衡器发往了 **服务器 B**。    
- 服务器 B 拿到 `SessionID` 后，在自己的内存里找了一圈，发现“查无此人”。于是，它会认为用户未登录，强制用户重新登录。    

这就是**分布式 Session 问题**：<mark class="hltr-red">Session 数据被孤立在了单个服务器的内存里，无法在集群中共享。</mark>

方案——<mark class="hltr-red">Centralized Session Storage (会话集中存储)</mark>
- **核心思想：** 将所有 Session 数据从应用服务器的内存中剥离出来，统一存储到一个**外部的、共享的、高性能的存储服务**中。    
- **实现方式：** 所有服务器都不在自己内存里存 Session 了。当需要读写 Session 时，它们都去访问这个中央存储服务。常用的中央存储有 **Redis**、Memcached 等内存数据库。
- **优点：**    
    - **高可用**：应用服务器本身变成了**无状态 (Stateless)** 的，任何一台宕机都不会影响 Session 数据。中央存储服务自身可以做高可用集群。        
    - **高扩展性**：可以轻松地水平扩展应用服务器的数量，而没有额外的 Session 同步开销。        
    - **简单可靠**：这是目前业界最成熟、最主流、最推荐的方案。        
- **缺点：**    
    - **增加了一次网络调用**：每次请求都需要访问一次中央存储，会带来微小的网络延迟。        
    - **引入了新的依赖**：需要额外部署和维护一个高可用的中央存储集群。

基于【会话集中存储】，具体落地有两种方案Tomcat + redis 和 Spring Session + redis
<mark class="hltr-yellow">方案一：Tomcat + Redis (容器层解决方案)</mark>
这种方案的核心思想是：**对应用代码完全透明，通过“改造” Web 容器 (Tomcat) 本身来实现 Session 的外部存储。**
- **核心组件：`Tomcat-Redis-Session-Manager`** 这是一个第三方的开源项目，它提供了一个自定义的 Tomcat `Session Manager` 实现。
- **实现原理与步骤：**    
    1. **替换 Tomcat 的“大脑”**：Tomcat 内部有一个默认的 `Manager` 组件，负责创建、管理和销毁 `HttpSession` 对象，默认情况下，它把 Session 存在服务器内存里。        
    2. **“手术”开始**：我们要做的就是替换掉这个默认的 `Manager`。        
        - 下载 `tomcat-redis-session-manager.jar` 及其依赖的 `jedis.jar` 等库文件。            
        - 将这些 `.jar` 文件放入 Tomcat 的全局 `lib` 目录下。            
        - 修改 Tomcat 的配置文件 `context.xml`（可以修改全局的，也可以修改单个应用的）。            
    3. **修改配置文件 `context.xml`：**        
```xml
<Valve className="com.orange.redis.RedisSessionHandlerValve" />
<Manager className="com.orange.redis.RedisSessionManager"
         host="your-redis-host"
         port="6379"
         database="0"
         maxInactiveInterval="60" />
```
这段配置告诉 Tomcat：“别用你自己的内存 Session 管理器了，改用我指定的这个 `RedisSessionManager`。以后所有 `getSession()`、`setAttribute()` 的操作，都请这位新经理代理，让他去和 Redis 服务器打交道。”
- **优点：**
    - **对应用透明**：你的 Web 应用代码**完全不需要做任何修改**。你依然使用标准的 `javax.servlet.http.HttpSession` API，应用本身根本不知道 Session 已经存到了 Redis 里。        
- **缺点：**    
    - **与容器强耦合 (Container Lock-in)**：这是它最致命的弱点。你的解决方案完全绑定在了 Tomcat 上。如果你想把应用换到另一个 Web 容器，比如 Jetty 或者 Undertow，这套方案就完全失效了，你必须重新寻找并配置适用于新容器的插件。        
    - **配置和运维复杂**：你需要手动管理 Tomcat 的底层配置和依赖的 jar 包，这在自动化部署和 Docker 容器化时代显得非常笨拙和不便。

<mark class="hltr-yellow">方案二：Spring Session + Redis (框架层解决方案)</mark>
这种方案的核心思想是：**不“改造”底层的 Web 容器，而是在应用框架 (Spring) 层面进行“拦截”和“代理”，将 Session 操作重定向到 Redis。**
- **核心组件：`Spring Session` 框架** 这是 Spring 全家桶中的一员，专门用于解决 Session 管理问题。    
- **实现原理与步骤：**    
    1. **添加依赖**：在你的 Spring Boot 项目的 `pom.xml` 或 `build.gradle` 文件中，只需添加 `spring-boot-starter-data-redis` 和 `spring-session-data-redis` 两个依赖。        
    2. **修改配置文件**：在 `application.properties` 或 `application.yml` 中配置 Redis 的地址、端口等信息。
```Properties
spring.redis.host=your-redis-host
spring.redis.port=6379
# 指定 Spring Session 使用 redis 来存储
spring.session.store-type=redis
```
3. **启用支持**：在你的 Spring Boot 启动类上加上一个注解 `@EnableRedisHttpSession`。
**完成了！** Spring Boot 的自动配置机制会完成所有剩下的魔法。它会自动创建一个名为 `SessionRepositoryFilter` 的过滤器。这个过滤器会拦截所有的 HTTP 请求，将原生的 `HttpServletRequest` 对象包装成一个自定义的对象，当你调用这个包装对象的 `getSession()` 方法时，它返回的不再是 Tomcat 原生的 `HttpSession`，而是一个由 Spring Session 管理的、背后由 Redis 存储的 Session 实现。
- **优点：**    
    - **容器无关 (Container Agnostic)**：这是**最大的优势**。因为所有逻辑都在 Spring 框架内部实现，所以你的应用（打成一个可执行 jar 包）可以无缝地部署在 Tomcat, Jetty, Undertow, Netty 等任何支持 Servlet 规范的容器上，Session 机制都能完美工作。这与云原生和微服务的理念高度契合。        
    - **与 Spring 生态深度集成**：可以享受到 Spring 框架带来的各种便利，如配置简单、事务管理、事件通知等。        
    - **易于管理和部署**：所有配置都在项目代码内部，随着项目一起打包和版本控制，非常适合自动化 CI/CD 流程。        
- **缺点：**    
    - **与框架耦合**：你的应用必须是基于 Spring 或 Spring Boot 构建的。当然，对于目前绝大多数 Java 应用来说，这并不是一个问题。

# 分布式事务
要理解各种解决方案，我们先要了解两种不同的设计哲学。
- **刚性事务 (Strong Consistency):** 追求强一致性，严格遵循 ACID。代表理论是 **2PC (Two-Phase Commit, 两阶段提交)**。
- **柔性事务 (Eventual Consistency):** 追求最终一致性，遵循 **BASE** 理论。它不要求数据在任何时刻都保持强一致，但承诺在一段时间后，数据最终会达到一致状态。
两种方案：
<mark class="hltr-pink">XA 又叫 2PC方案，两阶段提交</mark>
这是实现分布式强一致性的经典算法，很多传统方案（如 XA）都基于它。它引入一个**协调者 (Coordinator)** 和多个**参与者 (Participants)**。
1. **第一阶段 (准备阶段 / Voting Phase):**    
    - 协调者向所有参与者发送“准备”请求。        
    - 参与者执行本地事务操作，锁定资源，但不真正提交。        
    - 如果参与者能成功，就向协调者回复“Yes”；否则回复“No”。        
2. **第二阶段 (提交阶段 / Commit Phase):**    
    - **情况一：** 如果协调者收到了所有参与者的“Yes”，它就向所有参与者发送“Commit”请求。参与者收到后，完成本地事务的提交，释放资源。        
    - **情况二：** 如果协调者收到任何一个“No”，或者有参与者超时未响应，它就向所有参与者发送“Rollback”请求。参与者收到后，回滚本地事务，释放资源。      
- **优点：** 原理简单，实现了数据的强一致性。    
- **缺点 (致命的)：**    
    - **同步阻塞：** 在整个过程中，所有参与者都在等待，资源被长时间锁定，性能极差。        
    - **协调者单点故障：** 如果协调者宕机，所有参与者都会被阻塞，不知所措。        
    - **数据不一致风险：** 如果在第二阶段，协调者发出 Commit 后宕机，部分参与者收到了消息并提交，另一部分没收到，就会导致数据不一致。        
**结论：** 由于性能和可用性问题，2PC 方案在高性能、高可用的互联网应用中**几乎不被采用**。

<mark class="hltr-pink">TCC (Try-Confirm-Cancel)，柔性方案，主流</mark>
这可以看作是**应用层面的 2PC**，将事务的两个阶段完全交由业务代码来实现。每个服务都需要提供三个接口：
1. **Try:** 尝试执行业务。完成所有业务检查，并**预留 (Reserve)**必要的业务资源。例如，冻结用户的 100 元钱。    
2. **Confirm:** 确认执行业务。使用 Try 阶段预留的资源，**真正执行**业务。例如，从冻结的 100 元中扣款。    
3. **Cancel:** 取消执行业务。**释放** Try 阶段预留的资源。例如，解冻用户的 100 元钱。  
- **流程：**    
    - 一个全局的事务管理器先调用所有相关服务的 `Try` 接口。        
    - 如果所有 `Try` 都成功，管理器就依次调用所有服务的 `Confirm` 接口。        
    - 如果任何一个 `Try` 失败，管理器就依次调用所有**已成功的 Try** 服务的 `Cancel` 接口。        
- **优点：** 性能比 2PC 好（资源锁定时间短），开发者可以精细控制资源。    
- **缺点：** **代码侵入性极强**。你需要为每个业务操作都编写 `Try-Confirm-Cancel` 三个方法，开发成本很高。

<mark class="hltr-pink">Saga 模式 (长事务解决方案) ，柔性方案，【最常用】</mark>
Saga 是目前**微服务架构下最主流**的分布式事务解决方案。
- **核心思想：** 将一个大的分布式事务，拆分成一系列小的**本地事务**。每个本地事务由一个服务自己完成。如果其中某个步骤失败，Saga 会执行一系列**补偿事务 (Compensating Transactions)**来撤销之前已经成功执行的操作。
- **实现方式：**    
    - **Choreography (编排式):** 去中心化模式。每个服务在完成自己的本地事务后，发布一个**事件 (Event)**。下一个服务监听到这个事件后，开始执行自己的本地事务。        
        - **比喻：** 一场**接力舞**。每个舞者跳完自己的部分后，给下一个舞者一个信号，下一个舞者接上。如果有人跳错了，他需要发出一个“跳错了”的信号，前面的舞者听到后需要倒着跳回来。            
    - **Orchestration (协调式):** 中心化模式。有一个**中央协调器 (Orchestrator)**，它负责调用各个服务来执行本地事务。它像状态机一样，知道整个流程应该如何进行，如果某个步骤失败，由它负责调用补偿事务。        
        - **比喻：** 一位**乐队指挥**。指挥家按顺序指挥小提琴手、大提琴手、钢琴手演奏。如果钢琴手弹错了，指挥家会命令小提琴手和大提琴手停止或执行反向操作。            
- **优点：**    
    - **性能好，无阻塞：** 每个服务只执行本地事务，吞吐量高。        
    - **服务松耦合**（尤其是编排式）。        
- **缺点：**    
    - **不保证隔离性：** 在事务执行的中间状态，数据可能是不一致的。        
    - **补偿逻辑复杂：** 为每个操作设计健壮的补偿事务可能非常困难（例如，发短信操作如何补偿？）。
