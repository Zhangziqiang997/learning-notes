# 组成部分
Kubernetes，简称K8s，是一个开源的容器编排平台，其主要目的是自动化应用的部署、扩展以及管理。它可以帮助你管理容器化的应用，使得服务在不同的主机之间进行调度，从而实现负载均衡、高可用和自动伸缩。

<mark class="hltr-green">Kubernetes的主要组件及其作用可以分为以下几部分：</mark>
控制平面负责控制和管理各个 Node，而 Node 则负责实际运行各个应用服务。

1）**Master节点组件 / 控制平面**
- **API Server**：以前我们需要登录到每台服务器上，手动执行各种命令，现在我们只需要调用 k8s 的提供的 api 接口，就能操作这些服务资源，这些接口都由 **API Server** 组件提供。
	- 是整个 K8s 集群的唯一入口。它是集群的门户，其他组件和用户通过它与集群交互。我们用户通过 `kubectl` 命令对集群下达的指令，都必须经过 `API Server`。它负责对请求进行认证、授权和校验。
- **etcd**：一个高可用的键值数据库，是 K8s 集群的**唯一状态来源**。它完整地存储了整个集群的状态，包括有哪些节点、运行了哪些 Pod、配置是什么等等。Kubernetes的其他所有组件都是**无状态**的，它们通过查询`etcd`来获取集群的当前状态，**etcd是有状态的**。
- **Controller Manager**：找到服务器后，以前我们会手动创建，关闭服务，现在这部分功能由 Controller Manager（控制器管理器）来负责。
	- 这是 K8s 自动化能力的核心。它内部运行着多个控制器（如 `Node Controller`, `Replication Controller` 等），这些控制器像一个个不知疲倦的监控员，持续地将集群的“当前状态”**调整为你在 `etcd` 中记录的**“期望状态”**。
- **Scheduler**调度器：它的工作很简单但至关重要：**决定新创建的 Pod 应该被放置到哪个 Worker Node 上运行**。它会根据一系列复杂的算法来做决策，比如节点的资源剩余情况、Pod 的资源需求、<mark class="hltr-grey">亲和性/反亲和性</mark>策略等。
--- 
- **无状态**：组件本身不保存需要持久化的数据或会话信息。每一次请求都是独立的，不依赖于之前的任何请求。 K8s中：
	- scheduler 不需要记住它上次把Pod调度到了哪里。
	- controller-manager 不需要记住它正在管理哪些Pod。
	- apiserver 更不会记住任何用户会话。
	- 所有这些组件在需要做决策时，都会去一个<mark class="hltr-yellow">唯一的地方——即 etcd ——查询集群的当前状态</mark>。它们根据从etcd读到的“最新快照”来做出判断和行动。
	- **作用：**
		1. **高可用与易恢复**：如果一个无状态组件（如调度器）的实例崩溃了，你可以立刻启动一个新的实例。这个新实例启动后，只需要连接上`etcd`，就能立刻获取到集群的全部状态，并无缝接替工作。它不需要从复杂的本地数据中恢复。
		2. **水平扩展**：因为实例之间没有需要同步的内部状态，你可以轻松地运行多个`kube-apiserver`实例来分担负载，它们都指向同一个`etcd`后端。
- **亲和性/反亲和性规则**：是指导Kubernetes调度器如何将Pod**放置**在一起或**分散**开的高级调度策略。
	- **亲和性**：将某个Web应用Pod和它频繁访问的缓存Pod（如Redis）调度到同一台机器或同一个可用区，可以降低网络延迟。
	- **反亲和性：**
		- **高可用：** 这是最常见的用法。确保同一个应用的多个副本不会被调度到同一个节点上。这样，即使一个节点宕机，你的应用在其他节点上仍有副本可以继续服务。
		- **资源隔离：** 避免两个资源消耗大的Pod挤在同一台机器上，导致资源争抢。

2）**Node节点组件 / 工作节点**：
首先梳理一下层级关系：`Node` >> `Pod` >> `Container`
- **Node**：是计算资源的**物理基础**，提供了 CPU、内存、存储和网络能力。一个 K8s 集群由一个或多个 `Node` 组成。你的所有应用最终都是在这些 `Node` 上运行的。
- **Pod**：`Pod` 是 K8s 中**最小的部署和调度单元**。Pod 运行在 Node 之上。调度器 (`Scheduler`) 的核心工作，就是决定将一个 `Pod` 分配到哪个最合适的 `Node` 上去运行。
- **Container**：`Container`是真正**运行你的应用程序**的地方。它打包了你的应用代码及其所有的依赖。Pod 包含一个或多个 Container。同一个 Pod 里的所有容器：
	- **共享网络空间**：它们可以用 `localhost` 互相通信，就像同一套房里的不同房间共享同一个 Wi-Fi。	    
	- **共享存储卷 (Volumes)**：Pod 可以定义存储卷，然后挂载给内部的所有容器共享使用，就像套房里的储藏室可以被所有房间的人共用。
---
- **Kubelet**：前面提到控制平面会用 **Controller Manager** （通过 API Server）控制 Node 创建和关闭服务，那 Node 也得有个组件能接收到这个命令才能去做这些动作，这个组件叫 **kubelet**，它主要负责管理和监控 Pod。
	-  负责生命周期管理。它会定期从 `API Server` 接收该节点上应该运行哪些 Pod 的指令，然后确保这些 Pod 里的容器被正确地创建、运行和保持健康。
- **Kube-proxy**：它负责 Node 的网络通信功能，有了它，外部请求就能被转发到 Pod 内。负责实现 K8s 的 `Service` 网络概念。它在每个节点上维护网络规则，处理集群内部服务之间的通信以及从外部到集群内部的流量路由和负载均衡。
- **Container Runtime**：运行容器的软件，如Docker、containerd等。这是真正负责**运行容器的底层软件**。Docker 是最著名的一种，但 K8s 也支持其他符合 CRI (Container Runtime Interface) 标准的运行时，如 `containerd` 或 `CRI-O`。
---

**控制平面和 Node** 共同构成了一个 **Cluster**，也就是**集群**。
在公司里，我们一般会构建多个集群, 比如测试环境用一个集群，生产环境用另外一个集群。
同时，为了将集群内部的服务暴露给外部用户使用，我们一般还会部署一个入口控制器，比如 **Ingress 控制器（比如 Nginx）**，它可以提供一个入口让外部用户访问集群内部服务。

---

3）**高层组件 / API对象 / 资源**：
是K8s提供给用户（开发者/运维）使用的、更高层次的抽象
- **Pod**：Kubernetes中最小的部署单位，通常包含一个或多个容器，运行在同一个网络命名空间，使用相同的存储卷。是这个操作系统里运行的**一个或多个紧密相关的进程**。
	- **亲密性**：Pod内的容器共享网络命名空间（可以用`localhost`通信）和存储卷，它们被紧密地部署在同一台主机上。
- **Deployment**：一个**定义Pod期望状态的管理器**。
	- 它描述了：我需要运行什么样的Pod（模板），需要运行多少个副本（Replicas）。
	- 声明式管理Pod和ReplicaSet，提供更新和回滚机制，确保Pod达到期望的状态。
	1. **定义期望状态：** 你在 `Deployment` 的 YAML 文件里声明：“我需要我的应用 `my-app` 的 `v1` 版本，运行 3 个副本”。    
	2. **保持期望状态 (自愈)：** `Deployment` 会创建一个 `ReplicaSet`，由 `ReplicaSet` 确保集群中始终有 3 个 `my-app` 的 Pod 在运行。如果某个 Pod 挂了，`ReplicaSet` 会立刻再创建一个新的。    
	3. **管理应用发布 (滚动更新 - Rolling Update)：** 当你想把应用从 `v1` 升级到 `v2` 时，你只需要修改 `Deployment` 的镜像版本。它会非常智能地、**逐个地**用 `v2` 版本的 Pod 替换 `v1` 版本的 Pod，从而实现**零停机时间**的平滑升级。    
	4. **支持回滚 (Rollback)：** 如果发现 `v2` 版本有问题，一条命令就能让 `Deployment` 将应用状态回滚到之前的 `v1` 版本。
- **Service**：一个**定义了一组Pod访问策略的抽象层**。为一组功能相同的 Pod 提供一个统一、稳定的访问入口
	- **为什么需要它** `Deployment` 保证了 Pod 的数量和版本，但 Pod 本身是“短暂”的，它们可能随时被销毁重建，每次重建后 IP 地址都会改变。 提供Pod的永久访问方法，通过标签选择器将多Pod指定为一个服务，实现负载均衡。
	- **工作原理：** `Service` 通过**标签选择器 (Label Selector)** 来识别它应该代理哪些 Pod。比如，你给所有 `my-app` 的 Pod 都打上 `app: my-app` 的标签，然后创建一个 `Service`，让它选择所有带这个标签的 Pod。
<mark class="hltr-cyan">总结：</mark>
- 通过 `Deployment` 来**部署和管理**你的应用 Pod（定义版本、副本数、更新策略）。
- Pod 是你应用容器的**实际载体**。    
- 你通过 `Service` 来**暴露**你的应用（提供一个稳定的访问入口，并做负载均衡）。

- **Namespace**：**集群内部的虚拟集群**，用于在物理集群中划分出多个逻辑上隔离的空间。
	- 提供逻辑隔离机制，用于同一个集群中资源的分组和隔离。
- **ConfigMap和Secret**：都是用来将**配置数据**与**应用镜像**解耦的API对象。
	- 用于存储配置数据和敏感信息，分别以非加密和加密方式存储。
		- **ConfigMap**：用于存储**非机密**的配置数据，如配置文件、环境变量、命令行参数。
		- **Secret**：用于存储**敏感**信息，如密码、OAuth令牌、SSH密钥。数据默认以Base64编码（**注意：这不是加密！**）。
	- **为什么需要它们**：避免将配置信息硬编码在Docker镜像中，从而实现**一套镜像，多环境部署**。

# 部署服务

首先我们需要编写 **YAML 文件**，在里面定义 Pod 里用到了哪些镜像，占用多少内存和 CPU 等信息。  
然后使用 kubectl 命令行工具执行 `kubectl apply -f xx.yaml` ，此时 kubectl 会读取和解析 YAML 文件，将解析后的对象通过 API 请求发送给 Kubernetes 控制平面内 的 **API Server**。 API Server 会根据要求，驱使 **Scheduler** 通过 **etcd** 提供的数据寻找合适的 **Node**， **Controller Manager** 会通过 API Server 控制 Node 创建服务，Node 内部的 **kubelet** 在收到命令后会开始基于 **Container runtime** 组件去拉取镜像创建容器，最终完成 **Pod** 的创建。

# 调用服务

以前外部用户小明，直接在浏览器上发送 http 请求，就能打到我们服务器上的 Nginx，然后转发到部署的服务内。  
用了 k8s 之后，外部请求会先到达 k8s 集群的 Ingress 控制器，然后请求会被转发到 k8s 内部的某个 Node 的 **Kube Proxy** 上，再找到对应的 pod，然后才是转发到内部**容器服务**中，处理结果原路返回，到这就完成了一次服务调用。

# 滚动更新
滚动更新是 Kubernetes Deployment 控制器的一种自动化策略，它通过**逐步用新版本的 Pod 替换旧版本的 Pod**，来实现**零停机部署**。

1. **命令式**操作，你直接告诉 Kubernetes **要做什么**（“把镜像改成v2”）。
	`kubectl set image deployment/my-deployment my-container=my-image:v2`

2. **声明式**操作，你告诉 Kubernetes **你期望的状态是什么**（“这是我的完整配置，请确保集群状态与之匹配”）。
	`kubectl apply -f deployment-v2.yaml`

| 特性       | `kubectl set image deployment/my-deployment ...` | `kubectl apply -f deployment.yaml`                             |
| -------- | ------------------------------------------------ | -------------------------------------------------------------- |
| **操作模式** | **命令式**                                          | **声明式**                                                        |
| **作用对象** | **单个字段**（通常是镜像版本）                                | **整个资源对象**（Deployment, Service等）                               |
| **配置来源** | 命令行参数                                            | YAML/JSON 配置文件                                                 |
| **版本控制** | **难以追溯**。命令记录在历史中，但无法直观看到这次更改的完整上下文。             | **天然支持**。配置文件可以存入Git，每次变更都有记录、可审查、可回滚。                         |
| **协作影响** | **不利于协作**。其他人不知道你通过命令行修改了配置，容易导致环境不一致。           | **利于协作**。团队共享同一份配置文件，是**单一可信来源**。                              |
| **幂等性**  | 是。多次执行结果一致。                                      | 是。Kubernetes 会进行差异合并，确保最终状态与文件一致。                              |
| **典型场景** | **快速调试、紧急热修复、临时验证**。                             | <mark class="hltr-pink">**CI/CD流水线、 GitOps工作流、生产环境部署**。</mark> |

# 回滚

指定回滚到某个特定的修订版本
	`kubectl rollout undo deployment/<deployment-name> --to-revision=<revision-number>`

# K8s 如何配置资源配额

资源配额是Kubernetes中一种在Namespace命名空间级别限制资源消耗的机制，
确保一个命名空间内的所有资源对象不会集体超支
资源配额可以限制和管理命名空间中可用的资源，例如 CPU、内存、持久存储等，从而防止某个命名空间占用过多的资源。

	资源配额就是房东给每个房间安装的**总水表、总电表**和**房间人数上限**。
	 它不关心房间里是A一个人用电，还是A、B、C三个人合起来用电；它只关心这个房间的**总用水量、总用电量和总人数**不能超过限额。这样就能防止某个房间（命名空间）耗尽整栋公寓（集群）的资源，导致其他房间断电断水。

1. 第一步，创建命名空间：
	`kubectl create namespace mynamespace`
2. 创建资源配额 yaml 文件（resource-quota.yaml）：
``` YAML
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-resources
  namespace: my-team # 指定生效的命名空间
spec:
  hard:
    # 计算资源配额
    requests.cpu: "10"     # 该命名空间内所有Pod的CPU请求总和不能超过10核
    requests.memory: 20Gi  # 该命名空间内所有Pod的内存请求总和不能超过20GiB
    limits.cpu: "20"       # 该命名空间内所有Pod的CPU限制总和不能超过20核
    limits.memory: 40Gi    # 该命名空间内所有Pod的内存限制总和不能超过40GiB

    # 对象数量配额
    pods: "50"             # 该命名空间内最多只能有50个Pod
    services: "10"         # 最多只能创建10个Service
    secrets: "20"          # 最多只能创建20个Secret
    configmaps: "20"       # 最多只能创建20个ConfigMap
    persistentvolumeclaims: "10" # 最多只能创建10个PVC
```
3. 应用配额
	`kubectl apply -f my-quota.yaml`
