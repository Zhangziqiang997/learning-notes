## 统一镜像，多入口点

``` yaml
image: mineru-vllm:latest        # 所有服务使用同一镜像
entrypoint: mineru-vllm-server   # 不同的入口点
entrypoint: mineru-api
entrypoint: mineru-gradio
```
“统一镜像，多入口点”模式，就是指将一个项目的所有相关服务、代码和依赖全部打包进一个单独的、自包含的 Docker 镜像中。然后，在运行时（例如通过 Docker Compose 或 Kubernetes），根据需要指定不同的启动命令 (Entrypoint/Command)，来激活这个镜像中的特定功能，使其扮演不同的服务角色。
优点：
1. **极大地简化了 CI/CD (持续集成/持续部署)**    
    - **单一构建流程**: 你只需要为整个项目维护**一个** `Dockerfile` 和**一套** CI/CD 流水线。无论哪个服务的代码发生变化，构建过程都是一致的。这大大降低了运维的复杂性。        
    - **单一交付产物**: 每次构建只产生**一个**镜像 `mineru-vllm:version-xyz`。这个镜像就是你所有服务的唯一“发行版”。管理、存储、扫描和分发这一个镜像，远比管理三个独立的镜像要简单得多。        
2. **保证了环境的绝对一致性**    
    - 这是最重要的优点之一。因为所有服务都源自同一个镜像，所以它们的操作系统、系统库、Python 依赖版本、CUDA 版本等基础环境是**完全一致**的。  
    - 这从根本上杜绝了那种最令人头疼的问题：“奇怪，API 服务能跑，但 Gradio 服务因为某个库的版本不对而崩溃了”。        
3. **促进了代码和资源的共享**    
    - 所有服务都可以无缝地共享镜像内的通用代码库、工具函数，甚至是模型文件。        
    - 在这个项目中，`mineru-api` 和 `mineru-gradio` 很可能都依赖于同一个模型加载和推理的底层代码。将它们打包在同一个镜像中，避免了重复打包这些共享资源，也简化了代码的管理。        
4. **简化了本地开发和测试**    
    - 开发者只需要 `docker build` 一次，就能在本地获得一个包含所有功能的环境。然后通过 Docker Compose，可以像搭积木一样，轻松地启动任意的服务组合进行联调测试。
