
#### 1. 基于慢查询日志排查出订单业务的慢SQL，根据执行计划发现没走索引，建立联合索引优化慢SQL，执行耗时从970ms降低到210ms。
**1. 发现问题：慢查询日志**
- **作用**：线上数据库服务会开启**慢查询日志**功能，并设置一个阈值（比如 `long_query_time = 1`，表示超过 1 秒的查询）。    
- **实践**：这个案例中，一条订单业务的 SQL 语句执行耗时 **970ms**，超过了阈值，因此被记录到了慢查询日志中。    

**2. 分析问题：执行计划 (`EXPLAIN`)**
- **作用**：通过对慢 SQL 使用 `EXPLAIN` 命令，可以查看它的**执行计划**。    
- **实践**：在这个案例中，`EXPLAIN` 的结果显示查询没有走索引。这通常体现在 `type` 字段为 `ALL`（全表扫描），或者 `key` 字段为 `NULL`。    
- **为什么没走索引？** 可能是因为：    
    - **查询条件没有合适的索引**。        
    - **查询条件对索引列进行了函数运算**，导致索引失效。        
    - **使用了 `LIKE '%...'` 这样的模糊查询**，无法利用索引。        

**3. 解决问题：建立联合索引**
- **作用**：根据 `EXPLAIN` 的分析结果，为 SQL 语句的查询条件建立**联合索引**。    
- **实践**：假设这条慢 SQL 是这样的： `SELECT * FROM orders WHERE user_id = ? AND order_status = ? ORDER BY create_time DESC;` 这条 SQL 的查询条件是 `user_id` 和 `order_status`，排序条件是 `create_time`。 为了让查询走索引，并且避免额外的排序开销，我们可以建立一个**联合索引**：`(user_id, order_status, create_time)`。    
    - **为什么是联合索引？**        
        1. `user_id` 和 `order_status` 是查询条件，将它们放在索引的最左边，可以利用**最左匹配原则**来快速过滤数据。            
        2. `create_time` 是排序条件，将其也包含在索引中，可以利用索引的有序性直接进行排序，**避免了 `Using filesort`** 这个耗时操作。            
        3. 如果查询只需要这几个字段，那么这个索引就实现了**索引覆盖**，连**回表**都不需要了，查询效率会达到最高。

#### 3. 使用Redis的bitmap位图来存储用户签到与补签信息，解决了使用数据库存储签到信息占用太多空间的问题，一人签到一个月只占用4字节。
1. **确定存储结构**
    - **键（Key）**：我们可以用 `signin:[用户ID]:[年-月]` 来作为键名，例如 `signin:123:2025-09`。这个键代表了用户 `123` 在 `2025年9月` 的签到记录。
    - **值（Value）**：这个值就是一个 Bitmap，它由一系列的 0 和 1 组成。
        
2. **存储签到**
    - 当用户在 `9月6日` 签到时，我们执行以下命令：
        代码段
```redis
SETBIT signin:123:2025-09 5 1
```
- `SETBIT` 命令的作用是：    
    - 将键 `signin:123:2025-09`        
    - 在**第 5 个位**上（索引从 0 开始，所以第 6 天的索引是 5）        
    - 设置为 `1`。        
- 如果用户在 `9月25日` 签到，我们执行 `SETBIT signin:123:2025-09 24 1`。

3. **统计签到天数**
    - 如果我们想知道用户在这个月签到了多少天，可以使用 `BITCOUNT` 命令：
```Java
BITCOUNT signin:123:2025-09
```
- 这个命令会返回 Bitmap 中所有 `1` 的数量，也就是用户签到的天数。

4. **补签**
    - 如果用户想在 `9月1日` 补签，我们只需执行：
```Java
SETBIT signin:123:2025-09 0 1
```

<mark class="hltr-cyan">为什么说“一人签到一个月只占用4字节”？</mark>

- **一个字节（Byte）** 等于 **8 位（bit）**。
- 一个月最多有 31 天，所以我们需要 31 个位来存储签到信息。
- `31`位 约等于 `4`个字节（31/8 约等于 3.875）。

相比于使用关系型数据库，为每个用户的每天签到都插入一条记录（可能包含用户 ID、日期、状态等字段），这种方式能够极大地节省存储空间，非常适合处理海量的签到数据。

#### 4. 开发AI大模型上传功能，模型权重较大，基于分片上传+断点续传+秒传实现大文件上传功能。
<mark class="hltr-purple"> **1. 分片上传（Chunked Upload）</mark>
这是大文件上传的基础，它将一个大文件在前端进行切片，然后将每个切片（chunk）作为独立的文件上传到服务器。
**工作流程：**
1. **前端切片**：前端使用 JavaScript 的 `File` 对象和 `Blob` 对象的 `slice()` 方法，将大文件按照预设的固定大小（如 1MB 或 4MB）切分成多个小文件。
2. **并发上传**：前端通过并发请求（例如使用 `Promise.all` 或线程池控制）将这些切片逐一上传到服务器。
3. **后端合并**：服务器接收到所有切片后，会根据前端传递的切片索引和文件标识符，将所有切片按照正确的顺序合并成一个完整的文件。
**优点：**
- **提高稳定性**：单个小文件上传失败时，只需重传这一个切片，而不是整个文件。
- **提升速度**：利用浏览器并发请求的优势，可以同时上传多个切片，显著提高上传速度。
- **减轻服务器压力**：单个请求的负载变小，减少了服务器的单次处理压力。
<mark class="hltr-purple">**2. 断点续传（Resumable Upload）**</mark>
断点续传是基于分片上传实现的，它使得文件上传过程可以在任何时候中断，并在之后从中断处继续上传。
**工作流程：**
1. **前端记录**：在切片上传前，前端会为整个文件生成一个**唯一标识符**（如文件的 MD5 值）。上传过程中，前端会记录已成功上传的切片索引。
2. **服务器检查**：在每次上传开始前，前端会向服务器发送一个请求，附带文件唯一标识符。服务器查询已上传的切片列表，并返回给前端。
3. **继续上传**：前端接收到服务器的响应后，只重新上传那些**未成功上传的切片**。
**优点：**
- **极大地提升用户体验**：解决了因网络中断、浏览器崩溃或用户主动暂停而导致上传失败的问题，用户可以随时继续上传
- **节省资源**：无需重复上传已成功的部分。
<mark class="hltr-purple">**3. 秒传（Instant Upload）**</mark>
秒传是利用文件的唯一标识符，实现快速上传的功能。
**工作流程：**
1. **前端计算哈希**：在文件上传前，前端使用 MD5 或 SHA-1 算法计算文件的完整哈希值（`hash`）。
2. **服务器查询**：前端携带这个 `hash` 值向服务器发送一个“秒传”请求。
3. **快速完成**：服务器在文件库中查询是否已经存在一个具有相同 `hash` 值的文件。
    - 如果**存在**，服务器直接返回成功，前端无需进行任何上传，整个过程在瞬间完成。
    - 如果**不存在**，服务器则返回失败，前端继续执行分片上传的流程。
**优点：
- **极致的用户体验**：对于已上传过的文件，上传时间几乎为零。
- **节省服务器存储空间**：避免了重复存储相同文件。

#### 5. 为了避免订单接口重复提交，使用防重Token+Redis保证接口的幂等性，避免出现重复数据。

核心思想：**在执行关键操作前，先通过一个唯一的“令牌”（Token）来检查，这个操作是否是第一次执行。**
三个步骤：
##### 1. 前端获取 Token
- **后端生成 Token**：当用户进入需要防止重复提交的页面（比如订单创建页面）时，前端会向后端发起请求。后端服务会生成一个**全局唯一的、不可预测的字符串**作为 Token（如 UUID），并将其存储到 **Redis** 中，同时为这个 Token 设置一个合理的过期时间（比如 10 分钟）。
	- 这里我们使用的关键命令是 `SET` 搭配 `EX` 选项
```Redis
# 命令格式: SET key value EX seconds
# 假设生成的Token是 "b16f261b-8a87-4a7b-a01d-52a1d2f07293"
SET idempotency:token:b16f261b-8a87-4a7b-a01d-52a1d2f07293 1 EX 600
```
- `idempotency:token:` 是一个统一的前缀，方便管理。Value设为 `1` 没有特殊含义，只是占位。`EX 900` 代表这个Key在900秒（15分钟）后会自动从Redis中删除。

- **后端返回 Token**：后端将这个 Token 返回给前端。
- **前端存储 Token**：前端将 Token 嵌入到表单中，或者存储在本地缓存（如 `localStorage`）中。
##### 2. 接口提交与校验
- **前端携带 Token**：当用户点击“提交订单”按钮时，前端会将这个 Token 作为请求参数（通常放在请求头或请求体中）一起发送给后端。
- **后端校验 Token**：后端接收到请求后，会做两件事情：
    1. **检查 Token 是否存在于 Redis 中**。如果不存在，说明这个 Token 已经过期或者已经被使用过，这次请求被认为是重复提交，直接拒绝并返回失败信息。
    2. **原子性删除 Token**。如果 Token 在 Redis 中存在，说明这是一个合法的首次请求。后端会立即**原子性地**将这个 Token 从 Redis 中删除。
```
# 命令格式: DEL key 
DEL idempotency:token:b16f261b-8a87-4a7b-a01d-52a1d2f07293
```
##### 3. 业务处理与响应
- **正常处理**：如果 Token 校验成功，后端就继续执行正常的业务逻辑，比如创建订单、扣减库存等。
- **返回结果**：业务处理完成后，后端返回成功响应给前端。由于 Token 已经被删除，即使用户再次提交，也无法通过校验，从而保证了接口的幂等性。

#### 6. 对于商品表百万条数据，使用联合索引+子查询分页方案优化深分页问题，执行耗时从1130ms降低到280ms。

有一个 `products` 表，有超过 200 万条数据，主键为 `id`。
**1. 未优化的深分页 SQL**
```sql
SELECT id, name, price FROM products ORDER BY create_time DESC LIMIT 100000, 10;
```
- **问题分析**：
    - `LIMIT 100000, 10` 的含义是，跳过前 100000 条记录，然后取接下来的 10 条。        
    - **MySQL 在执行时，必须先扫描并排序前面 100010 条记录**，然后丢弃掉前面的 100000 条，最后返回 10 条。        
    - 这种扫描和排序操作会消耗大量的 CPU 和 I/O 资源，导致查询性能急剧下降。当 `offset`（偏移量）越大，性能就越差。      

**2. 优化的分页 SQL**
优化的核心思想是：**避免数据库对大量数据的排序和扫描，只获取所需行的主键，然后通过主键去获取完整数据。**
**优化步骤：**
1. **子查询获取主键**：首先，通过一个子查询，只获取我们需要的 10 条记录的**主键（`id`）**。这个子查询的 `LIMIT` 和 `ORDER BY` 仍然会存在，但因为只查询主键，其扫描的数据量和返回的数据量都非常小。
```sql
SELECT id FROM products ORDER BY create_time DESC LIMIT 100000, 10;
```

2. **主查询获取完整数据**：然后，利用上一步得到的主键列表，通过 `WHERE IN` 或 `JOIN` 的方式，去查询完整的行数据。 
    ```SQL
    SELECT id, name, price FROM products
    WHERE id IN (
        SELECT id FROM products ORDER BY create_time DESC LIMIT 100000, 10
    )
    ORDER BY create_time DESC;
    ```
    - **为什么这更快？** 因为 `WHERE IN` 的查询是基于主键的，而**主键是聚簇索引**。通过主键查询单个或少量记录的效率非常高。        

**3. 联合索引优化**
为了让子查询本身也更快，我们需要在 `ORDER BY` 的字段上加上索引。
- 如果 `create_time` 上有索引，子查询的 `ORDER BY` 就会直接利用索引进行排序，**避免了额外的文件排序（`Using filesort`）**，大大提升了查询效率。    
- 为了更好的性能，你可以创建一个**联合索引**，例如 `(create_time, id)`。    
    - 这样做的好处是，子查询 `SELECT id FROM products ORDER BY create_time DESC LIMIT ...` 可以直接在**索引树**上完成，甚至不需要回表，实现了**索引覆盖**。


#### 7. 开发报表批量导出功能，使用线程池+CompletableFuture+EasyExcel异步批量导出，相比同步方案接口响应耗时从1780ms降低到310ms。

**1. 问题分析：同步方案的瓶颈**
传统的同步导出方案通常是一个单线程操作，其主要瓶颈在于：
1. **数据库查询**：当数据量很大时，一次性从数据库查询所有数据会非常耗时。    
2. **数据写入**：将大量数据写入 Excel 文件是一个 I/O 密集型操作。    
3. **网络阻塞**：在整个操作完成前，HTTP 连接会一直保持，用户会持续等待。   
当数据量达到几十万甚至上百万时，这种同步方案可能会导致接口超时，用户体验极差。

**2. 解决方案：异步批量导出**
这套方案通过引入多线程和异步编程，将耗时任务分解成更小的、可以并行处理的子任务，从而大大缩短了总的响应时间。
1. **线程池（ThreadPoolExecutor）**    
    - **作用**：提供了一组可复用的工作线程。避免了频繁创建和销毁线程的开销。        
    - **实践**：我们可以创建一个固定大小的线程池，用于执行数据查询和文件写入任务。        
2. **`CompletableFuture`**    
    - **作用**：是 Java 8 引入的异步编程工具，它让你可以将任务提交到线程池，然后获取一个“未来”（Future）的结果，而不需要同步等待。        
    - **实践**：        
        - **任务分解**：我们将数据查询任务分解为多个子任务，每个子任务负责查询一部分数据。            
        - **异步执行**：将这些子任务通过 `CompletableFuture.supplyAsync()` 提交到线程池中。            
        - **结果聚合**：使用 `CompletableFuture.allOf()` 方法，等待所有子任务执行完毕。当所有子任务都完成后，再进行下一步的文件合并和返回。            
3. **EasyExcel**    
    - **作用**：一个专业的 Excel 读写工具。       
	- 因为传统的Excel操作库（如Apache POI）是**DOM模式**，它会把整个Excel文件一次性加载到内存里再进行操作。当导出数据量达到几十万甚至上百万行时，非常容易引发**OOM（内存溢出）**。
	- 我们选择了阿里巴巴的EasyExcel，是基于**SAX模式**的，它采用流式读写，逐行处理数据，内存占用极低且恒定，不管导出多少数据，都不会有OOM的风险。
    - **实践**：        
        - **分批查询**：为了减少内存占用，我们不一次性查询所有数据，而是分批次查询。            
        - **分批写入**：`EasyExcel` 支持**分批写入**。我们可以让每个线程将查询到的数据分批写入一个临时文件，最后再将所有临时文件合并。            
        - **内存优化**：`EasyExcel` 采用流式写入，它不会一次性将所有数据加载到内存中，而是将数据流式地写入文件，极大地节省了内存。

#### 8. 使用消息唯一id + Redis保证Kafka消息的幂等性，避免消息重复消费问题。
这套方案通常由**生产者**和**消费者**配合完成，其核心思想是利用 Redis 的原子性操作，来记录和检查消息是否已经被消费过。
<mark class="hltr-blue">1. 生产者生成唯一ID</mark>
- **在消息发送前**，生产者为每条消息生成一个**全局唯一的消息 ID**。这个 ID 可以是一个 UUID，或者一个基于时间戳和序列号生成的分布式 ID。    
- **生产者将这个唯一 ID 嵌入到消息体中**，然后将消息发送到 Kafka 主题。   
<mark class="hltr-blue">2. 消费者消费消息</mark>
- **消费者接收消息**：消费者从 Kafka 拉取到消息后，首先解析出消息中的唯一 ID。    
- **原子性操作**：消费者在处理业务逻辑前，会先尝试向 **Redis** 写入这个消息 ID。通常使用 Redis 的 `SETNX`（`SET` if `N`ot e`X`ists）或 Lua 脚本来保证原子性。    
    - **`SETNX` 命令**：尝试将消息 ID 作为键名写入 Redis。        
    - **返回值**：        
        - 如果 `SETNX` 返回 **1**，说明这个 ID 之前不存在，写入成功，当前是第一次消费，可以继续处理业务逻辑。            
        - 如果 `SETNX` 返回 **0**，说明这个 ID 已经存在，表明这是重复消费，当前线程应该直接放弃处理，并返回成功。           
<mark class="hltr-blue">3. 业务处理与超时控制</mark>
- **正常处理**：如果通过了唯一性校验，消费者执行正常的业务逻辑，比如更新数据库、调用其他服务等。    
- **设置过期时间**：为了防止 Redis 键无限增长，需要为这个消息 ID 设置一个合理的**过期时间**（TTL）。这个 TTL 应该比最长的业务处理时间更长，以确保在业务处理期间，不会被误删。
