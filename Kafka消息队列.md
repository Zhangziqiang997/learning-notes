
消息队列（Message Queue）是一种**异步通信机制**，用于在**分布式系统**中解耦发送方和接收方之间的通信。它通过在消息生产者和消费者之间引入一个**中间缓冲区**（常见为 broker），将消息存储在 broker 中，然后由消费者从 broker 中读取和处理消息。

**常见用途**：

- **解耦**：生产者和消费者无需同时在线，生产者可以发送消息后立即返回，而消费者在合适的时机处理消息。
- **削峰填谷**：在高并发场景下，消息队列可以暂存大量请求，平滑高峰流量，避免系统过载。
- **异步处理**：可以将不需要立即处理的任务放入消息队列中异步执行，减少用户请求的响应时间。

- 为什么需要？

从本质上来说是因为互联网的快速发展，**业务不断扩张**，促使技术架构需要不断的演进。

# 消息队列模型

1. 队列模型/点对点模型 <mark class="hltr-green">一对一</mark>

- 在队列模型中，消息从生产者发送到队列，并且每条消息只能被一个消费者消费一次。消费之后，消息在队列中被删除。
- 适用于任务处理类场景，如一个任务只需要一个处理者执行
- 如果有多个消费者同时监听一个队列，它们会**竞争性地消费**队列中的消息，共同分担处理压力，实现负载均衡。
	- 通过多队列全量存储相同的消息，即数据的冗余可以实现一条消息被多个消费者消费，RabbitMQ 就是采用队列模型，通过 Exchange 模块来将消息发送至多个队列，解决一条消息需要被多个消费者消费问题。

2. 发布/订阅模型 <mark class="hltr-green">一对多</mark>

- 为了解决一条消息能被多个消费者消费的问题
- 引入 Topic主题，生产者发布消息到一个“主题”上，消费者通过订阅这个主题来接收消息

# 核心术语

- **生产者（Producer）**：负责向消息队列中发送消息的应用程序或服务。生产者将消息发送到指定的队列或主题（Topic），供消费者消费。

- **消费者（Consumer）**：从消息队列中读取和处理消息的应用程序或服务。消费者根据业务逻辑处理收到的消息，并可以向消息队列发送确认。

- **队列（Queue）**：存储消息的容器，消息按照先进先出（FIFO）的顺序在队列中存储。队列中的每条消息通常只能被一个消费者消费一次。

- **主题（Topic）**：用于在发布/订阅模型中，消息生产者将消息发布到一个主题，多个订阅该主题的消费者可以接收到相同的消息。

- **死信队列（Dead-Letter Queue, DLQ）：** 当消息因为消费失败、多次重试后未成功处理、消息过期或队列达到最大长度等原因被丢弃时，消息可以被转移到死信队列。开发者可以分析死信队列中的消息，找出失败原因并进行修复。

<mark class="hltr-yellow">高性能</mark>
- **分区（Partition）：** 一个 **Partition** 是一个 **Topic** 的物理划分。每个 Partition 都是一个有序的、不可变的消息序列。新消息总是以追加（append）的方式写入到 Partition 的末尾。
	- **并行处理：** 消息队列之所以能实现高吞吐量，很大程度上就是因为 Partition。生产者可以向多个 Partition 并行地发送消息，消费者也可以从多个 Partition 并行地拉取消息。这使得系统可以同时处理大量的读写请求。
	- **负载均衡**
	- **保证顺序：** 虽然一个 Topic 内的全局顺序无法保证（因为消息可以写入不同的 Partition），但一个 **Partition 内的消息是有序的**。这是许多业务场景（比如订单创建、支付、发货等）需要依赖的重要特性。
- **Offset：** Offset 是一个递增的、分区级别的唯一 ID。它标记了消费者在分区中已经消费到哪条消息。
	- 消费者每消费一条消息，就会更新它的 Offset。这使得消息队列能够跟踪每个消费者的消费进度，即使消费者宕机重启，也能从上次停止的地方继续消费，保证不丢失或重复消费消息。

# 如何保证消息不丢失？

这需要生产消息、存储消息和消费消息三个阶段共同努力才能保证消息不丢失。

![[Pasted image 20250828125456.png]]

可以看到消息的生命周期其实就经历了三个阶段，分别是生产消息、存储消息和消费消息。

## 生产消息

生产者发送消息ack至 Broker ，需要处理 Broker 的响应，不论是同步还是异步发送消息，同步和异步回调都需要做好 try-catch ，妥善的处理响应，如果 Broker 返回写入失败等错误消息，需要重试发送。当多次发送失败需要作报警，日志记录等。

这样就能保证在生产消息阶段消息不会丢失。

## 存储消息

消息持久化

需要在**消息刷盘**之后再给生产者响应，假设消息写入缓存中就返回响应，那么机器突然断电这消息就没了，而生产者以为已经发送成功了。

如果 Broker 是集群部署，有多副本机制，即消息不仅仅要写入当前 Broker，还需要写入副本机中。可以配置成至少写入两台机子后再给生产者响应，这样基本上就能保证存储的可靠了。

## 消费消息

这里经常会有同学犯错，有些同学当消费者拿到消息之后直接存入内存队列中就直接返回给 Broker 消费成功，这是不对的。

你需要考虑拿到消息放在内存之后消费者就宕机了怎么办。所以我们应该在**消费者真正执行完业务逻辑之后，再发送给 Broker 消费成功**，这才是真正的消费了。

所以只要我们在消息业务逻辑处理完成之后再给 Broker 响应，那么消费阶段消息就不会丢失。

- **拉取模式（Pull）：** 消费者主动从队列中拉取消息。即使消费者宕机，也不会有消息被“推”给它而丢失。
	优点：消费者可以根据自身负载决定拉取频率，避免过载；更适合批量处理。
	缺点：可能会导致消息延迟，实时性不如推模式，尤其是拉取频率较低时。
    
- **推模式（Push）：** 消息队列主动将消息推送给消费者。如果消费者来不及处理或宕机，消息可能会丢失。因此，推模式通常需要搭配消费者端的回执确认机制。
	优点：实时性好，消息可立即送达消费者。
	缺点：难以控制消费速度，容易导致消费者过载，尤其是在高并发时。

# 如何处理重复消息？

保证**幂等性**

需要改造业务处理逻辑，使得在重复消息的情况下也不会影响最终的结果。

主要是**利用唯一标识（ID）去重**。

在消息中引入全局唯一 ID，例如 UUID、订单号等，利用 redis 等缓存，或者数据库来存储消息 ID，然后消费者在处理消息时可以检查该消息 ID 是否存在代表此消息是否已经处理过。

**去重缓存的使用**：
- 使用 Redis 等缓存系统存储已经处理的消息 ID，以减少数据库的查询压力。
	- 依然使用消息的全局唯一 ID。消费者在处理消息前，先尝试将这个 ID 存入 Redis。可以使用 `setnx`（set if not exist）命令，如果返回成功（说明 ID 不存在），则继续处理业务；如果返回失败，则说明 ID 已存在，消息已被处理过。同时，为了防止数据无限增长，需要给这个 key 设置一个合理的过期时间。

- 可以为 Redis 中的去重记录设置过期时间，例如 7 天，以便自动清理历史消息，减小存储压力。

**去重表的设计**：
- 在数据库中创建一张去重表，用来存储已处理消息的 ID 及处理时间。在消费每条消息前，先查询该表。
- 对于高并发场景，可以结合数据库的唯一索引来避免多次插入同一个消息 ID（`insert into update on duplicate key...`），确保去重表中的记录唯一。
# 如何处理消息堆积？

1. **增加消费者线程数量**：提高并发消费能力。
2. **增加消费实例**：在分布式系统中，可以水平扩展多个消费实例，从而提高消费速率。
	Topic 对应的分区/队列数需要大于等于消费实例数，不然新增加的消费者是没东西消费的。因为一个Topic中，一个分区/队列只会分配给一个消费实例

3. **优化消费者逻辑**：检查消费者的代码，减少单个消息的处理时间。例如，减少 I/O 操作、使用批量处理等。

# 如何保证消息顺序性？
Kafka 通过分区（Partition）机制和消息键（Message Key）来保证消息的顺序性。在 Kafka 中，每个 Topic 可以分为多个分区，每个分区内的消息都是有序的。
因此，Kafka 提供了有限度的顺序性保证，具体来说：
1）在同一个分区内，消息是有序的。
2）靠消息键将相关消息分配到同一分区，可以保证这些消息在同一分区内依然有序。

在某些场景下，消息的顺序性是十分关键的：
1）金融交易系统：交易指令必须按正确的顺序执行，例如银行的转账操作。 
2）日志聚合：日志事件需要按发生的时间顺序进行处理，以便准确地重现事件顺序。 
3）库存管理系统：商品的出入库操作必须按照操作顺序执行，否则会造成库存记录的混乱。 
4）流媒体服务：视频或者音频流的帧数据需要按照播放顺序发送，否则会影响用户的观看体验。

<mark class="hltr-yellow"><mark class="hltr-grey">如果要实现端到端的顺序消费，需要从生产者、Broker（Kafka自身）、消费者三个环节来共同保证</mark></mark>。
1. 生产者端：保证消息进入正确的分区
这是保证顺序的**前提**。生产者在发送消息时，可以决定消息被发送到哪个分区。Kafka默认的分区策略（Partitioner）是：
- **如果指定了Key**：Kafka会对Key进行哈希计算，然后根据哈希值映射到一个具体的分区 (`hash(key) % numPartitions`)。这个过程是确定性的，意味着**只要Key相同，消息就一定会进入同一个分区**。
    - **业务实践**：这就是我们实现业务顺序的关键。例如，我们有一个订单的处理流程，包含“创建订单”、“订单付款”、“订单发货”三个事件。为了保证这三个事件被顺序处理，我们必须在发送这三条消息时，使用**同一个订单ID作为Key**。这样，这三条消息就会按发送顺序，被依次写入到同一个分区中。        
- **如果未指定Key**：生产者会以轮询（Round-Robin）的方式，将消息均衡地发送到不同的分区中，以实现负载均衡。**在这种情况下，顺序将完全无法保证。**   
> **注意**：为了防止因网络重试等原因导致的乱序，生产者端还需要配置 `max.in.flight.requests.per.connection = 1`，确保前一条消息发送成功后，再发送下一条。

2. Broker端：物理上保证分区内顺序
这是Kafka**内在的、最可靠的保证**。
- 一个分区在物理上就是一个**仅追加的日志文件 (Append-only Log)**。    
- 当消息被写入分区时，它们会被**顺序地追加**到日志文件的末尾。    
- 每条消息都会被赋予一个唯一的、单调递增的`offset`。Broker就是通过`offset`来维护分区内的消息顺序的。这个顺序是物理有序，不可改变。   
3. 消费者端：保证按顺序拉取和处理
- **消费规则**：在一个消费者组（Consumer Group）内，**一个分区最多只能被一个消费者实例所消费**。
- **拉取顺序**：消费者从分区拉取消息时，也是严格按照`offset`的顺序来拉取的。它会先拉取`offset=100`的消息，处理并提交消费位点后，才会去拉取`offset=101`的消息。

<mark class="hltr-orange">
如果顺序消费的消息，有一条失败了，怎么办？</mark>
首先，我们要明确问题的本质： 因为Kafka的顺序性是靠`offset`来保证的，一旦某条消息处理失败，我们就**不能提交它的`offset`**。这会导致消费者在下一次拉取时，**又一次拿到这条失败的消息**。如果这个问题不解决，消费者就会陷入无限重试的循环，导致整个分区被**“卡住”(block)**，后面的所有消息都无法被消费。

对于这个问题，没有一劳永逸的完美方案，而是需要根据业务场景和对失败的容忍度，来选择不同的处理策略。主要有以下三种方案：
### 总结

| 方案            | 顺序性保证     | 可用性 | 数据一致性     | 复杂度 | 推荐度   |
| ------------- | --------- | --- | --------- | --- | ----- |
| **无限重试**      | 极强        | 极差  | 强         | 低   | ★☆☆☆☆ |
| **DLQ死信队列方案** | 强 (对成功消息) | 强   | 中 (需后续修复) | 中   | ★★★★★ |
| **跳过消息**      | 弱         | 强   | 弱 (可能丢数据) | 低   | ★★☆☆☆ |
### 方案一：原地无限重试 (阻塞方案)

这是最简单粗暴的方案。当消费者处理某条消息失败时，捕获异常，打印日志，然后**不提交offset**。消费者会默认配置不断地重试。
- **流程**：    
    1. 消费消息 `offset=100`，处理成功。        
    2. 消费消息 `offset=101`，处理失败，抛出异常。        
    3. 程序捕获异常，**不进行offset提交**。        
    4. 消费者下次拉取，再次拉到 `offset=101` 的消息，再次尝试处理。        
- **优点**：    
    - 能**最大限度地保证消息的顺序性**。因为不处理成功绝不前进。        
- **缺点**：    
    - **极其危险**。如果失败是由一个无法自动恢复的原因（比如代码bug、消息格式错误）造成的，这个分区就会被**永久阻塞**，导致整个服务中断。所有和该分区相关的业务都会停止。        
- **适用场景**：    
    - 对顺序性的要求极高，宁可整个服务停掉也不能出现一条乱序，比如金融领域的交易撮合。        
    - 并且，你确信失败都只是临时的（比如网络瞬时抖动），很快就能恢复。        
### 方案二：有限次数重试 + 死信队列 (DLQ)
这是业界**最常用、最推荐**的健壮方案，它在顺序性和可用性之间做了一个很好的平衡。
- **流程**：    
    1. 消费者处理消息失败，捕获异常。        
    2. 程序进行**有限次数的同步重试**（比如3次），并设置一定的退避时间（比如每次间隔1秒）。这可以处理掉大部分临时性故障。        
    3. 如果重试3次后依然失败，程序将不再尝试。        
    4. 程序会将这条“有毒”的消息，连同失败原因、时间等元数据，发送到另一个专门的Kafka Topic，这个Topic就叫做**“死信队列” (Dead Letter Queue, DLQ)**。        
    5. **在将消息成功发送到DLQ之后，程序会手动提交当前消息的offset**。        
    6. 消费者继续前进，处理下一条消息 (`offset=102`)。        
- **优点**：    
    - **高可用**：主消费流程不会因为个别“坏消息”而被阻塞。        
    - **不丢数据**：失败的消息被安全地转存到了DLQ中，可以供后续人工排查或自动修复。        
    - **保证了大部分消息的顺序**：除了那条失败的消息，其他成功的消息依然是按顺序被处理的。        
- **缺点**：    
    - 对于产生失败消息的那个业务实体（比如那个特定的`orderId`），它的**事件顺序被打破了**。"订单付款"事件进入了死信队列，而"订单发货"事件却被正常消费了，这可能导致业务数据不一致，需要下游有补偿或对账机制。        
- **适用场景**：    
    - 绝大部分要求顺序消费的业务场景，如用户行为日志、订单状态流转等。只要能接受个别业务实体出现暂时的数据不一致。        

### 方案三：跳过当前消息
这是一个比较极端的方案，只在特定情况下使用。
- **流程**：    
    1. 处理消息失败，捕获异常。        
    2. 记录下失败日志，可能将失败消息存入一个本地文件或数据库。        
    3. **直接提交offset**，跳过这条消息，继续处理下一条。        
- **优点**：    
    - 实现简单，也能保证消费不被阻塞。        
- **缺点**：    
    - **可能会丢失数据**。如果本地日志记录不完善或出现问题，这条失败的消息就永远消失了。        
    - 同样会打破失败业务的事件顺序。        
- **适用场景**：    
    - 对数据丢失不敏感的业务，比如一些非关键的日志记录。只要保证系统可用性，丢几条消息也无所谓。



# Kafka 为什么性能高

## 顺序写

Kafka 将消息以追加（Append-Only）的方式写入磁盘。硬盘的顺序读写速度远超随机读写，这大大提升了消息的写入效率。

## 批量接收发送

生产者不是一条一条地发送消息，而是将多条消息打包成一个批次（Batch）发送。这样做可以减少网络请求的次数，降低了网络延迟和 I/O 压力。

生产者在发送消息前可以对消息进行压缩，在网络传输和磁盘存储上都节省了资源。消费者拉取到消息后，再进行解压。

## 零拷贝

当一个应用程序需要从磁盘读取数据并通过网络发送出去时，通常会经历 **4 次数据拷贝**和 **4 次上下文切换**：

1. **第一次拷贝：** 数据从磁盘复制到操作系统内核的读缓冲区。    
2. **第一次上下文切换：** 从内核态切换到用户态。    
3. **第二次拷贝：** 数据从内核读缓冲区复制到用户态的应用程序缓冲区。    
4. **第二次上下文切换：** 从用户态切换回内核态。    
5. **第三次拷贝：** 数据从用户态缓冲区复制到内核的网络缓冲区。    
6. **第三次上下文切换：** 从内核态切换到用户态。
7. **第四次拷贝：** 数据从内核网络缓冲区复制到网卡，通过网络发送出去。
