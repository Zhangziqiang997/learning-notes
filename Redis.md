## 哈希表
核心由两个结构体组成：`dict` 和 `dictht`。
###  `dictht` ：哈希表实体
`dictht`  (dictionary hash table) 是一个真正的哈希表结构。Redis 的 `dict` 结构中包含了**两个** `dictht` 实例，这是为了实现渐进式 Rehash。
``` c
// dict.h
typedef struct dictht {
	// 哈希表数组。数组的每个元素都是一个 dictEntry* 指针（指向一个链表）
	dictEntry **table;
	// 哈希表大小
	unsigned long size;
	// 哈希表大小掩码，用于计算索引值，总是等于 size - 1
	unsigned long sizemask;
	// 哈希表中已有节点（键值对）的数量
	unsigned long used;
} dictht;

```
- 使用“**链地址法**”（或叫拉链法）来解决哈希冲突
- `used` 记录当前哈希表中存储的节点的总数，判断是否需要扩容

### `dictEntry`：哈希表节点

```C
// dict.h
typedef struct dictEntry {
    // 键
    void *key;
    // 值
    union {
        void *val;
        uint64_t u64;
        int64_t s64;
        double d;
    } v;
    // 指向下一个哈希表节点，形成链表
    struct dictEntry *next;
} dictEntry;
```
- `union` 存储值
	- `void *val` 一个指针，指向一个 Redis 对象（如字符串对象、列表对象、集合对象等）。
	- `u64` `s64` 无/有符号64 位整数。
	- 当值是整数或浮点数时，可以直接存储在 `dictEntry` 中
###  `dict` ：字典（哈希表的封装）
包含了两个 `dictht` 实例以及 Rehash 相关的状态。

```C
// dict.h
typedef struct dict {
    // 类型特定函数
    dictType *type;
    // 私有数据
    void *privdata;
    // 两个哈希表
    dictht ht[2];
    // rehash 索引
    // 当 rehash 不在进行时，值为 -1
    int rehashidx; 
    // 当前正在运行的迭代器数量
    unsigned long iterators; 
} dict;
```
- `dictType *type;`
	- 指向 `dictType` 结构的指针。`dictType` 包含了一系列函数指针，用于操作特定类型的键和值，例如：
	    - `hashFunction`: 哈希函数，用于计算键的哈希值。
	    - `keyCompare`: 键比较函数，用于判断两个键是否相等。
	    - `keyDup`: 键复制函数。
	    - `valDup`: 值复制函数。
	    - `keyDestructor`: 键析构函数。
	    - `valDestructor`: 值析构函数。
	- 这种设计，有高度通用性，可以存储任何类型的键值对，只要提供相应的操作函数。
- `dictht ht[2];`
    - **核心中的核心！** 一个包含两个 `dictht` 实例的数组。
    - `ht[0]` 是**当前使用的哈希表**。
    - `ht[1]` 是**Rehash 时使用的临时哈希表**。在正常情况下，`ht[1]` 的 `table` 指针为 `NULL`。
- `int rehashidx;`
    - **渐进式 Rehash 的关键标志！**
    - 如果其值为 `-1`，表示当前**没有在进行 Rehash**。
    - 如果其值 `>= 0`，表示**正在进行 Rehash**。它的值记录了当前 Rehash 进度，即 `ht[0]` 中已经迁移到 `ht[1]` 的数组索引位置。例如，`rehashidx = 10` 表示 `ht[0]` 中索引 0 到 9 的桶（bucket）已经全部迁移完毕，下一次将从索引 10 开始迁移。
- `unsigned long iterators;`
    - 记录当前有多少个迭代器正在遍历这个字典。这个值主要用于在 Rehash 过程中保证迭代的安全性。
### 渐进式Rehash(哈希表扩容/缩容)
- **扩容：**
    - 服务器目前没有在执行 `BGSAVE` 或 `BGREWRITEAOF` 命令，并且 `ht[0].used / ht[0].size >= 1`（即负载因子 >= 1）。
    - 服务器正在执行 `BGSAVE` 或 `BGREWRITEAOF` 命令，并且 `ht[0].used / ht[0].size >= 5`（即负载因子 >= 5）。
    - **为什么有区别？** 因为 `BGSAVE`/`BGREWRITEAOF` 会 fork 子进程，操作系统会使用写时复制。为了避免内存过度膨胀，Redis 在执行这些命令时会提高扩容的阈值，尽量减少 Rehash 带来的内存写入。
    - 扩容后 `ht[1]` 的大小是 `ht[0]` 的第一个大于等于 `used * 2` 的 2 的幂。
- **缩容：**
    - 当负载因子小于 `0.1` 时 (`ht[0].used / ht[0].size < 0.1`)。
    - 缩容后 `ht[1]` 的大小是 `ht[0]` 的第一个大于等于 `used` 的 2 的幂。

## 集合

- 无序性
- 唯一性
- 元素类型：字符串
- 高效性：添加、删除、查找元素的平均时间复杂度都是 **O(1)**
### 底层实现
采用了**两种编码方式**，并根据实际情况进行自动转换。
1. **`intset` (整数集合)**
	专门为存储整数而设计的一种紧凑的、有序的内存数据结构。
	它本质上是一个**连续的内存块**，没有指针开销，非常节省内存。

2. **`hashtable` (哈希表)**
	当集合**不满足 `intset` 的条件**时（即：包含了非整数字符串，或者整数数量超过了阈值），Redis 会将其底层实现转换为 `hashtable`。
### **使用场景**
- **标签系统**：存储用户的兴趣标签，避免重复。
- **唯一用户集合**：记录访问过某个页面的唯一用户，方便进行分析。

## Ziplist 和 Quicklist 

### **核心定义与本质区别**

#### **1. Ziplist（压缩列表）**

- **是什么**  
    一块**连续内存空间**组成的紧凑型数据结构，通过特殊编码存储所有元素（包括字段名、字段值、长度、偏移量等）。
- **设计哲学**  
    **极致内存优化**：牺牲部分操作性能，换取内存占用最小化（无指针开销，数据紧密排列）。
- **内部结构**
```
<zlbytes> <zltail> <zllen> <entry1> <entry2> ... <entryN> <zlend>
```

- `zlbytes`：总字节数
- `zltail`：尾元素偏移量（支持快速反向遍历）
- `zllen`：元素数量（超过 `2^16-2` 时需遍历统计）
- `entry`：变长编码元素（前缀编码存储长度和类型）

#### **2. Quicklist（快速列表）**

- **是什么**  
    **双向链表 + Ziplist 的混合结构**：链表节点每个指向一个独立的 Ziplist。
- **设计哲学**  
    **平衡内存与性能**：用 Ziplist 减少节点内存碎片，用链表解决 Ziplist 大数据量下的性能问题。
- **内部结构**
```
quicklistNode {
    prev *quicklistNode    // 前驱节点
    next *quicklistNode    // 后继节点
    zl *ziplist            // 指向一个Ziplist
    count uint32           // 当前Ziplist元素数
    ... 
}
```

### **二、关键差异对比（表格分析）**

|**维度**|**Ziplist**|**Quicklist**|
|---|---|---|
|**内存布局**|连续内存块（无碎片）|链表节点 + 分散的Ziplist（可控碎片）|
|**操作复杂度**|**中间插入/删除 O(N)**（需移动数据）|**中间操作 O(N/M)**（M=单个Ziplist大小）|
|**内存效率**|极高（无指针开销，数据紧凑）|较高（链表指针开销 + Ziplist连续内存）|
|**扩容成本**|高（可能触发realloc+memmove）|低（仅影响单个Ziplist）|
|**最大容量**|受限于单块内存大小（通常 < 32KB）|理论无上限（链表动态扩展）|
|**适用场景**|小数据量（Hash/Set/List的默认编码）|大数据量List（Redis 3.2+默认编码）|
|**配置参数**|`hash-max-ziplist-entries`等|`list-max-ziplist-size`（单个Ziplist大小）<br>`list-compress-depth`（压缩层数）|

---

### **三、设计取舍与工程实践**

#### **1. 为什么需要 Quicklist？**

- **Ziplist 的缺陷**：
    
    - **性能瓶颈**：当元素增多时，中间插入/删除需移动大量数据（如 `linsert` 操作耗时激增）。
    - **内存风险**：大 Ziplist 可能导致 `malloc` 分配连续大内存失败（尤其在32位系统）。
    
- **Quicklist 的解决方案**：
    - **分片存储**：将大 List 拆分为多个小 Ziplist，操作仅影响局部节点。
    - **压缩优化**：对链表中间节点（非头尾）的 Ziplist 进行 LZF 压缩（`list-compress-depth` 控制），进一步节省内存。

### **四、性能与内存实测场景**

<mark class="hltr-red">当列表长度超过 Ziplist 的阈值时，Redis 会使用 Quicklist 存储列表。</mark>
#### **场景1：小数据量 List（< 1000 元素）**

- **Ziplist**：内存占用更低（无链表指针），操作性能接近。
- **Quicklist**：因链表指针开销，内存略高，但差异可忽略。  
    **结论**：两者均可，Ziplist 略优（Redis 自动选择）。

#### **场景2：大数据量 List（10万+ 元素）**

- **Ziplist**：
    
    - 中间插入耗时：`O(N)` → 需移动整个列表数据（如 10万元素耗时秒级）。
    - 内存碎片：连续大块内存分配失败风险高。
    
- **Quicklist**（`list-max-ziplist-size=8KB`）：
    - 中间插入耗时：`O(N/M)` → 仅操作单个 Ziplist（如 8KB 约 500 元素，耗时毫秒级）。
    - 内存碎片：分散的小块内存更易分配，压缩后节省 50%+ 内存。  
        **结论**：Quicklist 完胜。


# Redis持久化


是将 Redis 服务器内存中的**数据状态**，以一种可恢复的形式，**保存到磁盘**上的机制。

## RDB

通过**快照**的方式，将某个时间点上 Redis 的所有数据生成一个**二进制文件**保存到磁盘。

### 工作原理
1. 触发
	1. 手动触发：通过客户端向 Redis 服务器发送 `SAVE` 或 `BGSAVE` 命令。
	2. 在 `redis.conf` 配置文件中，设置满足特定条件时自动执行 `BGSAVE`。
2. 执行过程 (`BGSAVE` - 推荐方式)
	- Redis 主进程会 `fork`（创建）一个**子进程**。
	- **子进程**负责将当前内存中的所有数据写入到一个新的 RDB 文件（例如 `dump.rdb`）中。
	- **主进程**在此期间继续**处理客户端的请求**，不受影响。
	- 当子进程完成写入后，会用新的 RDB 文件替换掉旧的 RDB 文件。
	- 整个过程是一个**异步**操作。
3. 执行过程 (`SAVE` - 不推荐):
    - Redis 主进程会**阻塞**所有客户端请求。
    - 由主进程自己负责将数据写入 RDB 文件。
    - 在写入完成之前，Redis 无法提供任何服务。
    - 这是一个**同步**操作，在生产环境中应**严格避免**使用，因为它会导致服务长时间不可用。
### 优点
1. **文件紧凑，加载速度快：** RDB 是二进制压缩文件，体积小。Redis 启动时加载 RDB 文件恢复数据的速度远快于 AOF。
2. **对性能影响小：** `BGSAVE` 操作由子进程完成，不会阻塞主进程，对线上服务影响极小。
3. **适合备份：** 单个 RDB 文件包含了某个时间点的完整数据集，非常适合用于灾难恢复、全量备份和数据迁移。你可以轻松地将一个 RDB 文件拷贝到其他服务器上恢复数据。
### 缺点
1. **数据丢失风险高：** 这是 RDB 最大的缺点。由于是定时快照，如果 Redis 在两次快照之间意外宕机，那么这段时间内的所有写操作都会**永久丢失**。例如，你配置了 `save 60 10000`，最坏情况下，你可能会丢失近 60 秒的数据。
2. **`fork` 操作的潜在问题：** 当 Redis 数据集非常大时（例如几十 GB），`fork` 子进程这个操作本身可能会比较耗时，因为操作系统需要复制主进程的页表。在内存紧张或数据量巨大的情况下，`fork` 可能会导致短暂的秒级卡顿。虽然主进程不阻塞，但 `fork` 的瞬间服务是不可用的。
3. **版本兼容性：** 不同版本的 Redis 可能使用不同版本的 RDB 格式。虽然 Redis 向后兼容，但用新版本生成的 RDB 文件可能无法被旧版本的 Redis 加载。

## AOF

通过记录所有写操作命令的方式，提供了比 RDB 更高的数据安全性。

### 工作原理
AOF 的核心是 **“只追加”** 。它像一个操作日志，记录了所有改变数据状态的命令。
1. **命令写入：** 当 Redis 执行一个写命令（如 `SET mykey value`）时，该命令会被**追加**到内存中的一个 AOF 缓冲区中。
2. **缓冲区同步到文件：** 根据配置的 `appendfsync` 策略，操作系统会将 AOF 缓冲区中的内容写入到磁盘上的 AOF 文件（例如 `appendonly.aof`）中。
3. **数据恢复：** 当 Redis 重启时，它会重新读取 AOF 文件，并**从头到尾依次执行**文件中记录的所有命令，从而将内存数据恢复到崩溃前的状态。
### 重写
只包含能够恢复当前数据集所需的最小命令集，异步操作

1. Redis 会 `fork` 一个子进程。
2. 子进程**不读取旧的 AOF 文件**，而是直接**扫描当前内存中的数据**。
3. 子进程将内存中的每个键值对，转换成一条对应的写命令（例如，对于一个值为 `"hello"` 的字符串键 `mykey`，会生成 `SET mykey hello`），并将这些命令写入到一个**临时的 AOF 文件**中。
4. 在子进程重写期间，主进程依然在处理新的写命令。这些新的写命令会同时被**追加到旧的 AOF 文件**和一个**重写缓冲区**中。
5. 子进程完成重写后，主进程会将重写缓冲区中的所有新命令追加到**临时 AOF 文件**的末尾。
6. 最后，主进程用这个**临时 AOF 文件原子性地**替换掉旧的 AOF 文件。

### 写回策略
- `always`：
    - **同步：** 每个写命令执行后，都**立即**调用 `fsync()` 系统调用，将数据从操作系统缓冲区**强制写入**物理磁盘。
    - **优点：** **数据安全性最高**。理论上最多只会丢失一个命令的数据（即正在写入但未 `fsync` 的那个）。
    - **缺点：** **性能最差**。因为每次写操作都要等待磁盘 I/O 完成，会严重拖慢 Redis 的速度。
- `everysec` (默认值，**推荐**)：
    - **同步：** 每个写命令先写入 AOF 缓冲区，然后由一个**后台线程**每秒执行一次 `fsync()`。
    - **优点：** **性能和数据安全性的最佳平衡**。即使宕机，最多只会丢失**1秒内**的写操作数据。
    - **缺点：** 仍然有 1 秒的数据丢失风险。
- `no`：
    - **同步：** 将写入操作完全交由操作系统控制。操作系统会在自己的缓冲区满时，或者根据其他策略，才将数据写入磁盘。
    - **优点：** **性能最高**，因为完全不进行 `fsync`。
    - **缺点：** **数据安全性最低**。不可控因素最多，宕机时可能丢失数秒甚至更长时间的数据。

| 特性        | RDB                   | AOF                    |
| --------- | --------------------- | ---------------------- |
| **数据安全性** | **低**，有分钟级数据丢失风险      | **高**，最多秒级数据丢失         |
| **文件体积**  | **小**，二进制压缩格式         | **大**，文本格式，记录所有写操作     |
| **恢复速度**  | **快**，直接加载二进制文件       | **慢**，需要重放所有命令         |
| **对性能影响** | **小**，`BGSAVE` 由子进程完成 | **中**，`fsync` 线程消耗 I/O |
| **数据格式**  | 二进制，不可读               | 文本，可读（Redis 协议）        |
| **使用场景**  | 备份、灾难恢复、对数据丢失不敏感的场景   | 要求高数据安全性的主数据库场景        |

## RDB-AOF 混合持久化
### 工作原理
混合持久化的核心思想是：**将 RDB 快照作为 AOF 文件的一部分**

AOF重写时：
- 当 AOF 重写被触发时，子进程不再像传统 AOF 那样生成纯文本的命令文件。
- 子进程会**先将当前内存数据以 RDB 格式写入**到临时的 AOF 文件中。
- RDB 数据写入完成后，子进程会继续将重写缓冲区中积累的新命令，以**AOF 格式追加**到这个临时文件的末尾。
- 最终，这个临时文件的前半部分是紧凑的 RDB 数据，后半部分是增量的 AOF 命令。

数据恢复：
- Redis 重启时，加载这个混合格式的 AOF 文件。
- 它会先**快速加载文件开头的 RDB 部分**，这大大加快了启动速度。
- RDB 部分加载完成后，再**重放文件末尾的 AOF 命令**，将数据恢复到最新的状态。

## 核心实现方案

在 Redis 中实现分布式锁的常见方法是通过`set ex nx` 命令 + lua 脚本组合使用。
确保多个客户端不会获得同一个资源锁的同时，也保证了安全解锁和意外情况下锁的自动释放。

1）加锁：`SET lock_key uniqueValue EX expire_time NX` 2）解锁：使用 lua 脚本，先通过 get 获取 key 的 value 判断锁是否是自己加的，如果是则 del。<mark class="hltr-red">【先判断key，再删除，要保证原子性->lua】</mark>

**Q:** 那为什么set命令不用放到lua中？

**A:** 因为setnx是单个命令，redis执行是单线程，天然保证原子性

lua 脚本如下：
```lua
if redis.call("GET",KEYS[1]) == ARGV[1]
then
    return redis.call("DEL",KEYS[1])
else
    return 0
end
```

首先**锁需要有过期机制**。假设某个客户端加了锁之后宕机了，锁没有设置过期机制，会使得其他客户端都无法抢到锁。

`EX expire_time` 就是设置了锁的过期，单位是秒。还一个 `PX`， 也是过期时间，单位是毫秒。

然后，在 2.6.12 版本之前只有 `SETNX` 即 SET if Not eXists，它表示如果 key 已存在，则什么都不会做，返回 0，如果不存在则会设置它的值，返回 1。

那个时候 `SETNX` 和过期时间的设置就无法保证原子性，如果客户端在发送完 `SETNX` 之后就宕机了，还没来得及设置过期时间，一样会导致锁不会被释放。

因此 2.6.12 版本之后，优化了 SET 命令，使得可以执行 `set ex px`。

最后，再解释下什么是 `uniqueValue`，翻译过来就是一个唯一的值。

之所以要设置唯一值是为了**防止被别的客户端给释放了**。
	我们来看下这个场景：
		1.  客户端 1 加锁成功，然后执行业务逻辑，但执行的时间超过了锁的过期时间
		2.  此时锁已经过期被释放了，客户端 2 加锁成功
		3.  客户端 2 执行业务逻辑
		4.  客户端 1 执行完了，执行释放锁的逻辑，即删除锁。
客户端 2 的锁被删除了，所以每个客户端加锁（客户端可能是每个线程），需要是设置一个唯一标识，比如一个 uuid，防止锁被别的客户端误释放了。

因为需要先判断锁的值和唯一标识是否一致，一致后再删除释放锁，这里就涉及到两步操作，只有使用了 lua 脚本才能保证原子性，这也是为什么释放锁需要用 lua 脚本的原因。

## 看门狗

<mark class="hltr-green">拿到锁的线程挂了怎么办 -> 给锁加过期时间，自动释放 -> 线程还没执行完，超时，锁自动释放怎么办
-> 看门狗机制 -> 看门狗线程设置为守护线程，其生命周期依赖于业务线程，挂了JVM会自动终止守护线程</mark>

Redisson 的看门狗（watchdog）主要用来避免 Redis 中的锁在超时后业务逻辑还未执行完毕，锁却被自动释放的情况。它通过定期刷新锁的过期时间来实现自动续期。

**主要原理**：

1. **定时刷新**：如果**当前分布式锁未设置过期时间**，Redisson 基于 Netty 时间轮启动一个定时任务，定期向 Redis 发送命令更新锁的过期时间，默认每 10s 发送一次请求，每次续期 30s。
2. **释放锁**：当客户端主动释放锁时，Redisson 会取消看门狗刷新操作。如果客户端宕机了，定时任务自然也就无法执行了，此时等超时时间到了，锁也会自动释放。

### 如果获取锁的客户端挂了怎么办？

续期是通过定时任务执行的，如果当前客户端宕机，那么定时任务就没了

按照默认每 10s 续期 30s 的情况来看，锁不可能无限续期，等 30s 时间一到，集群中的其他客户端就可以获取锁，不会阻碍正常业务的执行。

## 可重入

- Redisson 支持可重入锁，持有锁的线程可以多次获取同一把锁而不会被阻塞。具体是利用 Redis 中的哈希结构，哈希中的 key 为线程 ID，Field为`UUID + threadId` ，value为锁的**重入次数**。

- 如果重入则 `value +1`，如果释放则 `value -1`，减到 0 说明锁被释放了，则 `del` 锁。

	- Redisson 底层使用了 **`HSETNX`** 而不是 `SETNX`。
	- 为什么内层字段要拼接 UUID + threadId？
		- 如果是分布式（集群）环境下，线程id可能会重复，所以要拼接；
		- 同时避免锁id重复导致的误删

## RedLock 红锁

Redlock 是 Redis 官方推荐的一种实现分布式锁的算法，适用于集群环境下。

发生了主从切换，**但从节点上不一定已经同步了主节点的锁信息**。所以新的主节点上可能没有锁的信息。此时另一个业务去加锁，一看锁还没被占，于是抢到了锁开始执行业务逻辑。

此时就发生了两个竞争者同时进入临界区操作临界资源的情况，可能就会发生数据不一致的问题。

**Redlock 的基本思想**：

- 部署多个 Redis 实例（通常为 5 个）。
- 客户端在大多数实例（如至少 3 个）上请求锁，并在一定时间内获得成功，表示加锁成功。
- 使用 Redlock 可以提供更高的容错性，即使部分 Redis 实例故障，仍然可以获得锁。

**Redlock 的实现流程**：

- 客户端尝试在每个 Redis 实例上加锁，必须在有限时间内（通常为锁的过期时间）完成所有实例的加锁。
- 如果大多数实例（N/2 + 1）加锁成功，则表示加锁成功。
	- 超过半数： 因为某个线程加锁超过半数，就不可能有其他线程超过半数了
- 否则，客户端将释放所有已经加锁的实例，重新尝试。

**Redlock 的缺点包括**：

- **复杂性**：实现 Redlock 需要多个 Redis 实例，增加了系统的复杂性和维护成本。
    
- **时间同步依赖**：Redlock 依赖于多个节点的系统时间的一致性。如果节点之间的时间不同步（例如发生时间回拨），可能导致锁的有效性受到影响。
    
- **不适用于高并发**：在高并发场景下，因需要访问多个实例同时尝试获取锁，可能会导致锁获取的性能下降。
    
- **锁的续期问题**：在长时间的操作中，可能需要手动续期锁，因为涉及多个实例，增加了实现的复杂度和风险。

### 1. 不淘汰（默认）
- 当运行内存超过最大设置内存的时候，不会淘汰数据，而是直接返回报错禁止写入
- 主要用作数据库，而不是缓存。

### 3. 最近最少使用 LRU
<mark class="hltr-red">实际无法找到最完美的LRU，采用近似算法</mark>

### 4. 最不经常使用 LFU
实际无法找到最完美的LFU，采用近似算法

---
### 5. 过期淘汰
这类策略只考虑那些设置了 TTL（Time To Live，生存时间）的键。对于那些没有设置 TTL 的键，它们被认为是“永久的”，不会被淘汰。

- **定期删除**：Redis 每隔一定时间（默认是 100 毫秒）会随机检查一定数量的键，如果发现过期键，则将其删除。这种方式能够在后台持续清理过期数据，防止内存膨胀。
- **惰性删除**：在每次访问键时，Redis 检查该键是否已过期，如果已过期，则将其删除。这种策略保证了在使用过程中只删除不再需要的数据，但在不访问过期键时不会立即清除。

---

### 6. 在所有键中淘汰
这类策略考虑 Redis 中的**所有键**，无论它们是否设置了 TTL。当内存不足时，即使是“永久”键也可能被淘汰。

## 缓存一致

产生**缓存一致性**问题的根本原因在于：**数据有两个副本**，一个在速度快的内存（缓存）中，一个在速度慢的磁盘（数据库）中。当数据发生变更时，如果不能同时、原子性地更新这两个副本，就会出现不一致的状态。

### Cache-Aside (旁路缓存) - 最常用
这是最经典、最广泛的缓存模式。应用程序代码**显式地**负责维护缓存和数据库。

#### 读流程

1. 应用先从**缓存**中读取数据。
2. 如果缓存中**有**（Cache Hit），则直接返回。
3. 如果缓存中**没有**（Cache Miss），则从**数据库**中读取数据。
4. 将从数据库中读取到的数据**写入缓存**。
5. 返回数据。

#### 写流程

1. **先更新数据库**。
2. **再删除缓存**中对应的旧数据。

Q: <mark class="hltr-yellow">为什么是“删除”而不是“更新”？</mark>
A: 删除缓存比更新缓存更好
- **避免无效更新：** 如果写操作非常频繁，但读操作很少，那么每次写都去更新缓存，大部分更新都是徒劳的，因为缓存可能很快就会因为过期而被淘汰，或者因为长时间未被读取而被淘汰。删除操作更轻量。
- **避免复杂计算：** 如果缓存的数据是需要经过复杂计算或从多个表关联查询得到的，那么更新缓存需要重新执行这些复杂逻辑，而删除缓存则不需要，等到下次读请求时再按需计算即可（懒加载）。
- **简化一致性：** 删除操作是幂等的。即使删除失败，最多也只是导致一次 Cache Miss，然后从数据库重新加载，不会导致数据错误。而更新操作如果失败，可能导致缓存停留在某个中间状态。
#### 优点
- **简单直观：** 实现简单，易于理解和维护。
- **解耦：** 缓存和数据库的逻辑完全由应用层控制，耦合度低。
- **容错性好：** 即使缓存服务宕机，应用仍可绕过缓存直接访问数据库（只是性能下降）。
#### 缺点
- **存在短暂的不一致窗口：** 这是最主要的缺点。在“更新数据库”和“删除缓存”之间，存在一个时间差。
- **代码侵入性强：** 业务代码中需要混入缓存逻辑。

#### 深度剖析
由于 Cache-Aside 是最主流的模式，我们重点分析它的一致性问题以及如何优化。
##### 1. “先删缓存，再更新数据库”可行吗？
1. 线程 A：删除缓存。
2. 线程 B：读请求，Cache Miss，去读数据库（读到旧数据）。
3. 线程 B：将旧数据写入缓存。
4. 线程 A：更新数据库（成功）。
5. **结果：** 缓存中是旧数据，数据库是新数据。**不一致！**
这个顺序的问题在于，在“删除缓存”和“更新数据库”的窗口期，任何读请求都可能导致旧数据被重新加载到缓存。这个不一致窗口比“先更新数据库，再删除缓存”的窗口要长且更容易发生。
**结论：** **“先更新数据库，再删除缓存”是更优的选择。** 虽然它依然有理论上的不一致窗口，但在实践中，由于数据库的写操作通常比缓存的删除操作慢，这个窗口非常小。而且，可以通过其他手段进一步缩小这个窗口。

##### 2. 如何解决 Cache-Aside 的最终一致性问题？

目标是让不一致的窗口尽可能小，或者在发生不一致时能快速修正。
1. **延迟双删**
	1. 先**更新数据库**。
	2. 再**删除缓存**。
	3. **休眠一小段时间**（例如，几百毫秒）。
	4. **再次删除缓存**。

	- 第一次删除是为了清除旧数据。
	- 休眠是为了等待那些在“更新数据库”和“第一次删除缓存”之间发生 Cache Miss 的线程，将旧数据加载到缓存。
	- 第二次删除是为了清除这些被重新加载的旧数据。

	- **优点：** 能极大地提高一致性，有效解决并发读写导致的不一致。
	- **缺点：**
		- **增加了写延迟：** 每次写操作都要多等待一次休眠和一次删除。
		- **休眠时间难以确定：** 休眠时间需要大于一次数据库读操作 + 一次缓存写操作的时间。这个时间不好估算，太短可能无效，太长影响性能。

2. **删除缓存重试机制**
- **问题：** “更新数据库”成功了，但“删除缓存”失败了怎么办？这会导致缓存长期处于脏数据状态。
- **解决方案：**
    - **同步重试：** 在代码中捕获删除缓存的异常，然后进行几次重试。简单，但会阻塞写请求。
    - **异步重试（推荐）：**
        1. 将删除失败的缓存 key 放入一个**消息队列**（如 Kafka, RabbitMQ）或 Redis 的 List/Set 中。
        2. 有一个**独立的消费者服务**，订阅这个队列。
        3. 消费者从队列中获取 key，尝试删除缓存。
        4. 如果删除失败，可以记录日志并放入重试队列，进行指数退避重试。
- **优点：** 解耦了写操作和缓存删除操作，提高了系统的健壮性和可用性。即使删除缓存失败，也有后台机制保证最终会被删除。
- **缺点：** 架构变得更复杂，需要引入消息队列等中间件。

3. **订阅数据库的 Binlog**
- **思想：** 先更新数据库再删缓存，那么数据库更新成功了，就会产生一条binlog，监听这个binlog，然后删除缓存，删除失败就重试。 将缓存删除操作与业务代码解耦，由数据变更事件来驱动。
- **流程（以 MySQL 为例）：**
    1. 业务代码只负责**更新数据库**。
    2. 使用一个服务（如阿里巴巴的 **Canal**）**订阅 MySQL 的 Binlog**。
    3. 当数据库发生变更（`UPDATE`, `DELETE`）时，Canal 会接收到变更事件。
    4. Canal 解析事件，获取变更的表名、主键等信息。
    5. Canal 根据预设的规则，向 Redis 发送**删除缓存**的命令。
- **优点：**
    - **完全解耦：** 业务代码非常干净，只关心数据库。
    - **高可靠性：** 数据库的 Binlog 是最终的数据源，只要 Binlog 存在，缓存删除操作就不会丢失。
    - **适用于复杂场景：** 当一个数据变更需要影响多个缓存时，这种方式非常方便。
- **缺点：**
    - **架构最复杂：** 需要引入和维护 Canal 等中间件。
    - **存在延迟：** 从 Binlog 产生到被消费，存在一定的延迟（通常是毫秒级），这个延迟就是不一致窗口。
    - **对数据库有侵入性：** 需要开启数据库的 Binlog。

### Read-Through (读穿透)
在这种模式下，应用只与**缓存服务**交互。当缓存未命中时，由**缓存服务自己**负责从数据库加载数据。

#### 流程
1. 应用向缓存请求数据。
2. 如果缓存命中，直接返回。
3. 如果缓存未命中，**缓存服务**会自动去数据库中加载数据。
4. 缓存服务将数据加载到自身，并返回给应用。

#### 优点
- **简化应用代码：** 应用层不需要关心 Cache Miss 的处理逻辑，代码更简洁。
- **数据源统一：** 缓存层封装了对数据源的访问。

#### 缺点
- **灵活性差：** 通常需要使用支持此特性的特定缓存库或中间件。
- **写操作依然要处理：** Read-Through 只解决了读的问题，写操作通常还需要结合 Write-Through 或 Write-Behind。

### Write-Through (写穿透)
与 Read-Through 类似，应用也只与**缓存服务**交互。当应用写数据时，它只更新缓存，由**缓存服务**负责同步更新数据库。

#### 流程
1. 应用向缓存写入数据。
2. 缓存服务**立即更新**自身的数据。
3. 缓存服务**同步地**将更新写入到数据库。
4. 只有当数据库也更新成功后，缓存服务才向应用返回成功。

#### 优点
- **强一致性：** 由于是同步更新，可以保证缓存和数据库的强一致性（在单线程模型下）。
- **简化应用代码：** 应用层代码非常干净。

#### 缺点
- **性能差：** 每次写操作都需要等待数据库 I/O 完成，延迟高，吞吐量低。这违背了使用缓存来提升写性能的初衷。
- **缓存服务复杂：** 缓存服务需要承担数据同步的责任。

### Write-Behind (写回/异步写)
这是 Write-Through 的异步版本，旨在提高写性能。
#### 流程
1. 应用向缓存写入数据。
2. 缓存服务**立即更新**自身的数据，并**立即向应用返回成功**。
3. 缓存服务**异步地**、批量地将更新写入到数据库。

#### 优点
- **极高的写性能：** 应用的写请求只需要操作内存，延迟极低，吞吐量极高。
- **削峰填谷：** 可以将数据库的随机写操作合并成顺序的批量写，减轻数据库压力。

#### 缺点
- **数据一致性风险高：** 这是最致命的缺点。如果缓存服务在数据写入数据库之前宕机，那么这部分数据就会**永久丢失**。
- **实现复杂：** 需要处理异步队列、失败重试、数据排序等复杂问题，保证最终一致性非常困难。

### 总结
|模式|一致性|性能|复杂度|适用场景|
|---|---|---|---|---|
|**Cache-Aside**|**最终一致性**|**高**|**低**|**绝大多数互联网应用，通用首选**|
|**Read-Through**|最终一致性|中|中|读多写少，希望简化应用代码|
|**Write-Through**|**强一致性**|**低**|中|对一致性要求极高，能接受低性能|
|**Write-Behind**|**最终一致性**|**极高**|**高**|写密集型，能容忍数据丢失风险|

## 缓存击穿 vs. 缓存穿透 vs. 缓存雪崩

这是三个经常被混淆的概念，区分它们非常重要。

|问题类型|**缓存击穿**|**缓存穿透**|**缓存雪崩**|
|---|---|---|---|
|**根本原因**|**热点 Key 过期**|**请求不存在的 Key**|**大量 Key 同时过期**|
|**请求特点**|请求的是**存在**的、**热点**的 Key|请求的是**不存在**的 Key|请求的是**存在**的 Key|
|**影响范围**|**单个**热点 Key|**大量**不同的、不存在的 Key|**大量**不同的、存在的 Key|
|**后果**|数据库瞬间被**单点**高并发查询打垮|数据库被大量无效查询打垮，缓存无意义|数据库被大量查询打垮，缓存大规模失效|
|**比喻**|一把**热门**的钥匙坏了，所有人都在门口等|一群小偷拿着**不存在**的钥匙挨家挨户试|整个小区的**门锁**同时过期，所有人都在门口等|
**一句话总结：**

- **雪崩**是**集体**的、**灾难性**的。
- **击穿**是**单点**的、**瞬时**的。
- **穿透**是**分散**的、**慢性**的。

## 缓存击穿

一个**非常热点**的 `key`，在**缓存过期的一瞬间**，同时有**大量的并发请求**涌过来。由于这个 `key` 在缓存中已经不存在了，这些并发请求会**全部穿透缓存，直接打到后端的数据库**上，瞬间给数据库带来巨大的压力，甚至可能导致数据库宕机。

典型场景：
1. **热点新闻/事件：** 某个突发新闻的详情页 `key`，在缓存过期时，恰好有海量用户涌入查看。
2. **秒杀活动：** 秒杀商品的库存信息 `key`，在活动开始时或缓存过期时，被大量用户同时查询。
3. **热门商品/用户：** 电商平台的爆款商品详情页，或社交网络中的超级大V的个人主页 `key`，在缓存过期瞬间被大量请求。
4. **系统首页/配置信息：** 系统的全局配置或首页数据 `key`，在缓存过期时，被所有用户访问。

### 核心方案
**避免在热点 `key` 过期的瞬间，让大量并发请求直接打到数据库上。** 围绕这个核心，有几种非常有效的解决方案。

#### 互斥锁 / 分布式锁 (Mutex Lock / Distributed Lock)
当缓存未命中时，**只允许一个请求去查询数据库并重建缓存**，其他所有请求都必须**等待**。

**实现流程：**
1. 线程 A 发现缓存中没有数据（Cache Miss）。
2. 线程 A 尝试获取一个**与 `key` 相关的分布式锁**（例如，使用 Redis 的 `SET key value NX PX 10000` 命令）。
3. **获取锁成功：**
    - 线程 A 去数据库查询数据。
    - 线程 A 将查询到的数据写入缓存，并设置一个合理的过期时间。
    - 线程 A **释放分布式锁**。
    - 线程 A 返回数据。
4. **获取锁失败：**
    - 线程 B, C, D... 也发现 Cache Miss，并尝试获取同一个锁，但都失败了。
    - 它们不会立即去查数据库，而是进入一个**短暂的休眠**（例如 `sleep(50)` 毫秒）。
    - 休眠结束后，它们会**重新尝试从缓存中读取数据**。
    - 此时，线程 A 已经重建了缓存，所以它们会 Cache Hit，直接从缓存获取数据并返回。

**优点：**
- **思路简单，效果显著：** 能非常有效地阻止大量请求穿透到数据库。
- **数据一致性高：** 保证了同一时间只有一个请求能去重建缓存，避免了缓存和数据库的短暂不一致。

**缺点：**
- **增加了请求的等待时间：** 没有获取到锁的请求需要等待，增加了用户的响应延迟。
- **实现复杂：** 需要引入分布式锁，并处理好锁的获取、释放、超时、死锁等问题。
- **锁的竞争：** 在极高并发下，锁本身也可能成为性能瓶颈。

#### 缓存永不过期 (Logical Expiration)
不给热点 `key` 设置物理上的过期时间（`EXPIRE`），而是**在数据中增加一个逻辑上的过期时间字段**。当应用读取数据时，判断其是否已逻辑过期。如果已过期，则启动一个后台异步线程去更新缓存，而当前请求直接返回“过期”的旧数据。

**实现流程：**
1. **数据写入缓存时：**
    - 将数据库中的数据（如商品信息）和一个逻辑过期时间（如 `expire_timestamp`）打包成一个对象（或使用 Hash 结构）存入 Redis。
    - **不设置 Redis Key 的 TTL。**
2. **数据读取时：**
    - 应用从 Redis 中读取整个数据对象。
    - 检查 `expire_timestamp` 是否小于当前时间。
    - **如果未过期：** 直接返回数据。
    - **如果已过期：**
        1. **立即将当前（旧的）数据返回给用户。** (保证服务可用性)
        2. **启动一个后台异步线程**，去执行“更新数据库”和“更新缓存”的操作。
        3. 后台线程获取最新的数据库数据，更新缓存中的 `info` 和 `expire_timestamp`。

**优点：**
- **彻底杜绝击穿：** 因为缓存物理上永不过期，所以永远不会有“缓存过期瞬间”这个时间点，从根本上避免了击穿。
- **用户体验好：** 用户请求几乎不会因为缓存重建而等待，总能拿到数据（即使是稍微旧一点的）。
- **性能高：** 没有锁竞争，没有大量请求阻塞。

**缺点：**
- **数据一致性是最终一致性：** 在缓存逻辑过期到后台线程更新完成这段时间内，用户读取到的是旧数据。需要业务能接受这种短暂的不一致。
- **需要额外的存储空间：** 需要在缓存中存储逻辑过期时间。
- **需要处理缓存更新失败的情况：** 如果后台线程更新缓存失败，那么这个 `key` 将会一直保持旧数据。需要监控和补偿机制。
- **不适用于所有数据：** 对于数据一致性要求极高的场景（如金融账户余额），不适用。

#### 后台自动刷新
这是方案二的一种更自动化、更优雅的变体。

不仅是在用户请求时发现数据过期才去刷新，而是建立一个**独立的后台任务**，**主动地、定期地**去刷新即将过期的热点 `key`。
**实现流程：**
1. **识别热点 Key：** 通过监控系统（如 Redis 的 `KEYS` 命令在生产环境禁用，应使用 `SCAN` 或监控工具）或业务逻辑，识别出系统中的热点 `key`。
2. **建立刷新任务：**
    - **方案A (定时任务)：** 使用调度器（如 Quartz, SchedulerX）创建一个定时任务，每隔一段时间（如 `key` 的 TTL 的 1/3 或 1/2）去查询数据库，主动更新缓存中的热点 `key`。
    - **方案B (消息队列)：** 在写入缓存时，同时将 `key` 和其 TTL 信息发送到一个延迟消息队列中。消息的延迟时间设置为 `TTL - Δt`（例如，提前 5 分钟）。当消费者收到消息时，说明这个 `key` 即将过期，于是消费者去数据库查询最新数据并更新缓存。
3. **用户请求：** 用户请求来时，缓存中的数据总是“新鲜”的（因为后台任务在它过期前就刷新了），几乎不会发生 Cache Miss。

**优点：**
- **用户体验最佳：** 用户请求几乎总是能命中缓存，且数据是新鲜的。
- **系统最稳定：** 将缓存维护的压力从业务请求中剥离，由后台系统承担，使主业务流程更稳定、更纯粹。
- **可预测性强：** 缓存刷新是可控的、可预测的，而不是被动的。

**缺点：**
- **架构复杂：** 需要引入后台任务系统或消息队列，增加了系统的复杂度和维护成本。
- **资源消耗：** 后台任务会持续消耗系统资源（CPU、网络、数据库连接）。
- **识别热点 Key 的挑战：** 如何准确、实时地识别出所有热点 `key` 是一个挑战，可能需要复杂的监控和分析。

### 方案对比
|方案|**一致性**|**性能/用户体验**|**实现复杂度**|**适用场景**|
|---|---|---|---|---|
|**互斥锁**|**强一致性**|**中** (有等待)|**中** (需处理锁)|对数据一致性要求高，可接受短暂延迟的场景。|
|**缓存永不过期**|**最终一致性**|**高** (无等待)|**低** (逻辑简单)|对可用性要求极高，能接受短暂数据延迟的场景。|
|**后台自动刷新**|**强/最终一致性**|**极高** (无等待)|**高** (架构复杂)|热点 Key 明确且固定，对系统稳定性和用户体验要求极高的场景。|

## 缓存穿透
查询一个根本不存在的数据。
请求会**绕过缓存，直接查询数据库**。当有**大量**此类请求（查询不存在的数据）时，数据库会因为处理大量无效查询而承受巨大压力，最终可能导致性能下降甚至服务不可用。

典型场景：
1. **恶意攻击：** 这是最常见也是最危险的原因。攻击者通过程序，构造大量随机的、不合法的 ID（如 UUID、负数、超长字符串等）来请求你的服务接口，意图耗尽你的数据库资源，造成服务拒绝。
2. **业务逻辑缺陷：**
    - **不存在的用户/商品 ID：** 用户在输入框里输入了一个不存在的用户 ID 进行查询，或者爬虫程序在爬取商品信息时，遍历了不存在的商品 ID。
    - **错误的查询条件：** 前端传递了错误的参数，导致后端生成了一个不可能存在的 `key` 去查询。
    - **数据已被删除：** 某条数据在数据库中已经被物理删除，但缓存中可能没有同步删除（或已过期），后续请求再来查询这个已删除的数据时，就会穿透。

### 核心方案
**拦截对“不存在数据”的请求，避免它们频繁地访问数据库。** 围绕这个核心，有以下几种主流的解决方案。

#### 缓存空对象
当一个请求查询数据库，发现数据不存在时，不要直接返回“不存在”，而是将这个“不存在”的结果也缓存起来。这样，后续再有相同的请求来查询这个不存在的 key 时，就可以直接从缓存中获取“不存在”的结论，而无需再次查询数据库。

**实现流程：**
1. 线程 A 请求 `key = "product:99999"`。
2. 先查缓存，发现没有（Cache Miss）。
3. 去数据库查询 `product_id = 99999`。
4. 数据库返回“该商品不存在”。
5. **将一个特殊的“空值”写入缓存**，`key` 仍然是 `"product:99999"`，`value` 可以是一个空字符串 `""`、一个特殊对象 `null`，或者一个固定的字符串如 `"NULL"`。
6. **为这个空值设置一个较短的过期时间**（例如 30 秒到 5 分钟）。
7. 返回“该商品不存在”给客户端。
8. 线程 B 在短时间内也请求 `key = "product:99999"`。
9. 查缓存，发现 `key` 存在，值为 `"NULL"`（Cache Hit）。
10. 直接返回“该商品不存在”给客户端，**整个过程没有访问数据库**。

**优点：**
- **实现简单，思路清晰：** 代码改动小，易于理解和维护。
- **效果立竿见影：** 能非常有效地拦截对同一不存在的 `key` 的重复查询。

**缺点：**
- **缓存空间浪费：** 如果有大量不存在的 `key` 被查询，会缓存大量的空值，占用宝贵的内存空间。
- **数据不一致窗口：** 如果数据库中后来**新增**了这条数据，在缓存的空值过期之前，应用仍然会读取到“不存在”的结果。这就是为什么需要给空值设置一个较短的 TTL，以便在数据更新后能尽快恢复一致。
- **对随机攻击效果有限：** 如果攻击者每次请求的 `key` 都不同（如随机 UUID），那么缓存空对象的方法就失效了，因为每个 `key` 都只会穿透一次，但成千上万个不同的 `key` 累加起来，依然会给数据库带来巨大压力。

#### 布隆过滤器
这是一种更高级、更优雅的解决方案，特别适用于**海量数据**且 `key` 随机性强的场景。

**原理：** 
布隆过滤器是一种**概率型数据结构**，用于**判断一个元素**是否**可能**存在于一个集合中。它有一个非常重要的特点：
- 如果布隆过滤器说“某个元素不存在”，那么它**一定不存在** 。
- 如果布隆过滤器说“某个元素存在”，那么它**可能存在，也可能不存在**（有一定的误判率）。**
利用这个特性，我们可以在查询缓存和数据库**之前**，先过一遍布隆过滤器。

**实现流程：**
1. **初始化：**
    - 将所有**可能存在**的 `key`（例如，数据库中所有商品的主键 ID）全部加载到布隆过滤器中。这个初始化过程通常在应用启动时完成。
    - 布隆过滤器本身可以存在 Redis 中（使用 `RedisBloom` 模块），也可以存在本地内存中（如 Google 的 Guava 库）。
2. **查询流程：**
    1. 线程 A 请求 `key = "product:99999"`。
    2. **先查询布隆过滤器**：`bf.exists product:99999`。
    3. **布隆过滤器返回 `false`（不存在）：**
        - **直接返回“数据不存在”**，**终止后续所有流程**。**这是布隆过滤器的核心价值所在！**
    4. **布隆过滤器返回 `true`（可能存在）：**
        - 说明这个 `key` **可能**在数据库中，需要继续后续的正常流程。
        - 去查缓存 -> 查数据库 -> 写入缓存等。

**优点：**
- **极高的拦截效率：** 对于确定不存在的 `key`，可以在第一步就将其拦截，完全避免了缓存和数据库的访问，对数据库的保护能力最强。
- **内存占用极小：** 布隆过滤器本身非常节省内存，相比于缓存大量空对象，内存效率高得多。一个能存储 1 亿个 `key` 的布隆过滤器，可能只需要几百 MB 的内存。
- **对随机攻击免疫：** 无论攻击者构造多少个不同的随机 `key`，只要它们不在布隆过滤器中，都会被瞬间拦截。

**缺点：**
- **存在误判：** 布隆过滤器可能会将一个不存在的 `key` 误判为存在。这会导致请求继续去查询缓存和数据库，但不会导致数据错误（因为数据库最终会返回不存在）。误判率可以通过调整布隆过滤器的大小和哈希函数数量来控制。
- **实现复杂度增加：** 需要引入和维护布隆过滤器。
- **不支持删除：** 标准的布隆过滤器不支持删除元素。如果数据库中的数据被删除，布隆过滤器中对应的 `key` 依然存在，这会导致误判。不过，对于缓存穿透场景，这通常不是大问题，因为查询一个已删除的数据，数据库最终也会返回“不存在”。如果需要精确删除，可以使用“计数布隆过滤器”（Counting Bloom Filter），但会占用更多内存。
- **数据同步问题：** 当有新数据插入数据库时，需要同步更新布隆过滤器，将新的 `key` 加入其中。这需要一个可靠的同步机制。

#### 前端/网关层拦截
在请求到达应用服务之前，在网关或前端层面进行参数校验，过滤掉明显非法的请求。
**实现方式：**
1. 参数合法性校验
2. IP 黑名单/限流
3. 前端校验

### 方案对比
|方案|**拦截效果**|**资源消耗**|**实现复杂度**|**适用场景**|
|---|---|---|---|---|
|**缓存空对象**|**中** (对重复请求有效)|**中** (占用缓存内存)|**低** (代码简单)|不存在的 `key` 集中且重复的场景。|
|**布隆过滤器**|**高** (对随机请求也有效)|**低** (内存效率高)|**中** (需维护布隆过滤器)|数据量巨大，`key` 随机性强，或面临恶意攻击的场景。|
|**前端/网关拦截**|**低** (仅拦截非法格式)|**低** (在网关层消耗)|**低** (网关配置)|作为**第一道防线**，拦截明显非法的请求。|

## 缓存雪崩
**缓存雪崩** 指的是在**某一个时间点**，**缓存中大量的 Key 同时失效**（过期），或者**缓存服务本身突然宕机不可用**。这导致瞬间有**海量的请求**无法从缓存获取数据，所有这些请求**像雪崩一样，全部涌向后端的数据库**，给数据库带来无法承受的巨大压力，最终导致数据库崩溃，进而引发整个系统的连锁崩溃。

缓存雪崩主要分为两大类：
1. 大量 Key 同时过期
	为大量的 Key 设置了相同的过期时间。
2. 缓存服务宕机
	1. **Redis 服务本身故障：**
	    - **进程 OOM：** Redis 内存使用超过 `maxmemory` 且淘汰策略不当，导致进程被操作系统杀死。
	    - **主从切换/集群故障：** 在主从或集群架构中，如果发生故障转移，可能会出现短暂的缓存不可用窗口。
	    - **网络分区：** 应用服务器与 Redis 服务器之间的网络中断，导致应用无法访问缓存。
	    - **Redis 所在宿主机宕机：** 物理机或虚拟机级别的故障。
	2. **容量规划不足：** 缓存服务承载的请求量或数据量超过了其自身处理能力，导致响应缓慢或超时，应用在等待超时后，会认为缓存不可用，转而去请求数据库。

#### 【事前】为 Key 设置随机的过期时间
- **原理：** 不要给所有 Key 设置一个固定的 TTL，而是在基础 TTL 上增加一个**随机值**。
- **实现：** 当你设置 Key 的过期时间时，可以这样计算：`final_ttl = base_ttl + random(0, N)`
- **效果：** 这样一来，即使有一批 Key 是在同一时间被设置的，它们的过期时间也会被分散在一个时间范围内（如 3600 秒到 3900 秒之间），从而避免了“雪崩式”的同时失效。

#### 【事前】缓存高可用架构
- **Redis 主从复制：**
    - **原理：** 搭建一主一从或一主多从的架构。主节点负责写操作，从节点通过复制同步主节点的数据，负责读操作。
    - **作用：** 当主节点宕机时，可以手动或通过哨兵/集群机制，快速将一个从节点提升为新的主节点，实现服务的高可用。这大大降低了单点故障的风险。
    - **缺点：** 主从切换过程中可能会有短暂的服务中断，且写操作依然在单点上。
- **Redis 哨兵模式：**
    - **原理：** 在主从复制的基础上，引入一组 Sentinel（哨兵）节点。Sentinel 会持续监控主从节点的状态。
    - **作用：** 当主节点宕机时，Sentinel 会自动完成故障发现、故障转移（选举新的主节点），并通知应用层新的主节点地址。整个过程是自动化的。
    - **优点：** 实现了 Redis 的高可用和自动故障转移。
- **Redis 集群模式：**
    - **原理：** 将数据自动分片到多个 Redis 节点。每个节点负责一部分数据（Slot）。集群内置了高可用机制，每个主节点都有对应的从节点。
    - **作用：**
        1. **水平扩展：** 解决了单机内存和性能瓶颈。
        2. **高可用：** 当某个主节点宕机时，其从节点会自动提升为主节点，保证服务不中断。
    - **优点：** 是目前 Redis 大规模、高可用部署的**最佳实践**。

#### 【事前】服务熔断与降级
这是一种应用层的自我保护机制。
当检测到对数据库的请求失败率或响应时间超过某个阈值时，熔断器会“跳闸”）。
- **熔断：** 在一段时间内，应用将**直接拒绝**所有对数据库的请求，而是返回一个预设的降级结果（如“系统繁忙，请稍后再试”、“默认数据”或“缓存过期数据”）。这保护了数据库，给了它喘息和恢复的机会。
- **降级：** 在熔断期间，提供一个有损但可用的服务，而不是完全不可用。
- **效果：** 即使缓存雪崩发生，应用层也能通过自我牺牲，保护住数据库这个核心资产，避免整个系统崩溃。

#### 【事中】限流
在应用层或网关层，对访问数据库的接口进行严格的限流。

#### 【事中】**队列削峰**
- **原理：** 对于非核心的、可异步化的写操作，不要直接写入数据库，而是将其发送到消息队列（如 Kafka, RabbitMQ）中。
- **作用：** 消息队列可以作为巨大的缓冲区，将瞬时的写洪峰削平，由后台消费者按照数据库能承受的速度来消费和处理。
- **效果：** 有效保护了数据库不被写请求冲垮。但此方案主要针对写操作，对于雪崩中大量的读请求效果有限。

#### 【事后】缓存预热
**核心思想：** 快速恢复缓存服务，并重新预热数据。
- **原理：** 在缓存服务重启或故障恢复后，**不要等用户请求来被动地、缓慢地重建缓存**。而是启动一个后台任务，主动地将热点数据从数据库加载到缓存中。
- **实现：**
    - **手动预热：** 在系统发布或重启后，运维或开发人员手动执行脚本，将已知的热点 Key 加载到缓存。
    - **自动预热：** 在应用中维护一个热点 Key 列表，应用启动时或定时任务中，自动加载这些 Key。
- **效果：** 避免了缓存“冷启动”阶段，让系统在恢复后能立即提供高性能的服务，防止二次雪崩。

# 集群

## 主从复制

Redis 主从复制 是指将一台 Redis 服务器的数据，复制到一台或多台其他 Redis 服务器的过程。
- **主节点：** 被复制的服务器，称为 **Master**。它负责处理所有的**写操作**（如 `SET`, `INCR`, `DEL`）。
- **从节点：** 复制主节点数据的服务器，称为 **Slave**（在 Redis 5.0+ 中也称为 **Replica**）。它默认情况下**只处理读操作**（如 `GET`, `HGET`），并将所有写操作转发给主节点。
- **主要目的** :
1. **数据冗余与高可用：** 这是主从复制最核心的价值。主节点是单点，一旦宕机，所有数据和服务都会丢失。通过主从复制，数据有了多个副本。当主节点宕机时，可以手动或自动地将一个从节点提升为新的主节点，从而实现服务的**故障转移**，保证系统的高可用性。
2. **读写分离：** 将读操作和写操作分离到不同的服务器上。主节点承担所有写压力，从节点分担读压力。这极大地提升了整个系统的**读性能**和**并发处理能力**，尤其适用于“读多写少”的场景。
3. **横向扩展：** 当单台 Redis 服务器的读请求成为瓶颈时，可以通过增加从节点的数量来**水平扩展**系统的读能力，轻松应对不断增长的读流量。
4. **数据备份与离线分析：** 从节点可以作为主节点的实时备份。此外，可以在一个从节点上执行耗时的离线分析任务（如 `KEYS`, `SCAN`），而不会影响主节点的性能。

### 核心工作原理
Redis 主从复制的核心是**命令传播**。主节点会将所有导致数据状态改变的**写命令**，发送给所有从节点，从节点执行这些相同的命令，从而实现与主节点的数据同步。
这个过程主要分为三个阶段：**连接建立**、**数据同步**和**命令传播**。

#### 阶段一：连接建立

#### 阶段二：数据同步
这是主从复制最复杂、最关键的阶段。当从节点首次连接主节点，或者因网络中断等原因断开重连后，需要确保自己的数据与主节点完全一致。Redis 使用了**两种同步方式**：**全量复制** 和 **增量复制**。

1. **全量复制**
当从节点与主节点数据差异过大，或者从节点是第一次复制时，会触发全量复制。
- **流程：**
    1. **从节点发送 `PSYNC` 命令：** 从节点向主节点发送 `PSYNC ? -1` 命令。`?` 表示从节点不知道主节点的 `runid`（运行 ID），`-1` 表示请求全量复制。
    2. **主节点响应 `FULLRESYNC`：** 主节点收到 `PSYNC ? -1` 后，会执行 `BGSAVE` 命令，在后台生成一个 **RDB 快照文件**。同时，主节点会开始**缓冲**从 `BGSAVE` 开始之后收到的所有写命令。
    3. **主节点发送 RDB 文件：** 主节点将生成的 RDB 文件发送给从节点。
    4. **从节点加载 RDB 文件：** 从节点接收到 RDB 文件后，会**清空自己的旧数据**，然后加载这个 RDB 文件，使其数据状态与主节点 `BGSAVE` 时刻的状态一致。
    5. **主节点发送缓冲的命令：** 从节点加载完 RDB 文件后，主节点会将之前缓冲的那些写命令，发送给从节点。从节点执行这些命令，从而追赶上主节点在生成 RDB 期间的最新状态。
    6. **同步完成：** 至此，全量复制完成，主从节点数据进入一致状态。
- **缺点：** 耗时、资源开销大、网络开销大。

2. **增量复制**
增量复制是 Redis 2.8 之后引入的重大优化，用于处理**短暂断线重连**后的同步。它避免了每次断线重连都进行昂贵的全量复制。
- **核心概念：复制偏移量 和 复制积压缓冲区**
    - **复制偏移量：**
        - 主节点和从节点都会维护一个**复制偏移量**（`master_repl_offset` 和 `slave_repl_offset`）。
        - 主节点每发送一个写命令，自己的偏移量就增加。
        - 从节点每接收一个写命令，自己的偏移量就增加。
        - 通过比较主从节点的偏移量，可以知道数据是否同步。
    - **复制积压缓冲区：**
        - 主节点内部维护了一个固定长度的、先进先出的**环形缓冲区**（默认 1MB，可通过 `repl-backlog-size` 配置）。
        - 主节点在将写命令发送给从节点的同时，也会将写命令**写入**到这个缓冲区中。
        - 这个缓冲区存储了最近的一部分写命令，相当于一个“命令日志”。
- **流程：**
    1. **从节点发送 `PSYNC` 命令：** 当一个已经与主节点同步过的从节点断线重连时，它会发送 `PSYNC <master_runid> <slave_repl_offset>`。
        - `<master_runid>`：上次同步的主节点 ID，确保是从同一个主节点复制。
        - `<slave_repl_offset>`：从节点当前的复制偏移量。
    2. **主节点判断：** 主节点收到 `PSYNC` 后，会进行判断：
        - 如果请求的 `runid` 与自己的不匹配，说明从节点记错了主节点，会触发全量复制。
        - 如果 `runid` 匹配，主节点会检查从节点的 `repl_offset` 是否还在自己的**复制积压缓冲区**中。
    3. **执行增量复制：**
        - **如果在缓冲区内：** 主节点会从 `slave_repl_offset` 开始，将积压缓冲区中后续的所有写命令，发送给从节点。从节点执行这些命令，即可完成同步。这个过程非常轻量。
        - **如果不在缓冲区内：** 说明断线时间太长，主节点缓冲区中已经找不到从节点需要的数据了。此时，主节点会退而求其次，触发**全量复制**。
- **优点：**
    - **高效：** 对于短时间内的网络抖动，增量复制非常快速，避免了全量复制的巨大开销。
    - **实时性好：** 从节点能很快追赶上主节点的最新状态。

#### 阶段三：命令传播
在数据同步（无论是全量还是增量）完成之后，主从节点就进入了**命令传播**阶段。
- **流程：**
    1. 主节点每执行一个写命令，除了修改自己的数据外，还会将该命令**异步地**发送给所有已连接的从节点。
    2. 从节点接收到命令后，执行该命令，使自己的数据状态与主节点保持一致。
    3. 这个过程是持续的、实时的，只要主节点有写操作，就会不断进行。
- **心跳机制：**
    - 为了维持连接和检测状态，从节点会**每秒**向主节点发送一个 `REPLCONF ACK <replication_offset>` 命令。
    - **作用：**
        1. **心跳：** 告诉主节点自己还活着。
        2. **报告偏移量：** 让主节点知道从节点的同步进度。
        3. **超时检测：** 如果主节点长时间（可配置，默认 60 秒）没有收到从节点的 `ACK`，会认为从节点已下线，断开连接。

### 常见问题

#### 复制延迟
- **现象：** 主节点的数据已经更新，但从节点还未同步，导致从节点读取到的是旧数据。
- **原因：**
    - 主节点写操作繁忙，命令传播队列有积压。
    - 网络延迟或带宽不足。
    - 从节点自身负载过高，执行命令慢。
- **应对：**
    - 优化网络。
    - 监控从节点负载。
    - 对于强一致性要求的读操作，可以强制路由到主节点。

#### 主从节点数据不一致
- **现象：** 由于网络分区或其他原因，主节点在某个时刻没有从节点，但仍然在处理写操作。当网络恢复后，从节点可能会丢失一部分数据，导致与主节点不一致。
- **应对：** Redis 本身是异步复制，无法完全避免这种情况。对于要求数据强一致性的场景，可以考虑使用 `WAIT` 命令（让主节点等待至少 N 个从节点复制成功后再返回），但这会牺牲性能。更彻底的方案是使用 Redis Cluster 或其他强一致性数据库。

#### 主节点宕机与故障转移
- **问题：** 主节点是单点，一旦宕机，整个写服务就中断了。
- **解决方案：** **Redis Sentinel（哨兵）** 或 **Redis Cluster**。
    - **Sentinel**：一个独立的监控和自动故障转移系统。它能监控主从节点，当主节点宕机时，能自动将从节点提升为新的主节点，并通知其他从节点去复制新的主节点。
    - **Cluster**：内置了高可用和分片功能，当某个分片的主节点宕机时，其从节点会自动提升为主节点。

## 哨兵
**Redis Sentinel**（哨兵）是 Redis 官方提供的一个**高可用解决方案**。它是一个或多个 **Sentinel 进程**组成的**分布式系统**，用于**监控**运行中的 Redis 主从实例，并在主节点发生故障时，**自动**完成故障发现和故障转移，并将通知应用层新的主节点地址。

- **解决什么问题：**
	在纯主从复制架构中，我们面临一个致命的弱点：**主节点是单点**。
	- **问题：** 当主节点宕机时：
	1. 所有写服务中断。
	2. 从节点虽然拥有完整数据，但它们仍然是“只读”的，无法自动提升为主节点。
	3. 需要人工介入：运维人员发现故障，手动执行 `SLAVEOF NO ONE` 将一个从节点提升为主节点，然后修改其他从节点的配置，让它们去复制新的主节点，最后通知应用层更新主节点地址。
	4. 这个过程**耗时、易出错**，且在恢复期间服务不可用。

- **三大核心任务：**
	1. **监控：** Sentinel 会持续不断地检查你的主节点和所有从节点是否在正常工作。
	2. **通知：** 当被监控的 Redis 实例出现问题时，Sentinel 可以通过 API（如脚本、其他程序）向系统管理员或应用程序发送通知。
	3. **自动故障转移：** 这是 Sentinel 最核心、最强大的功能。当一个主节点无法正常工作时，Sentinel 会启动一个自动化的流程，将一个从节点选举为新的主节点，并让其他从节点开始复制新的主节点，同时通知客户端新的主节点地址。

### 核心机制：主观下线 与 客观下线

#### 1. 主观下线
- **定义：** 单个 Sentinel 实例认为某个 Redis 实例（主或从）已经“下线”了。
- **判断依据：** Sentinel 默认每秒向所有被监控的实例发送一个 `PING` 命令。如果在 `down-after-milliseconds`（可配置，默认 30 秒）时间内，实例没有有效回复（如 `PONG`, `LOADING`, `MASTERDOWN`），那么这个 Sentinel 就会认为该实例进入了 <mark class="hltr-pink">**SDOWN**</mark> 状态。
- **特点：** “主观”意味着这只是**一个 Sentinel 的个人观点**。它可能是因为 Sentinel 和实例之间的网络分区导致的误判。
#### 2. 客观下线
- **定义：** **足够数量**的 Sentinel 都认为某个主节点已经“下线”了。
- **判断依据：** 当一个 Sentinel 将主节点标记为 SDOWN 后，它会向其他所有 Sentinel 发送一个 `SENTINEL is-master-down-by-addr` 命令，询问它们对该主节点的看法。
- **投票：** 其他 Sentinel 会根据自己与主节点的连接情况，回复“是”或“否”。
- **达成共识：** 当收集到足够多的“是”票数（这个数量由 `quorum` 配置项决定）后，Sentinel 就会将该主节点标记为 <mark class="hltr-pink"> **ODOWN** </mark>。
- **特点：** “客观”意味着这是**一个 Sentinel 集群的集体决策**。它有效避免了因单个 Sentinel 的误判而导致的故障转移，大大提高了系统的健壮性。
**关键点：** **ODOWN 只针对主节点！** 从节点不会被标记为 ODOWN，因为从节点的故障不会影响整个系统的写服务，不需要触发复杂的故障转移流程。

### 核心流程：自动故障转移
当一个主节点被标记为 ODOWN 后，Sentinel 集群就会启动自动故障转移流程。这个过程由一个**领导者 Sentinel** 来协调。
1. **选举领导者 Sentinel**
    - **问题：** 在一个 Sentinel 集群中，由谁来主导这次故障转移？不能所有 Sentinel 一拥而上。
    - **解决方案：** Sentinel 集群会通过 **Raft 算法** 选举出一个领导者 Sentinel，由它来负责执行后续的故障转移步骤。只有领导者会执行操作，其他 Sentinel 会观察和确认。
2. **选择新的主节点**
    - 领导者 Sentinel 会在所有健康的从节点中，根据一系列**优先级**选择一个最合适的作为新的主节点。
    - **选择标准（按优先级排序）：**
        1. **优先级：** 从节点可以通过 `replica-priority` 配置项设置一个优先级（数字越小，优先级越高）。领导者会选择优先级最高的从节点。
        2. **复制偏移量：** 如果优先级相同，会选择**复制偏移量最大**的那个从节点。这意味着这个从节点的数据最新，与旧主节点的数据最接近。
        3. **Run ID：** 如果偏移量也相同，会选择 **Run ID 最小**的那个从节点。这是一个确定性的选择，避免随机性。
3. **提升新主节点**
    - 领导者 Sentinel 向被选中的从节点发送 `SLAVEOF NO ONE` 命令，使其成为新的主节点。
    - 新主节点会丢弃其旧的主从复制 ID，生成一个新的。
4. **修改其他从节点的复制目标**
    - 领导者 Sentinel 向所有其他从节点发送 `SLAVEOF <new_master_ip> <new_master_port>` 命令，让它们开始复制新的主节点。
5. **通知旧主节点（如果它恢复了）**
    - 如果旧的（宕机的）主节点此时恢复了，领导者 Sentinel 也会向它发送 `SLAVEOF` 命令，使其成为新主节点的一个从节点。
6. **更新配置和通知客户端**
    - 故障转移完成后，Sentinel 集群会更新自己的配置信息，记录新的主节点。
    - Sentinel 会通过 `+switch-master` 频道发布一个消息，告知所有订阅了该频道的客户端主节点已经发生变化。客户端可以监听这个事件，并更新自己的连接信息。

### 部署
**关键原则：** **Sentinel 本身也必须是高可用的！** 如果只有一个 Sentinel 进程，那么这个 Sentinel 自己就成了单点。因此，**至少需要部署 3 个或 5 个（奇数个）Sentinel 实例**，并将它们分布在不同的物理机或虚拟机上。

### 常见问题
#### 脑裂
- **现象：** 由于网络分区，主节点与 Sentinel 集群（以及从节点）失联，但主节点自身仍在运行，并可能还在接收写请求。此时 Sentinel 集群会认为主节点已下线，并选举一个新的主节点。这就出现了**两个主节点**，一个在旧分区，一个在新分区，导致数据写入分裂。
- **Sentinel 的应对：**
    - `min-replicas-to-write` 和 `min-replicas-max-lag` 配置。
    - 主节点可以配置：`min-replicas-to-write 1` 和 `min-replicas-max-lag 10`。
    - 这意味着：**主节点至少需要有 1 个从节点的复制延迟小于 10 秒，才能接受写请求。**
    - 当脑裂发生时，旧主节点因为与所有从节点失联，不满足 `min-replicas-to-write` 条件，会**拒绝所有写请求**，从而避免了数据写入旧分区，大大降低了脑裂的风险。
- **最佳实践：** 对于要求数据强一致性的场景，务必在主节点上配置这两个参数。

#### Sentinel 宕机
- **问题：** 如果部署的 3 个 Sentinel 中，有 2 个同时宕机，那么剩下的 1 个 Sentinel 无法达到 `quorum`（假设为 2），将无法进行故障转移。
- **应对：** 这正是需要**分布式部署**和**奇数个** Sentinel 的原因。它能容忍最多 `(N-1)/2` 个 Sentinel 实例宕机。对于 3 个 Sentinel 的部署，可以容忍 1 个宕机。对于 5 个，可以容忍 2 个。确保你的 Sentinel 部署本身是高可用的。

#### 从节点选举失败
- **问题：** 在故障转移时，如果所有从节点都与旧主节点失联，或者所有从节点本身也不健康，那么 Sentinel 将无法找到合适的从节点来提升为主节点。
- **应对：** 保证从节点的健康和网络稳定性。如果所有从节点都不可用，那么故障转移本身就失去了意义。此时，系统需要人工介入。

## 集群 Cluster
**Redis Cluster** 是 Redis 官方提供的**分布式解决方案**。它将数据**自动分片**到多个 Redis 节点，并内置了**高可用**和**故障转移**能力，能够水平扩展以支持**TB级数据**和**百万级QPS**。

- **解决的核心问题**
	在讨论 Cluster 之前，我们回顾一下之前方案的局限性：
	- **单机 Redis：** 内存、CPU、网络 I/O 都有单机瓶颈，且是单点故障。
	- **主从 + Sentinel：**
	    - 解决了**高可用**问题（通过 Sentinel 自动故障转移）。
	    - 解决了**读扩展**问题（通过增加从节点）。
	    - **但没有解决写扩展问题！** 所有的写操作仍然集中在单个主节点上。当写请求量巨大时，单个主节点依然是瓶颈。
	    - **没有解决数据容量问题！** 单个主节点的内存是有限的，当数据量超过单机内存时，就无法存储了。
- **核心特性**
	1. **数据分片：** 将整个数据集自动分割成 16384 个哈希槽，并分布到集群中的多个主节点上。每个主节点负责一部分哈希槽。
	2. **高可用：** 采用**主从模型**，每个主节点可以有多个从节点。当主节点宕机时，其从节点会自动提升为新的主节点，实现故障转移。
	3. **水平扩展：** 当需要增加存储容量或吞吐量时，可以动态地向集群中添加新的节点，并自动将一部分哈希槽迁移到新节点上，实现平滑扩容。
	4. **去中心化：** 集群中的所有节点通过 **Gossip 协议** 进行通信，共同维护集群的状态信息，没有像 Sentinel 那样的中心化监控节点。客户端可以连接到集群中的任意一个节点。

## `SETNX` 命令

`SETNX` 是 **"SET if Not eXists"** 的缩写，意思是如果给定的键不存在，就设置这个键的值，否则什么也不做。

它的返回值非常简单且明确：
- **1**：表示键**设置成功**。
- **0**：表示键**设置失败**，因为该键已经存在。

### 如何使用 SETNX 实现分布式锁？

使用 `SETNX` 实现分布式锁的基本思想是：利用它 **“要么成功，要么失败”** 的原子性。当一个客户端需要获取锁时，它尝试去设置一个特定的键，如果设置成功，就意味着它获得了锁。

下面是使用 `SETNX` 实现分布式锁的简化步骤：

1. **加锁（Lock）**
    
    - 客户端 A 尝试执行 `SETNX lock_key 1`。
        
    - 如果命令返回 **1**，表示设置成功，客户端 A **获得了锁**，可以执行业务逻辑。
        
    - 如果命令返回 **0**，表示设置失败，客户端 B 已经持有锁，客户端 A **获取锁失败**，需要等待或重试。
        
2. **解锁（Unlock）**
    
    - 客户端 A 执行完业务逻辑后，需要执行 `DEL lock_key` 来**释放锁**。

### 缺陷 和 优化

<mark class="hltr-pink">死锁（Deadlock）：</mark>

- 如果客户端 A 在获得锁后，还没来得及执行 `DEL` 命令就崩溃或宕机了，那么 `lock_key` 将永远不会被删除。
    
- 这会导致其他所有客户端都无法再获得这个锁，从而造成死锁。
    

为了解决这个问题，我们需要引入一个**过期时间（Expiration Time）**。

在 Redis 2.6.12 之后的版本中，`SET` 命令引入了**原子性的** `NX` 和 `EX` 参数，可以完美地解决这个问题。

一个**正确的**分布式锁实现应该像这样：

`SET lock_key unique_value NX EX 10`
- **`lock_key`**：用作锁的键名。
- **`unique_value`**：一个唯一的值，通常是 UUID。**这是为了避免锁被其他客户端误删**。
- **`NX`**：`Not Exists`，只有当 `lock_key` 不存在时才设置。
- **`EX 10`**：`Expiration`，设置键的过期时间为 10 秒。

这样，即使客户端在释放锁之前崩溃，这个锁也会在 10 秒后自动过期，防止死锁。【但是不是原子性】

### 解锁的原子性

当客户端需要释放锁时，不能简单地执行 DEL，因为这可能导致误删。例如，客户端 A 业务执行时间超过了锁的过期时间，锁自动释放，客户端 B 获得了锁。这时客户端 A 恢复过来执行 DEL，就会把客户端 B 的锁给删除了。

正确的解锁操作应该是一个**原子性的 Lua 脚本**，确保只有持有锁的客户端才能成功释放锁。
```Lua
if redis.call("get",KEYS[1]) == ARGV[1]
then
    return redis.call("del",KEYS[1])
else
    return 0
end
```
这个脚本的逻辑是：**先检查 `lock_key` 的值是否等于我们设置的 `unique_value`，如果相等，才执行删除操作**。

# 不用Reidsson分布式锁，如何保证集群线程安全？
## **1. ZooKeeper 分布式锁**
你可以把 **ZooKeeper** 想象成一个**分布式文件系统**，它有强一致性的特性，非常适合用来做协调服务。
- **工作原理**：
    1. **加锁**：每个要获取锁的客户端都在 ZooKeeper 的一个指定路径下创建一个**临时顺序节点**。
    2. **获取锁**：客户端会检查自己创建的节点的序号是否是当前路径下序号最小的。
    3. **如果序号最小**，表示获取锁成功。
    4. **如果序号不是最小**，客户端就监听（Watch）前一个节点的删除事件。
    5. **解锁**：当持有锁的客户端执行完业务逻辑后，删除自己创建的节点。前一个节点被删除后，ZooKeeper 会通知下一个节点，让它去检查自己是否是最小的节点，如果是则获取锁。
- **优点**：
    - **强一致性**：ZooKeeper 保证了数据操作的原子性和顺序性。
    - **天然的监听机制**：通过 Watch 机制，可以避免客户端的轮询，减轻了服务器压力。
- **缺点**：
    - **性能相对较低**：因为需要向 ZooKeeper 集群写入数据，并且涉及到网络通信和节点监听，在高并发场景下性能不如 Redis。
    - **实现复杂**：需要自己处理顺序节点、监听等逻辑，比 Redis 复杂得多。

## **2. 数据库分布式锁**
你可以利用数据库的**行锁**或**唯一索引**来实现分布式锁。
- **工作原理**：
    1. **加锁**：客户端尝试在数据库中插入一条记录（例如，一个 `lock` 表），如果插入成功，则获得锁。
    2. **利用唯一索引**：在 `lock` 表的 `lock_name` 字段上创建唯一索引。如果多个客户端同时插入同一条记录，只有一个会成功。
    3. **解锁**：执行完业务逻辑后，删除这条记录。
- **优点**：
    - **简单易实现**：利用数据库的原子操作，逻辑简单。
    - **强一致性**：依赖于数据库的事务和 ACID 特性。
- **缺点**：
    - **性能问题**：数据库的并发处理能力有限，锁的竞争可能导致数据库成为性能瓶颈。
    - **死锁问题**：如果客户端崩溃，没有释放锁，会造成死锁。需要配合定时任务来清理超时的锁。
    - **扩展性差**：单个数据库实例的并发量有限，难以扩展。

# Redis 事务
Redis 支持事务，**但它的事务与 MySQL 中的事务有所不同**。MySQL 中的事务主要支持 ACID 的特性，而 Redis 中的事务主要保证的是多个命令执行的原子性，即所有的命令在一个原子操作中执行，不会被打断。

<mark class="hltr-grey">我们一一对比下 Redis 中的事务是否符合 ACID：</mark>
- 原子性：Redis 事务提供部分原子性，但不是完整的ACID事务。如果事务中的一个命令失败，其他命令仍然会继续执行，**事务不会自动回滚**。
- 一致性：无法回滚，无法保证一致性。
- 隔离性：Redis 事务使用的是单线程模型，在执行事务的过程中，其他命令不会插入到事务执行的过程中，但无法设置不同隔离级别。
- 持久性：Redis 事务的持久性依赖于配置（如 RDB 快照和 AOF），但理论上还是会丢数据。

还有一个很重要的点，就是 **MySQL 中的事务是支持回滚的，而 Redis 中的事务是不支持回滚的**。
### Redis 的事务

Redis 支持事务，通过使用 `MULTI`、`EXEC`、`WATCH` 和 `DISCARD` 等命令来实现。事务中的命令会被排队并在调用 `EXEC` 时一次性执行，保证了事务的原子性。

具体实现流程如下：

1. **开始事务**：使用 `MULTI` 命令开始一个事务，之后的所有命令都会被排队。
2. **添加命令**：在事务中添加命令，这些命令不会立即执行，而是存储在队列中。
3. **执行事务**：使用 `EXEC` 命令执行队列中的所有命令，确保原子性。
4. **取消事务**：使用 `DISCARD` 命令可以放弃事务，清空命令队列。
5. **监视键**：使用 `WATCH` 命令可以监视一个或多个键，如果在事务执行前这些键被修改，则 `EXEC` 将不会执行，确保数据一致性。

### Redis 事务不支持回滚功能

Redis 事务一旦执行，所有命令都会被应用到数据中，即使其中某个命令失败。因此，在设计时需充分考虑可能的错误。

从 Redis 2.6.5 开始，服务器会在累计命令的过程中检测到错误，此时执行 EXEC 会拒绝执行事务，并且返回一个错误，同时求丢该事务。

但如果事务执行的过程中发生了错误，**Redis 会继续执行剩下的命令，而不会对事务进行回滚**，这个是 Redis 和 MySQL 最不一样的地方，并且也**不支持多种隔离级别的设置**，因为 Redis 是单线程执行，只能是串行隔离级别。

可以认为 Redis 的事务是一个残血的事务，**更多只是一个噱头，不是我们平时理解的事务**。

# Lua脚本

Redis 的 Lua 脚本功能允许用户在 Redis 服务器端执行自定义的 Lua 脚本，以实现原子操作和复杂逻辑。 其核心点包括：

- **原子性**：Lua 脚本的所有命令在执行过程中是原子的，避免了并发修改带来的问题。
- **减少网络往返次数**：通过在服务器端执行脚本，减少了客户端和服务器之间的网络往返次数，提高了性能。
- **复杂操作**：可以在 Lua 脚本中执行复杂的逻辑，比如批量更新、条件更新等，超过了单个 Redis 命令的能力。

例如常见基于 Redis 实现分布式锁就需要结合 lua 脚本来实现。

**lua 本身是不具备原子性的**，但由于 Redis 的命令是单线程执行的，它会把整个 lua 脚本作为一个命令执行，会阻塞其间接受到的其他命令，这就保证了 lua 脚本的原子性。

# Redis 序列化协议
Redis客户端和服务端之间通信所使用的协议，叫做 **RESP (REdis Serialization Protocol)**，也就是Redis序列化协议。从Redis 2.0开始，它就成为了标准的通信方式。

它规定了客户端和服务端之间发送的每一条消息，都必须遵循一个简单的格式：
**消息的第一个字节，必须是一个特殊的字符，用来表明这条消息的数据类型。**
- `+` 代表这是一句简单的、单行的肯定答复。    
- `-` 代表这是一条错误信息。    
- `:` 代表这是一个数字。    
- `$` 代表后面跟着一段确定长度的文本。    
- `*` 代表后面跟着一个列表（比如一条命令和它的参数）。   
这套规则非常简单，所以客户端和服务端的代码实现起来很容易。同时，它又是文本协议，人类可以直接阅读，非常便于调试。你甚至可以用`telnet`工具直接连接Redis服务器，手动敲出RESP格式的命令来和它交互。

RESP协议将所有传输的数据结构分为5种类型，每种类型都以一个特殊字符开头，并以`\r\n` (CRLF) 作为结尾。
#### 1. 五种数据类型
1. **Simple Strings (简单字符串)**    
    - **格式**: 以 `+` 开头。        
    - **例子**: `+OK\r\n`        
    - **用途**: 用于服务器返回简单的、非二进制的状态信息，比如`SET`命令成功后返回的"OK"。        
2. **Errors (错误)**    
    - **格式**: 以 `-` 开头。        
    - **例子**: `-ERR unknown command 'badcmd'\r\n`        
    - **用途**: 当出现错误时，服务器返回给客户端的信息。        
3. **Integers (整数)**    
    - **格式**: 以 `:` 开头，后跟一个十进制数字。        
    - **例子**: `:1000\r\n`        
    - **用途**: 服务器返回整数的场景，比如 `INCR`、`LLEN`、`SCARD` 等命令的返回值。        
4. **Bulk Strings (批量字符串)**    
    - **格式**: 以 `$` 开头，后跟字符串的字节长度，然后是`\r\n`，最后是字符串本身和`\r\n`。        
    - **例子**: 要表示字符串 "hello"，RESP格式为 `$5\r\nhello\r\n`。        
    - **用途**: 这是最重要的类型之一，用于传输**二进制安全**的字符串，最大长度可达512MB。客户端发送的命令参数、服务器返回的`GET`结果等，都用这种格式。它还能表示`null`（`$-1\r\n`）。        
5. **Arrays (数组)**    
    - **格式**: 以 `*` 开头，后跟数组中元素的数量，然后是`\r\n`，接着是数组中每个元素自身的RESP格式。        
    - **例子**: 一个包含两个Bulk String "hello" 和 "world" 的数组表示为： *2\r\n$5\r\nhello\r\n$5\r\nworld\r\n`
    - **用途**: 这是另一个至关重要的类型。**客户端向服务器发送的所有命令，都是以RESP数组的形式发送的**。服务器返回多个值的命令（如`LRANGE`, `HGETALL`）也使用数组。

#### 2. 命令交互示例
让我们来看一个完整的 `SET mykey "hello"` 命令的交互过程，你就能彻底理解RESP了：
1. **客户端封装命令**: 客户端会将 `SET mykey "hello"` 这条命令，转换成一个包含三个元素的RESP数组。    
    - 元素1: "SET" -> `$3\r\nSET\r\n`        
    - 元素2: "mykey" -> `$5\r\nmykey\r\n`        
    - 元素3: "hello" -> `$5\r\nhello\r\n`        
    - 整个数组: `*3\r\n`        
    所以，客户端最终发送给服务器的原始字节流是： `*3\r\n$3\r\nSET\r\n$5\r\nmykey\r\n$5\r\nhello\r\n`
2. **服务器解析与执行**: 服务器收到这段字节流，看到 `*3` 就知道接下来要解析一个包含3个元素的数组。它依次解析出 "SET", "mykey", "hello"，然后执行相应的操作。    
3. **服务器返回结果**: `SET` 命令执行成功后，服务器需要返回 "OK"。它会将其封装成一个Simple String格式的响应。 所以，服务器最终返回给客户端的原始字节流是： `+OK\r\n`    

**总结一下RESP的优点：**
- 实现简单，客户端开发容易。    
- 协议解析性能高。因为数据长度是预先指定的，所以不需要扫描特殊字符来判断结尾，可以直接读取指定长度的数据。    
- 人类可读，便于调试。
