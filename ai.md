
# 为什么选择 Spring Boot 3 + Spring AI 这个技术栈？

- **突出 Spring AI 的优势：** 结合 Java 生态（尤其是 Spring）的无缝集成、强大的依赖注入和 AOP 功能、企业级的稳定性和成熟的社区生态。如果公司本身是 Java 技术栈，这一点是巨大优势。
    
- **对比 LangChain/LlamaIndex：** 可以承认 Python 生态在 AI 领域的快速和灵活，但强调 Spring AI 在与现有企业级 Java 应用整合方面的便利性、类型安全和性能优势。
    
- 说明这是一个经过深思熟虑的权衡（Trade-off）结果，而不是盲目跟风。

# 介绍一下 AI 超级智能体项目的整体架构设计，有哪些模块？

1）客户端：这是用户与系统交互的前端界面，负责发送用户请求和展示结果。这个项目我使用了 AI 来生成前端。

2）API 接口层：作为后端应用的门户，负责接收客户端请求，并与下游服务通信。我使用 Spring MVC 实现。

3）AI 应用层：这是项目的核心业务逻辑所在，包含了我开发的具体AI应用，如“AI 恋爱大师”和“ YuManus 自主规划智能体”。

4）AI 能力层：这一层封装了底层的AI核心技术，主要包括：

- 大模型：提供基础的自然语言理解、生成和推理能力。
- RAG 知识库：通过检索增强生成技术，使AI能够利用外部知识回答问题。
- 工具：允许AI调用外部函数或服务来执行特定任务，如文件操作、联网搜索等。
- MCP：一种标准协议，用于AI模型与外部工具和服务进行标准化交互。

我利用 Spring AI 框架实现了这一层，并且进行了一些定制开发。

5）外部服务：为 AI 能力层提供支持，包括：

- 向量数据库：用于存储文档向量，支持RAG的相似性搜索。
- 三方 API：如搜索API、地图API等，通过工具调用或MCP服务接入。
- 文件存储：用于存储和管理相关文件。
- MCP服务：外部提供的符合MCP标准的工具服务。
# 什么是 Spring AI 框架？它有哪些核心特性？ 

Spring AI 是一个基于 Spring 生态系统的 AI 应用开发框架，主要目标是简化 AI 功能与 Spring 应用的集成。它通过提供统一的 API 和抽象，让 Java 开发者可以更便捷地接入和使用各种 AI 大模型及相关技术，忽略底层实现的差异。通过官网，我们可以了解到 Spring AI 框架有以下核心特性：

- 跨 AI 供应商的可移植 API 支持：为聊天、文本转图像和嵌入模型提供了统一的 API，支持同步和流式调用，支持访问特定模型的功能。
- 广泛的 AI 模型供应商支持：支持包括 OpenAI、微软 Azure、Anthropic、Google、Ollama 在内的主流 AI 模型供应商。
- 结构化输出：能够将 AI 模型的输出自动映射到 POJO，方便在 Java 应用中处理。
- 向量数据库集成：支持与多种主流向量数据库的集成，还提供了跨向量存储的可移植 API。
- 工具/函数调用 (Tool/Function Calling)：支持模型请求执行客户端定义的工具和函数，扩展模型的能力。
- ETL框架：提供了文档抽取、转换和加载（ETL）的组件，用于数据工程和 RAG 知识库的构建。
- Spring Boot 自动配置与启动器：为 AI 模型和向量存储提供了自动配置和 Starter 依赖，简化项目配置。
- ChatClient API：提供了类似于 WebClient 和 RestClient 的流式API，提高与 AI 模型交互的便捷性。
- Advisors API：一种拦截器机制，封装了常见的 AI 功能，如对话记忆、RAG 等，还支持自定义，可以在调用 AI 前后执行额外操作。
# 你如何利用 Spring AI 开发应用？用到了哪些特性？

1）大模型接入与调用：

- 使用`spring-ai-alibaba-starter`和`spring-ai-ollama-spring-boot-starter`简化大模型的配置和集成。
- 通过 Spring AI 的`ChatModel`和更高级的`ChatClient API`与 AI 大模型交互，实现对话功能。

2）Advisors：

- 利用`MessageChatMemoryAdvisor`实现了多轮对话中的上下文记忆功能。
- 为了增强应用的可观测性，我还自定义了 Advisor，例如用于记录AI请求和响应日志的`MyLoggerAdvisor`，以及用于尝试提高模型推理能力的`ReReadingAdvisor`。

3）对话记忆：

- 项目初期使用了基于内存的`InMemoryChatMemory`。
- 为了实现对话记忆的持久化，我自定义了`FileBasedChatMemory`，并使用 Kryo 序列化库解决了`Message`对象层级复杂难以直接序列化的问题。

4）结构化输出：通过 Spring AI 的结构化输出转换器，我把 AI 模型的文本输出直接转换为Java 对象，方便后端处理和使用。

5）Prompt 模板：利用`PromptTemplate`管理和动态生成提示词，比如使用占位符替换变量、从外部文件加载复杂的提示词模板，提高了提示词的可维护性和灵活性。

6）RAG (检索增强生成)：

- 文档 ETL：使用了 Spring AI 的 ETL Pipeline 组件，如`MarkdownDocumentReader`加载本地Markdown知识库文档、`TokenTextSplitter`进行文本分割、`KeywordMetadataEnricher`自动为文档添加关键词元数据。
- 向量存储：集成`SimpleVectorStore`和`PgVectorStore`用于存储和检索文档向量。在集成 `PgVectorStore` 时，我还解决了多个`EmbeddingModel Bean`冲突的问题。
- 检索与增强：使用`QuestionAnswerAdvisor`和更灵活的`RetrievalAugmentationAdvisor`实现 RAG 流程，后者还结合了`VectorStoreDocumentRetriever`和`ContextualQueryAugmenter`等组件进行查询优化和空上下文处理。还实践了查询重写（`RewriteQueryTransformer`）等预检索优化技术。

7）工具调用 (Tool Calling)：

- 通过`@Tool`和`@ToolParam`注解定义了多种外部工具，如文件操作、联网搜索、PDF 生成等，并使用`ChatClient`的`tools()`方法将这些工具注册给 AI 模型，让 AI 能够调用这些工具完成特定任务。
- 在构建 YuManus 智能体时，我手动控制了工具的执行流程（ReAct模式），通过`DashScopeChatOptions`禁用 Spring AI 内部的自动工具执行，更方便、更精细地管理思考-行动循环。

8）MCP (模型上下文协议) 集成：

- MCP客户端：使用`spring-ai-mcp-client-spring-boot-starter`，配置 stdio 和 SSE 连接方式来调用外部 MCP 服务。通过`ToolCallbackProvider`将 MCP 服务提供的工具无缝集成到`ChatClient`的工具调用机制中。
- MCP 服务端：基于`spring-ai-mcp-server-webmvc-spring-boot-starter`开发了自定义的图片搜索 MCP 服务，使用`@Tool`注解暴露工具，然后通过`ToolCallbackProvider Bean`进行注册。

# 如何实现程序和 AI 大模型的集成？有哪些方式？ 

1. SDK接入

    * 实现方式：通过大模型提供的官方软件开发工具包直接调用AI服务。

    * 具体做法：在项目中引入阿里灵犀的SDK依赖，配置API Key，然后构造调用实例发送请求。

    * 优点：类型安全、错误处理完善、性能优化好。

    * 缺点：与特定厂商绑定、依赖特定版本、可能增加项目体积。

    * 适用场景：需要深度集成或特定模型提供商服务、对性能要求高的场景。

  

2. HTTP接入

    * 实现方式：直接通过REST API发送HTTP请求调用模型。

    * 具体做法：使用Hutool的HttpUtil构造请求体和头部，发送POST请求到大模型API端点。

    * 优点：语言无限制、不增加额外依赖、灵活度高。

    * 缺点：需要手动处理错误、序列化/反序列化复杂、代码冗长。

    * 适用场景：SDK不支持的语言、简单原型验证、临时性集成。

  

3. Spring AI框架

    * 实现方式：通过Spring AI的统一抽象接口调用不同的模型服务。

    * 具体做法：引入Spring AI依赖，配置模型参数，注入AIChatClient或ChatModel后端调用。

    * 优点：统一的抽象接口、易于切换模型提供商、与Spring生态完美融合、提供高级功能。

    * 缺点：增加抽象层、可能不支持特定模型的独特性、版本更新快。

    * 适用场景：Spring应用、需要支持多种模型、需要高级AI功能的场景。

  

4. LangChain4j框架

    * 实现方式：使用LangChain4j框架的组件结构构建AI调用链。

    * 具体做法：引入LangChain4j及相应组件的集成包，创建模型实例并调用。

    * 优点：提供完整的AI应用工具链、支持复杂工作流、丰富的组件和工具。

    * 缺点：学习曲线陡峭、文档相对较少、抽象层可能引入性能开销。

    * 适用场景：构建复杂AI应用、需要链式操作、RAG应用开发。

# 如何实现 AI 多轮对话功能？如何解决对话记忆持久化问题？ 

多轮对话功能的关键在于让 AI 具备 “记忆能力”，即能够记住与用户之前的对话内容并保持上下文连贯。在本项目中，我使用了 Spring AI 框架提供的对话记忆 (Chat Memory) 和 Advisor (顾问) 特性来实现这个功能。

为什么？
	快速原型开发、零外部依赖、部署简单。

具体实现操作上来说，我主要通过构造 `ChatClient` 来实现功能更丰富、更灵活的 AI 对话。`ChatClient` 支持使用 `Advisors`，可以理解为一系列可插拔的拦截器，在调用 AI 前后执行额外操作。其中`MessageChatMemoryAdvisor` 就是实现多轮对话的关键 `Advisor`，它的作用是从对话记忆中检索历史对话，并将对话历史作为消息集合添加到当前的提示词中，实现让 AI 模型能够 “记住” 之前的交流。

`MessageChatMemoryAdvisor` 依赖于 `ChatMemory` 接口的实现来存取对话历史。`ChatMemory` 接口中定义了保存消息、查询消息和清空历史的方法。

默认情况下会使用`InMemoryChatMemory`实现，从这个类名可以看出对话记忆仅存在于内存中，一旦服务重启，记忆就会丢失。为了解决这个问题，需要将对话记忆持久化。

Spring AI 提供了多种持久化方案，例如 `JdbcChatMemory` 可以将对话保存在关系型数据库中。但是在本项目中，考虑到 `spring-ai-starter-model-chat-memory-jdbc` 依赖版本较少且缺乏相关介绍，我选择了自定义实现 `ChatMemory` 接口的方式：

1. 开发一个 `FileBasedChatMemory` 类，它实现了 `ChatMemory` 接口。
2. 使用高性能的 Kryo 序列化库将对话消息（`Message` 对象及其子类）序列化后保存到本地文件中，读取时再进行反序列化。选择 Kryo 是因为 `Message` 接口有多种实现，结构不一，且没有无参构造和 `Serializable` 接口，普通的 JSON 序列化难以处理。

通过这种方式，我将对话的上下文信息持久化到了指定的文件目录，解决了内存记忆丢失的问题。

在本项目中，对话记忆持久化是通过自定义一个名为 `FileBasedChatMemory` 的类来实现的。

`FileBasedChatMemory` 的核心实现思路是将每个对话的聊天记录序列化后存储在单独的文件中，文件名与对话 ID 相关联。它主要依赖 Kryo 序列化库来处理 `Message` 对象的存取。

几个实现关键：

1）初始化与目录管理：`FileBasedChatMemory` 在构造时会接收一个文件目录路径 (`BASE_DIR`) 作为参数，所有对话记录文件都将存储在该目录下。如果指定的目录不存在，它会自动创建该目录。

2）Kryo序列化配置：因为 Spring AI 中的 `Message` 是一个接口，有多种不同的子类实现（如 `UserMessage`、`SystemMessage`），这些子类结构不一致，并且没有无参构造函数或未实现 `Serializable` 接口，导致直接使用标准 Java 序列化或普通的 JSON 序列化会出现报错。

3）核心方法实现（本质上是增删改查）：

- `add(String conversationId, List<Message> messages)`： 当需要添加新的消息到某个对话时，这个方法首先会根据 `conversationId` 获取或创建一个该对话的消息列表 (`getOrCreateConversation`)。然后将新消息追加到列表中，通过 `saveConversation` 方法将整个更新后的消息列表序列化回文件。
- `get(String conversationId, int lastN)`： 根据 `conversationId` 读取对应的对话文件，反序列化出完整的消息列表 (`getOrCreateConversation`)。然后，根据 `lastN` 参数返回最近的 N 条消息，如果消息总数少于 `lastN`，则返回所有消息。
- `clear(String conversationId)`： 根据 `conversationId` 找到对应的对话文件并将该文件删除。

# 什么是结构化输出？Spring AI 是怎么实现结构化输出的？ 

结构化输出是指将大语言模型返回的自由文本输出转换为预定义的数据格式，像 JSON、XML 或特定的 Java 类（POJO），对于需要可靠解析 AI 输出值并进行后续处理的需求来说非常重要。

Spring AI 通过 `StructuredOutputConverter` 机制实现结构化输出，其工作流程可以分为调用前和调用后两个阶段：

1. 调用前：`StructuredOutputConverter` 实现了 `FormatProvider` 接口。`FormatProvider` 的作用是提供特定的格式指令给 AI 模型，这些指令会附加到用户的提示词后面，明确告诉模型应该生成何种结构的输出。举个例子，它可能会包含类似 "Your response should be in JSON format. The data structure for the JSON should match this Java class: com.example.MyBean..." 这样的描述，并可能附带一个 JSON Schema 定义，引导模型生成符合指定格式的响应。
2. 调用后：`StructuredOutputConverter` 同时也实现了 Spring 的 `Converter<String, T>` 接口。这个 Converter 负责将大模型返回的文本输出（通常是 JSON 字符串）转换为开发者指定的目标类型 `T`（比如一个 Java Bean 对象、Map 或 List）。Spring AI 提供了多种内置的转换器实现，如 `BeanOutputConverter`（用于转换为 Java Bean，内部基于 ObjectMapper）、`MapOutputConverter` 和 `ListOutputConverter`。

也就是说，Spring AI 的结构化输出转换器首先通过修改提示词来规定模型按特定格式生成文本，然后将该文本转换为 Java 对象。使用 ChatClient 的 .entity(MyClass.class) 方法时，框架会自动处理这个过程，将模型的 JSON 输出映射到 MyClass 的实例。但是，这只是 “尽最大努力” 的转换，模型并不保证一定能严格按要求返回结构化数据！

# Re-Reading？

Re-Reading (重读)，也称为 Re2，是一种通过让大语言模型重新阅读问题来提高其推理能力的技术。核心思想是，对于复杂问题，重复阅读和审视问题有助于模型更好地理解题意和约束，从而生成更准确、更深入的回答，有文献研究证明这是有一定效果的。不过，这种方法会因为重复处理输入导致成本加倍，所以在面向 C 端开放的应用中需要谨慎使用。

在 Spring AI 中，可以通过自定义 Advisor 来实现 Re-Reading 功能：

1）创建自定义 Advisor 类：该类需要同时实现 `CallAroundAdvisor`（用于同步请求）和 `StreamAroundAdvisor`（用于流式请求）接口，让该类更通用 (在 Spring AI 1.0 版本中，上述两个接口需要更改为 `CallAdvisor` 和 `StreamAdvisor`)

2）修改用户提示词：在 Advisor 的前置处理逻辑中（例如 `aroundCall` 或 `aroundStream` 方法调用之前），对用户的原始输入文本进行改写。 改写的格式通常是将原始输入重复一遍，并用明确的指令引导模型重新阅读，通过看源码能够看到提示词：

```md
{Input_Query}
Read the question again: {Input_Query}
```
其中，`{Input_Query}` 是用户原始的提问内容。

3）传递给模型：将改写后的提示词传递给大语言模型进行处理。

# 如何实现RAG

在AI超级智能体项目中，我主要通过两种方式来构建和实现 RAG 知识库功能，这两种方式都基于 Spring AI 框架：

1）方案 1：Spring AI + 本地知识库

这种方式更侧重于 **通过编程** 完整地实现 RAG 的核心流程。
- 文档准备与读取：我首先准备了 Markdown 格式的知识文档（如《恋爱常见问题和回答 - 单身篇.md》等），并将它们存放在项目的类路径下。然后，利用 Spring AI 提供的 `MarkdownDocumentReader` 来读取这些本地文件。通过 `MarkdownDocumentReaderConfig`，我配置了文档的读取方式，并为每个文档添加了如文件名之类的元数据，以便后续进行更精确的检索。
- 向量转换和存储：读取到的文档列表会通过一个 `EmbeddingModel`（本项目中使用了 `dashscopeEmbeddingModel`）转换为向量。为了简化开发和学习，我在初期使用了 Spring AI 内置的、基于内存的 `SimpleVectorStore` 来存储这些文档向量。`SimpleVectorStore` 实现了 `VectorStore` 接口，该接口继承自 `DocumentWriter`，可以直接接收并处理文档列表的写入请求，内部完成向量化和存储过程。
- 查询增强和关联：当用户提问时，我通过 Spring AI 的 `QuestionAnswerAdvisor` 实现知识库的 “无感知” 调用。这个 Advisor 会查询 `SimpleVectorStore`，找出与用户问题相关的文档片段，并将这些检索到的内容作为上下文附加到用户的原始提问中，形成一个增强的 Prompt，最后发送给大语言模型生成回答。

2）方案 2：Spring AI + 云知识库服务 (以阿里云百炼为例)

这种方式则更多地依赖云平台提供的能力来简化 RAG 的构建。

- 云端知识库准备：我首先在阿里云百炼平台上传原始的知识文档。平台负责文档的解析、智能切分（将长文档分割成合适的片段）以及向量化，并将处理后的数据存储在向量数据库中。我们可以在百炼控制台手动管理这些知识库和文档切片。
- 程序对接与查询增强：Spring AI Alibaba 生态提供了与百炼平台集成的 `DashScopeDocumentRetriever`。我通过配置这个`DocumentRetriever`（主要提供 API Key 和百炼上知识库的名称），实现从云端的知识库中检索信息。为了将云端检索能力集成到 AI 对话流程中，我使用了 `RetrievalAugmentationAdvisor`，这是一个更灵活的Advisor，我可以通过它指定使用自定义的 `DocumentRetriever`。当用户提问时，这个 Advisor 会通过 `DashScopeDocumentRetriever` 从百炼知识库中获取相关内容，增强用户提问，然后交由大模型处理。

# 如何切分
目前主流的文档切分策略主要有以下几种：
1. **固定长度切分 (Fixed-size Chunking)**    
    - **原理：** 这是最简单粗暴的方法。直接按照一个固定的字符数（例如500个字符）来切割文档，为了避免一个完整的句子被从中间切断，通常会设置一个重叠区域（Overlap）。比如，前一个块是 `[0, 500]`，下一个块可以是 `[450, 950]`，中间有50个字符的重叠。        
    - **优点：** 实现简单，计算开销小，适用于任何类型的文本。        
    - **缺点：** 完全忽略了文本的语义结构。很容易将一个完整的语义单元（比如一个段落、一个函数定义）强行拆开，导致上下文信息丢失，影响检索效果。        
    - **面试官视角：** 如果你只答出这一种，我会认为你对这个领域的了解还比较初级。这通常是入门时会使用的方法。        
2. **递归/字符切分 (Recursive/Character Text Splitting)**    
    - **原理：** 这是固定长度切分的升级版，也是LangChain等框架默认的推荐方法。它会尝试按照一组预设的分隔符（例如：`\n\n` -> `\n` -> -> `''`）来进行递归切分。它会先尝试用段落分隔符`\n\n`来切，如果切出来的块仍然大于设定的`chunk_size`，它就会在这些大块上，继续用下一级的分隔符（比如换行符`\n`）来切，直到所有块都小于`chunk_size`。        
    - **优点：** 在一定程度上尊重了文档的自然结构（段落、句子），切分效果通常远好于固定长度切分。        
    - **缺点：** 效果依赖于分隔符的选择和文档本身的格式。对于格式混乱的文档，效果可能会退化成固定长度切分。        
    - **面试官视角：** 这是目前最常用、最通用的方法。能答出这一点，说明你具备了RAG应用开发的基本功。        
3. **语义/内容切分 (Semantic/Content-aware Chunking)**    
    - **原理：** 这是更高级的切分方法，它试图从内容的语义层面来理解和切分文档。主要有两种思路：        
        - **句子切分 (Sentence Splitting):** 使用NLP库（如NLTK, spaCy）将文本切分成独立的句子。因为句子通常是表达一个完整意思的最小单元。这种方法常用于构建Sentence-Window检索策略，即检索时找到最相关的句子，然后将这个句子前后的几个句子（形成一个窗口）一起作为上下文送给大模型。            
        - **语义相似度切分 (Semantic Similarity Chunking):** 将文档中的句子或小段落向量化，然后比较相邻句子/段落之间的语义相似度。当相似度出现一个明显的“断崖”时，就认为这里是一个新的语义主题的开始，应该在此处进行切分。这确保了每个切块内部的语义是高度一致的。            
    - **优点：** 切分质量最高，能够最大程度地保留语义的完整性和连贯性，非常适合问答场景。        
    - **缺点：** 计算复杂度高，需要额外的模型（如句向量模型）和计算资源，切分速度较慢。        
    - **面试官视角：** 如果你能深入讲解语义切分的原理和实现方式，会是一个非常大的加分项。这表明你不仅知道“怎么做”，还深入思考了“怎样做得更好”，具备了优化RAG系统性能的能力。        
4. **结构化切分 (Structured Chunking)**    
    - **原理：** 针对特定格式的文档，如Markdown, HTML, JSON, 或代码文件。这种方法会解析文档的结构化标记（如Markdown的`#`标题，HTML的`<div>`标签）来进行切分。例如，可以把每个一级标题`#`下的所有内容作为一个大块，或者把每个二级标题`##`下的内容作为一个块。        
    - **优点：** 对于格式规范的文档，效果极好，能完美保留文档的层级结构和逻辑。        
    - **缺点：** 适用范围有限，强依赖于文档的格式。        
    - **面试官视角：** 能提到这一点，说明你在实践中处理过不同类型的文档数据，考虑问题比较全面。

# RAG 流程


![[Pasted image 20250829120758.png]]RAG 文档处理流程的目的是把原始文档转化为 AI 可检索、可理解的格式，从而增强 AI 的回答质量。

这个流程主要包含以下四个核心步骤：

1）文档收集和切割：

收集 (Extract)：从各种数据源收集原始文档资料。比如本地文件系统中的 PDF、Markdown 文档，或者网页、数据库等，。
预处理与切割 (Transform)：对收集到的原始文档进行清洗，去除无关信息（比如 HTML 标签、广告内容）并统一格式，然后将篇幅较长的文档分割成更小、更易于处理的文本块。
2）向量转换和存储：

向量转换：使用专门的 Embedding 模型把上一步得到的每个文本块转换为高维的数字向量。
向量存储（Load）：把生成的向量及对应的原始文本块内容和元数据一起存入向量数据库中。
前两个步骤经历了完整的 ETL 流程，也就是对文档进行抽取、转换和加载。

3）文档过滤和检索：

查询处理：当用户提出一个问题（查询）时，这个查询文本也会被 同一个 Embedding 模型 转换为向量形式。

	Embedding 模型选了 BGE-M3 因为它同时支持中英文且效果好
	
相似度搜索：系统使用用户查询的向量，在向量数据库中搜索与之最相似的文档向量。一般会检索出 Top-K 个最相似的结果。

过滤机制：在检索过程中，可以根据文档的元数据进行过滤，例如只检索特定来源或特定时间范围内的文档，提高检索结果的精确性。
4）查询增强和关联：

上下文组装：把上一步检索到的相关文档块提取出来，跟用户的原始查询组合在一起，形成一个增强的提示。
内容生成：这个增强的提示会被发送给大模型。大模型利用这些检索到的、作为额外上下文的知识来生成回答。因为有了相关的、可能更新的外部知识，生成的回答通常比仅依赖其内部预训练知识更为准确和具体。
源引用与后处理：生成的回答可以附带信息来源的引用，即说明哪些信息来自哪些检索到的文档。还可以进行一些后处理，如格式化输出、内容摘要等，优化最终呈现给用户的结果。

# ETL？

![[Pasted image 20250829120914.png]]

ETL（Extract, Transform, Load）数据处理是一种标准的数据集成过程，指的是从各种数据源中抽取（Extract）数据，对数据进行清洗、转换、重构等转换（Transform）操作，最后将处理后的数据加载（Load）到目标数据仓库中。这个概念在大数据领域非常流行。

在构建 RAG 知识库时，ETL 负责数据准备和预处理的整个生命周期，保证输入到向量数据库的知识是高质量且易于检索的。

Spring AI 为 ETL 流程提供了 **模块化** 的组件支持，开发者可以方便地构建数据处理管道，我的实现步骤大致如下：

1. 抽取（Extract）：使用 `DocumentReader` 从数据源加载原始文档。我准备了 Markdown 格式的恋爱知识文档，然后引入了 `spring-ai-markdown-document-reader` 依赖，并创建了 `LoveAppDocumentLoader` 类加载读取文档。
2. 转换（Transform）：使用 `DocumentTransformer` 对加载的文档进行处理，我使用了两种方式进行了文档的拆分，包括 `MarkdownDocumentReader`自动拆分 + `TokenTextSplitter` 拆分，还使用了 `KeywordMetadataEnricher` 自动为文档补充关键词元信息。
3. 加载（Load）：使用 `DocumentWriter` 将转换后的文档写入存储库。在项目初期，我使用了 Spring AI 内置的、基于内存的 `SimpleVectorStore` 作为向量数据库。`SimpleVectorStore` 自身实现了 `VectorStore` 接口，而 `VectorStore` 接口又继承了 `DocumentWriter`，所以它可以直接接收 Document 列表进行存储。后来我还实践了把文档加载到基于 PostgreSQL 的 PGVector 向量数据库中。

# 如何实现 RAG 向量存储

在本项目中，我选择使用 PGVector 作为 RAG 知识库的向量存储方案。PGVector 是主流开源关系型数据库 PostgreSQL 的扩展，让 PostgreSQL 数据库能够存储高维向量数据，并执行高效的相似性搜索。

具体实现步骤：

首先需要一个 PostgreSQL 数据库实例，在该实例上安装并启用 PGVector 扩展，这里我使用了阿里云数据库，而不是自己在本地安装，节省了一些时间，而且能使用平台提供的监控功能。
在 Spring Boot 的配置文件中配置数据库连接信息。
通过 Spring AI 集成，在配置类中，通过 PgVectorStore.builder() 方法创建 VectorStore 实例。这个Builder 需要传入一个 JdbcTemplate 和一个明确指定的 EmbeddingModel。
配置完成后，这个自定义的 pgVectorVectorStore Bean 就可以像其他 VectorStore 实现一样被注入到服务中，用来添加文档和执行相似性搜索。
我选择 PGVector 作为向量存储方案主要基于以下几点考虑：

整合现有基础设施：很多企业和开发者已经在使用 PostgreSQL 作为其主要的关系型数据库。通过 PGVector 扩展，可以直接在现有的 PostgreSQL 数据库上增加向量存储和检索能力，不需要引入和维护一套全新的、独立的向量数据库系统。大大降低了架构复杂性、运维成本和学习曲线。
数据一致性和事务支持：将向量数据与业务数据存储在同一个数据库中，可以更容易地保证数据的一致性，并能利用 PostgreSQL 成熟的事务处理、备份恢复等特性。
成熟的生态系统：PostgreSQL 拥有非常成熟和庞大的生态系统，包括各种工具、驱动程序和社区支持。
Spring AI 支持：Spring AI 对 PGVector 提供了良好的集成支持，在 Java 和 Spring Boot 应用中接入和使用 PGVector 成本非常低。


# Tool Calling

工具调用 (Tool Calling)，也常被称为 Function Calling，是一种允许 AI 大模型在对话过程中，根据需要请求执行外部工具来完成特定任务的机制。

AI 模型本身可能不具备实时查询天气、操作数据库或访问特定 API 的能力，通过工具调用，它可以识别出什么时候需要这些外部能力，并生成一个包含工具名称和所需参数的请求。 应用程序接收到这个请求后，实际执行相应的工具，并将执行结果返回给 AI 模型，模型再基于这个结果继续对话或生成最终答案。

注意，真正执行工具的是我们的应用程序，而非 AI 服务器本身。

Spring AI 极大地简化了工具调用的实现：
1）定义工具：通常使用注解方式，在一个 Java 类中，将希望作为工具的方法标记上 @Tool 注解。方法的参数可以使用 @ToolParam 注解来提供描述和指定是否必需。Spring AI 支持多种 Java 类型作为参数和返回值，但返回值需要可序列化。

2）注册与使用工具：

按需使用：在构建 ChatClient 请求时，通过 .tools() 方法直接传入工具类的实例。
全局使用：在构建 ChatClient 时，通过 .defaultTools() 注册默认工具，这些工具对该 ChatClient 发起的所有对话都可用。
集中注册：可以创建一个配置类，使用 @Bean 方法将所有工具实例化并通过 ToolCallbacks.from() 统一注册为一个 ToolCallback[] 数组，方便管理和注入。
3）调用流程：当使用配置了工具的 ChatClient 进行对话时，如果 AI 模型判断需要使用某个工具，Spring AI 框架会自动执行以下步骤，无需开发者关心：

解析 AI 模型返回的工具调用请求（包含工具名和参数）。
根据工具名找到对应的 Java 方法并执行。
将工具执行的结果转换并返回给 AI 模型。
AI 模型根据工具结果生成最终回复。

文件操作工具 (FileOperationTool)：提供读取文件内容和将内容写入文件的能力。 为确保安全，所有文件操作都限定在项目根目录下的 /tmp/file 隔离目录中。

# 如何实现PDF生成

在本项目中，我实现 PDF 生成工具 (PDFGenerationTool) 的目的是根据 AI 提供的文件名和文本内容，创建一个 PDF 文档并保存到服务器本地。具体实现步骤如下：

1）选择库：我选用了流行的 Java PDF 处理库 iText 来实现 PDF 的生成。

2）添加依赖：在项目的 pom.xml 文件中添加 iText 相关的依赖，如 itext-core 和用于中文字体支持的 font-asian。

3）PDFGenerationTool类包含一个使用 @Tool 注解标记的 generatePDF 方法。此方法接收两个参数：fileName (希望保存的 PDF 文件名) 和 content (要写入 PDF 的文本内容)。

方法的具体内容包含以下部分：

文件路径与目录创建：PDF 文件会保存在一个统一的隔离目录。在写入前，会使用Hutool的 FileUtil.mkdir 保证该目录存在。
PDF 文档创建： 使用 PdfWriter 指定输出文件的路径，创建一个 PdfDocument 对象，关联 PdfWriter。然后创建一个 Document 对象，关联 PdfDocument，这是我们向 PDF 中添加内容的主要入口。
中文字体处理：为了正确显示中文内容，需要配置中文字体。可以直接使用 iText 内置的中文字体。 如果是生产级应用，可以下载并配置所需的字体文件。设置好的字体通过 document.setFont(font) 应用到文档。
内容添加：将传入的 content 文本包装在一个 Paragraph（段落）对象中，然后通过 document.add(paragraph) 方法将其添加到 PDF 文档中。
资源关闭：使用 try-with-resources 语句保证 PdfWriter、PdfDocument 和 Document 对象在使用完毕后能被正确关闭。
返回结果：方法成功执行后，返回一个表示 PDF 已成功生成并保存到指定路径的字符串消息。如果发生异常，则返回对应的异常信息。
通过这个工具，AI 可以将整理好的信息（如约会计划、分析报告）输出为 PDF 格式，方便用户查看和分享。

# MCP 

MCP（Model Context Protocol）模型上下文协议是一种开放标准，目的是标准化 AI 大模型与外部工具、资源和服务之间的交互方式。

MCP 采用客户端-服务器架构，一个客户端可以连接多个 MCP 服务器。

利用 Spring AI 可以轻松实现 MCP 客户端和服务端：

1）MCP 客户端实现

引入依赖：添加 spring-ai-mcp-client-spring-boot-starter 依赖。
配置连接：在 application.yml 中配置 MCP 客户端属性。
使用服务：Spring AI 会自动注入一个 ToolCallbackProvider Bean。通过这个 Provider 可以获取到所有已连接 MCP 服务提供的工具（ToolCallback对象），然后将这些工具绑定给 ChatClient，AI 就可以像调用普通工具一样使用 MCP 服务提供的功能了。
2）MCP 服务端实现

引入依赖：根据传输方式选择依赖，如 spring-ai-mcp-server-webmvc-spring-boot-starter。
配置服务：在 application.yml 中配置 MCP 服务端的基本信息（名称、版本）和传输方式（启用 stdio 或配置 SSE 端点路径）。
开发工具：在 Spring @Service 类中，使用 @Tool 和 @ToolParam 注解标记Java方法，将其定义为 MCP 服务对外提供的工具。
注册工具：在 Spring Boot 主应用类中，通过定义一个 ToolCallbackProvider 类型的 @Bean

在 AI 超级智能体项目中，我集成了地图 MCP 服务（以高德地图为例），让 AI 具备了根据位置信息推荐约会地点的能力。

# ReAct

ReAct，即 Reasoning + Acting（推理与行动），是一种结合推理和行动的智能体架构。它模仿人类解决问题时“思考 - 行动 - 观察”的循环。AI 首先对问题进行推理（Reason），将原始问题拆分为多步骤任务，明确当前要执行的步骤。然后，它会调用外部工具执行行动（Act），比如调用搜索引擎或访问网页。最后，它会观察（Observe）工具返回的结果，并将这些结果反馈给智能体，用于下一步的决策。这个过程会不断循环迭代，直到任务完成或达到预设的终止条件。

基于 ReAct 模式构建具备自主规划能力的 AI 智能体，核心在于实现这个 “思考-行动-观察” 的循环。这意味着智能体要实现：

1. 接收用户指令后，能分析并拆解成可执行的子任务。
2. 根据子任务的需要，从可用的工具集中选择合适的工具并执行。
3. 分析工具执行的结果，判断任务进展，决定下一步是继续调用工具、向用户澄清还是结束任务。

在本项目中，我实现的拥有自主规划能力的超级智能体就是基于 ReAct 模式，通过 ReActAgent 定义先 think 思考再 act 行动的流程，然后通过 ToolCallAgent 具体实现这两个方法。在 `think()` 方法中决定使用哪个工具（推理），在 `act()` 方法中执行工具（行动），并将执行结果作为下一步思考的输入（观察）。

# 你是怎么实现分层智能体架构的？ 

在本项目中，我参考了 OpenManus 的思想，实现了一个简化的分层智能体架构，主要包含以下几个层次：

1. BaseAgent：这是智能体的基类，定义了智能体的基本属性，比如名称、系统提示词、最大执行步骤、当前步骤、状态、ChatClient以及一个 `messageList` 用于维护对话上下文。还包含了一个核心的 `run` 方法，负责整个多步骤执行流程的控制，以及一个抽象的 `step` 方法，具体的单步执行逻辑交由子类实现。这种设计也是模板方法设计模式的体现。
2. ReActAgent：这个类继承自 `BaseAgent`，进一步将 `step` 方法的实现分解为两个核心的抽象方法：`think()`（思考）和 `act()`（行动）。它实现了 ReAct 模式的核心逻辑：在每一步中先思考（`think`），然后根据思考结果决定是否行动（`act`）。
3. ToolCallAgent：继承自 `ReActAgent`，作用是实现工具调用的能力。在 `think()` 方法中，它会根据当前的对话上下文和可用的工具列表，调用大模型决定需要使用哪些工具以及相应的参数，也就是“推理”阶段。在 `act()` 方法中，实际执行大模型在 `think` 阶段选定的工具。这里我利用了 Spring AI 提供的 `ToolCallingManager` 来执行工具调用、获取工具的执行结果。这个过程是“行动”和部分“观察”阶段。工具的执行结果会更新到 `messageList` 中，作为下一轮 `think` 的输入。而且，我在 `ToolCallAgent` 中通过设置 `DashScopeChatOptions` 的 `withProxyToolCalls(true)` 禁止 Spring AI 自动托管完整的工具调用流程，方便自行控制和观察 ReAct 的每一步。
4. YuManus：这是我最终实现的、可以直接使用的 AI 超级智能体实例，它继承自 `ToolCallAgent`。在这个类中，我为智能体设置了具体的名称、系统提示词、下一步提示词、最大执行步数，并注入了所有可用的工具和 `ChatModel`。

基于这个分层架构，我可以很方便地扩展出各种各样不同领域的智能体，除了通用智能体 YuManus 之外，我只需修改 Prompt，就能定义出软件工程师智能体、数据分析师智能体等等。

# 什么是 SSE？为什么 AI 应用接口选择 SSE 而不是 WebSocket？ 

SSE 服务器发送事件（Server-Sent Events）是一种用于从服务器到客户端的 **单向** 数据传输技术，基于 HTTP 协议实现。

连接一旦建立，服务器就可以持续地将新的信息流式传输给客户端，不需要客户端重复发起请求，非常适合需要服务器主动更新数据的场景，比如我们 AI 应用中逐步显示大模型生成的回复，实现类似打字机的效果。

不选择 WebSocket 的原因：

1. 实现成本：SSE 基于标准的 HTTP 协议，实现相对简单，前端使用 JavaScript 的 `EventSource` API 即可轻松接收。WebSocket 则是一个独立的、更复杂的双向通信协议，需要专门的连接管理和握手过程。
2. 通信模式：AI 应用的流式响应通常是服务器向客户端的单向数据推送，跟 SSE 的单向通信模型完美契合。WebSocket 提供的是全双工双向通信，对于 AI 流式输出这类场景，双向能力并不必要，反而增加了复杂性。
3. 内置机制：SSE 规范包含自动重连和事件 ID（`lastEventId`）机制。如果网络连接意外中断，浏览器可以自动尝试重连，还可以通过上次接收到的事件 ID 告诉服务器从什么地方继续，对于保证数据流的连续性很有用。这些在 WebSocket 需要开发者自行实现。
4. 资源消耗：一般来说，SSE 比 WebSocket 更轻量，WebSocket 需要建立和维护一个持久的 TCP 连接。

WebSocket 在需要真正实时双向交互（比如在线聊天室、协同编辑工具）的场景下是更优的选择，但对于 AI 应用的场景，SSE 更加简单、高效，比 WebSocket 更合适。

2025年5月9日，MCP（Model Context Protocol）迎来重磅升级——<mark class="hltr-orange">Streamable HTTP正式发布</mark>，取代了HTTP SSE, 成为AI模型通信的新标准！

`Streamable HTTP`可以看作是`SSE`的现代化演进，特别是在复杂的AI应用场景下，它解决了`SSE`的一些固有局TA限，提供了更灵活、更强大也更高效的通信方式。
`Streamable HTTP`并非一个全新的独立协议，而是一种更灵活地运用标准HTTP方法（主要是`POST`和`GET`）来实现双向、流式通信的**架构模式**。特别是在一个名为“模型上下文协议”（Model Context Protocol, MCP）的框架下，`Streamable HTTP`被提出并用以替代原有的`HTTP+SSE`模式。
其核心思想是：
- **统一端点**：客户端通过一个单一的HTTP `POST`请求与服务器建立通信。    
- **灵活响应**：服务器可以根据请求的性质，选择两种响应方式：    
    1. **同步JSON响应**：对于简单的、可以快速完成的请求，服务器直接返回一个标准的JSON对象，完成一次性的请求-响应。        
    2. **流式SSE响应**：对于需要长时间处理或需要持续更新的请求（比如AI模型生成内容），服务器会以`text/event-stream`的形式开始一个流式响应，本质上是在`POST`请求的响应体中启动一个SSE流。
- **双向通信**：虽然数据流的核心仍然是服务器到客户端的推送，但整个交互被封装在一次`POST`请求中，使得通信模型更加统一和直观。客户端发送指令，服务器以流或单个消息的形式响应。

1. 问：**Streamable HTTP**如何解决**SSE**不支持断线重连的问题？ 答：**Streamable HTTP**在每次通信时会记录id编号对应请求与响应，将这里请求与响应存储可断线重连进行恢复。
2. 问：**Streamable HTTP**如何解决**SSE**服务器需要维持长连接的问题？答：在需要发送响应过程中会保持连接，但一旦流式响应结束，服务器随后便会关闭流。
3. 问：**Streamable HTTP**如何解决**SSE**服务器消息只能通过SSE传输的问题？答：**Streamable HTTP**服务器可灵活选择是返回普通HTTP响应还是升级为SSE流，对于简单请求直接使用普通HTTP响应，对于内容复杂等需要流式传输的请求场景自动升级为SSE。
4. 问：**Streamable HTTP**如何解决**SSE**服务器基础设施兼容性限制？答：**Streamable HTTP**各基础设施的兼容性很完备。

# 遇到的挑战-AI超级智能体

如何确保智能体在执行复杂任务时，既能有效地分解和执行步骤，又能避免一些固有问题（陷入不必要的循环或者过度消耗资源）。

这类高级智能体优势在于能够根据用户需求自主推理和行动，直到完成目标。但这种自主性也带来了很多复杂性。

在实际测试中我观察到，尽管它能完成像“查找附近约会地点、结合网络图片制定详细约会计划并输出 PDF”这样的复杂任务，但执行步骤和调用工具的次数经常不是最优的。会因为 AI 的随机性或对任务理解的偏差，导致执行路径过长，在某些情况下反复尝试无效操作，导致增加 Token 的消耗，甚至有陷入无限循环的风险。

针对这个问题，我的主要解决方案是引入了循环检测和处理机制。具体做法是参考了像 OpenManus 这样的开源智能体框架的实现思路。我在智能体的基础类（如`BaseAgent`）中记录了执行步骤和历史消息上下文。通过比较最近的消息内容或智能体内部状态，判断智能体是否可能陷入了重复执行相似操作的循环。例如，可以设置一个重复阈值（`duplicateThreshold`），当检测到连续多次相似的助手响应或工具调用时，就认为智能体可能卡住了。

一旦检测到潜在的循环，我会主动干预，比如改写智能体下一步的提示，比如明确指示它“观察到重复响应，请考虑新策略，避免重复已尝试过的无效路径”，引导智能体尝试新的解决路径。此外，我还设计了最大执行步骤数（`maxSteps`）的限制，当达到这个限制后，任务会被强制终止，有效防止智能体无休止地运行。通过这些机制，有效缓解了自主规划智能体带来的上述挑战。


# 智能客服

## 发现问题

在项目初期，Agent的性能非常不稳定。具体表现为两个极端：有时，对于一个明确的问题，它会返回一些风马牛不相及的信息，这是一个典型的精确率低的问题。

另一些时候，它又会过于保守，对于一个答案明确存在于我们产品手册中的问题，它却回答‘对不起，我找不到相关信息’，这又是**召回率低**的问题。

## 分析问题

问题的核心出在知识检索这一步，而检索质量又直接受制于上游的**文档分块（Chunking）** 策略。经过排查，我发现阿里云百炼默认的‘智能分块’虽然在大多数情况下表现良好，但对于我们那些包含复杂表格、多级列表和大量脚注的PDF格式技术手册，处理得并不完美。

**语义分割不当**：有时，一个完整的技术规格表或者一个跨越两段的政策条款，被错误地从中间切开，导致任何一个单独的块都不再包含完整信息，从而在检索时无法被完整召回。

## 解决方法

我意识到不可能对所有文档都进行手动干预，所以我采取了‘抓大放小’的策略。我识别出那些最常被查询、且结构最复杂的关键文档（比如核心产品的用户手册和售后政策），然后利用百炼控制台提供的分块编辑功能，对这些文档的切分结果进行了**手动校对和精细调整**。

在优化了数据源头之后，我开始调整检索算法本身。我做了一系列的A/B测试来寻找最佳参数组合。最终，我将**召回的文档块数量（K值）从默认的3个略微增加到5个**，目的是为了扩大搜索网，提高找到相关信息的概率

那些极其刁钻、语义模糊的查询，我发现仅靠向量相似度检索有时仍然不够精确。因此，我进一步探索并从概念上验证了一个**重排（Re-ranking）** 阶段。具体做法是，在向量检索初步召回Top-5的文档块后，并不直接将它们全部交给LLM，而是先通过一个更轻量级的、计算成本更低的交叉编码器（Cross-encoder）模型，对这5个候选块与用户查询的真实相关性进行一次更精细的打分和排序。
这个额外的步骤，虽然增加了一点点延迟，但极大地提升了最终送入LLM的上下文质量，是解决复杂查询场景的‘杀手锏’。

# 使用RAG做电商，商品ID通常由数字和字母组成，重复率极高，因此向量相似度在召回效果上可能表现不佳，你是怎么做的？

纯粹的向量搜索在处理‘蓝色、适合跑步的鞋子’这类语义查询时表现出色，但对于‘查询SKU12345的库存’这类包含精确标识符的请求，效果很差。

我们的解决方案是采用**‘元数据过滤 + 向量搜索’的混合搜索策略**，并结合**工具调用（Tool Calling）**进行智能路由。

具体来说，我们是这样做的：

### 1. 数据索引阶段：将ID作为元数据（Metadata）存储

在构建知识库时，我们不会把商品ID（SKU）这样的标识符直接混在长文本描述里去生成向量。这样做会污染向量空间，因为`SKU123`和`SKU124`在语义上没有任何关联。

元数据：简单来说，元数据（Metadata）就是“描述数据的数据”（Data about data）。
它本身不是信息的核心内容，而是用来描述、解释、定位或管理核心内容的附加信息。
图书馆的书（最经典的例子）
核心数据： 书本里的文字、故事、知识内容。

元数据： 贴在书上的索引卡片
元数据（索引卡）不告诉你书里讲了什么故事，但它能让你快速地找到这本书，并了解它的基本属性。


我们的做法是：

- **向量化内容（Content for Vectorization）**: 只将商品的描述、名称、用户评论等**富含语义**的文本内容进行向量化。
    
- **结构化元数据（Structured Metadata）**: 将`product_id`、`sku`、`category_id`、`brand`、`price`、`stock_status`这些**精确匹配字段**，作为元数据与生成的向量一同存储在向量数据库中。
    

现在，向量数据库里的每条记录都包含两部分：一个用于相似度搜索的`vector`，和一组用于精确过滤的`metadata`。

### 2. 查询处理阶段：两步走的混合搜索策略

当用户查询时，我们并不是直接把整个问题扔去做向量搜索，而是分为两步：

**第一步：查询意图分析与实体提取**

我们会先用一个轻量级的模型（甚至可以用正则表达式）来分析用户输入，判断其中是否包含商品ID、订单号这类明确的实体。

**第二步：执行混合搜索**

- **如果查询中包含商品ID (例如："介绍一下商品ID-XYZ-001，它防水吗？")**
    
    - 我们执行 **“先过滤，后搜索” (Filter-then-Search)** 的策略。
        
    - **过滤 (Filter)**: 首先，我们向向量数据库发起一个**元数据过滤请求**，指令是：`WHERE product_id = 'ID-XYZ-001'`。这将搜索空间瞬间从数百万商品缩小到仅此一个商品。
        
    - **搜索 (Search)**: 然后，我们再对查询中剩余的语义部分（“它防水吗？”）进行向量搜索，但**只在刚才过滤出的那一个商品的相关文档中进行搜索**。这样既保证了ID的精确匹配，又利用了向量来理解用户的具体问题。
        
- **如果查询中不含商品ID (例如："我想要一双舒适的跑鞋")**
    
    - 这种纯语义查询，我们就直接执行**向量搜索**，在整个商品库中找出最相关的Top-K个商品，然后将它们的详细信息喂给大模型进行总结回答。
        

### 3. 进阶方案：基于工具调用的智能路由

对于更复杂的查询，比如“帮我查一下ID-XYZ-001和ID-ABC-002哪个更便宜？”，简单的RAG就吃力了。

这时，我们项目中的**工具调用（Tool Calling）**就派上了用场：

1. **定义工具**: 我们会定义两个工具：
    
    - `product_semantic_search`: 用于处理模糊的语义查询（内部使用混合搜索RAG）。
        
    - `get_product_details_by_id`: 一个直接调用数据库或API的函数，通过ID精确查询商品详情（如价格、库存）。
        
2. **智能路由**: 大模型（Agent）会首先分析用户的查询。
    
    - 当看到“哪个更便宜”和两个明确的ID时，它会判断出需要精确的价格信息。
        
    - 它会选择**不调用**RAG工具，而是**连续两次调用**`get_product_details_by_id`工具，分别传入`ID-XYZ-001`和`ID-ABC-002`。
        
    - 拿到两个工具返回的精确价格后，它再进行比较，并生成最终的自然语言回答。
        

**总结一下：** 面对商品ID这类非语义化、高区分度的字段，我们绝不依赖单一的向量搜索。而是通过**将ID存为元数据，执行‘先过滤、后搜索’的混合查询**来保证召回的精确性。对于更复杂的结构化查询，则升级为**基于工具调用的Agent架构**，让模型智能选择调用传统的数据库API，而不是RAG流程。这套方案有效地结合了传统数据检索的精确性和大模型语义理解的强大能力。”

# 如何理解Agent

**大模型 Agent **是基于大型语言模型并结合**模块化规划** 、**记忆**和**工具调用**的自主决策系统，它能够根据最终目标把复杂任务拆分成子任务，调用 API、检索数据库或使用插件，再通过内部循环不断优化执行流程，基本不需要人在每一步都监督。

主要区别如下：

1）**目标导向 vs. 被动响应**：传统大模型通常根据用户输入生成文本，缺乏主动性；而 Agent 以明确目标为驱动，能够主动规划并执行任务。

2）**记忆与状态管理**：Agent 具备短期和长期记忆能力，能够维护状态信息并根据历史经验调整行为；而传统大模型通常依赖于上下文窗口进行信息处理。

3）**多任务协同能力**：Agent 能够处理复杂的多步骤任务，协调多个子任务的执行；而传统大模型通常处理单一任务，缺乏任务协同能力。

4）**推理与环境适应能力**：和传统 AI 系统主要依赖预先设定的规则引擎或静态模型不同，大模型 Agent 具备多步推理能力和动态环境适应能力，它能在执行过程中不断评估结果、调整策略，真正实现端到端的目标导向执行。



# A2A

A2A 协议的核心作用是让“客户端代理”和“远程代理”之间顺利沟通。客户端代理负责创建并下发任务，而远程代理则根据这些任务提供信息或执行操作。整个过程中，A2A 提供了几个关键功能：

**能力发现**：每个代理都有一张“代理卡”，以 JSON 格式描述它能做什么。客户端代理可以通过这些卡片，找到最适合执行当前任务的远程代理，并通过 A2A 协议与之建立通信。

**任务管理**：客户端与远程代理的交互围绕任务展开。每个任务都有一个由协议定义的“任务对象”，并且具有生命周期。有些任务可以立刻完成，而对于运行时间较长的任务，代理之间可以持续沟通，保持状态同步，确保任务按预期推进。任务完成后会产生一个“工件”，也就是最终的执行结果。

**协作能力**：代理之间可以直接通信，发送包含上下文、回复、工件或用户指令的消息，便于协同完成更复杂的任务。

**用户体验协商**：每条消息可以包含多个“部分”，每部分代表一个完整的内容片段，比如生成的图像。每个部分都有明确的内容类型，客户端和远程代理可以据此协商最合适的展示格式，同时也可以决定是否支持如 iframe、视频、网页表单等用户界面功能，从而根据用户需求和设备能力，提供更好的使用体验。

# AI 应用的测试和效果评估？
AI 应用的测试和传统软件测试有相似之处，当然也有其独特性。

相似之处在于，我们仍然需要关注功能性、兼容性、安全性等方面。但其独特性在于，AI 系统的核心是模型和数据，因此测试的重点要向这两个方面倾斜。比如：

- 数据质量：模型是数据驱动的，输入数据的质量、分布以及是否能覆盖各种真实场景，对模型的表现影响很大。我们要测试模型在不同数据集、边缘案例以及对抗性样本下的表现。
- 模型性能：这涉及到模型的准确率、召回率、F1-score 等一系列评估指标。同时，我们也要关注模型在不同环境、不同输入扰动下的稳定性，即鲁棒性。例如，应用是否会对微小的、人眼难以察觉的输入变化产生截然不同的输出。
- 部署：应用最终部署到生产环境上。测试也需要覆盖模型与系统的交互、API 的稳定性以及在真实部署环境下的性能表现，比如延迟、吞吐量等。

效果评估更侧重于衡量 AI 应用在实际业务场景中产生的价值和影响。通常是一个持续的过程，而不仅仅是一次性的测试。

在效果评估方面，我们需要注意以下几个点：

- 明确评估指标：这些指标需要紧密围绕业务目标。例如，一个推荐系统可能关注点击率、转化率、用户停留时长等；一个风控模型更关注误报率、漏报率以及由此带来的经济损失减少。
- A/B 测试：将新功能与现有版本进行对比测试，观察其在真实用户和真实环境中的表现差异，这是评估效果最直接有效的方法之一。
- 成本效益分析：除了技术指标，还要从商业角度评估 AI 应用的投入产出比，考虑其带来的实际经济效益或社会效益。

AI 应用的测试和效果评估是一个系统性的工程，不只是要关注应用本身的技术指标，还要关注我们上面提到的内容，以及最终在真实业务场景中产生的价值，这是一个迭代和持续优化的过程。

## A2A 协议的工作流程是怎样的？ 

**1）发现**：客户端向 `/.well-known/agent.json` 发起 HTTP GET 请求，获取远端智能体的 Agent Card，其中包含智能体的唯一标识、能力清单、回调 URL、认证方式等元数据，帮助客户端快速了解并筛选合适的智能体来执行任务。  
**2）启动**：客户端根据业务需求生成唯一的 Task ID，然后通过 JSON‑RPC 调用 `tasks/send`（用于一次性请求，同步返回最终 Task 对象）或 `tasks/sendSubscribe`（用于订阅式请求，服务端通过 Server‑Sent Events 推送状态更新）向目标智能体发送任务请求。  
**3）处理**：远端智能体接收请求后，将任务状态从 `submitted` 切换到 `working`，在内部执行模型推理或外部工具调用；对于订阅式任务，会持续推送 `TaskStatusUpdateEvent` 和可选的 `TaskArtifactUpdateEvent`，让客户端实时获取进度或中间成果。  
**4）交互**：当智能体在处理过程中需要额外输入时，会发出 `input-required` 状态更新，并携带一条 `Message` 请求；客户端收到后可使用相同 Task ID 通过 `tasks/send` 或 `tasks/sendSubscribe` 补充用户输入，保持会话的连续性和上下文一致。  
**5）完成**：任务进入终态（例如 `completed`、`failed` 或 `canceled`），客户端可以选择主动拉取最终的 Task 对象，也可以继续通过 SSE 或 Webhook 机制订阅 `TaskStatusUpdateEvent`，并获取以 JSON Artifact 形式封装的最终结果。

# 通常如何实现长短期记忆机制？ 

1. **短期记忆**：利用**上下文窗口**（如Claude的100k tokens）缓存最近的对话或输入，或通过**滑动窗口**（Buffer Window）保留固定轮次的历史。

2. **长期记忆**：结合**摘要压缩**（Summarization）、**向量数据库**（如RAG检索增强）持久化存储关键信息。

# SpringBoot3、JDK21 的应用

一、项目背景与痛点

在我们的AI聊天机器人服务中，当用户量突破5000QPS时，出现了两个严重影响用户体验的问题：
1. 并发瓶颈：传统线程池在处理大量流式请求时，线程资源被耗尽，导致响应延迟激增（从平均300ms上升到2s）
2. 日志混乱：日志系统输出的都是平面化文本，当出现请求超时时，无法快速定位具体调用链路。例如用户反馈"第3次对话卡顿"，需要手动关联10+条日志才能还原上下文。

二、技术选型依据

我们选择Spring Boot 3和JDK 21的两个核心特性来解决这些问题：

1. Spring Boot 3的Observability模块：通过Micrometer框架实现结构化日志，这与Spring Boot 3对Jakarta EE 9的全面支持密切相关
2. JDK 21的虚拟线程（Virtual Thread）：这是Project Loom的核心成果，专为解决高并发场景下的线程资源问题设计
## 日志
传统日志痛点：
```java
2025-09-04 14:20:00 [INFO] Model call started
2025-09-04 14:20:01 [INFO] Model returned 200 tokens
```
这种平面日志无法回答：
- 这是哪个用户的请求？
- 模型调用具体耗时多久？
- 是否关联了正确的trace ID？
**解决方案：** 在Spring Boot 3中，我们通过ObservationRegistry注册自定义观察器，将日志升级为结构化数据：
```java
// 核心逻辑伪代码
Observation observation = Observation.start("model.inference", registry);
observation.context().put("traceId", UUID.randomUUID().toString());
observation.context().put("user.id", userId);
observation.context().put("model.version", "v3.2");
```
最终输出的日志变为JSON格式：
```json
{
  "timestamp": "2025-09-04T14:20:00.123Z",
  "level": "INFO",
  "event": "model.inference",
  "traceId": "abc123",
  "userId": "user456",
  "modelVersion": "v3.2",
  "durationMs": 1280,
  "tokenCount": 200
}
```
实际收益：
	日志查询效率提升80%：通过traceId可直接关联整个请求链路
	故障定位时间从小时级缩短到分钟级：结合Grafana可快速筛选特定模型版本的异常日志
	支持自动生成调用拓扑图：通过日志分析自动生成服务依赖关系图

## 虚拟线程
**传统线程问题：** 在处理SSE流式响应时，每个请求需要保持一个线程持续2-5秒。当并发量超过线程池容量（默认200）时：
1. 新请求需要等待空闲线程
2. 等待队列满后直接触发熔断降级
3. 线程上下文切换导致CPU利用率飙升

**虚拟线程改造：** 我们将每个请求处理单元改造成虚拟线程
```java
// 伪代码示例
Thread.ofVirtual().name("ai-stream-", i).start(() -> {
    processStreamRequest(request, emitter);
});
```
