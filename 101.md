
# Spring

<mark style="background: #FFF3A3A6;">依赖注入框架，帮助管理Java对象，再集成一些其他内容</mark>
## **Spring是为了解决什么问题？**

1. 配置复杂性（XML地狱）：

   * 传统JavaEE开发（例如，使用EJB - Enterprise JavaBeans）需要大量的XML配置文件。你需要配置数据源、事务管理、安全等等，而且这些配置通常非常冗长和复杂。稍微改动一点逻辑，就可能需要修改好几处不相关的XML配置，开发效率低下，维护起来也特别困难。

2. 代码耦合度高（难以测试和维护）：

3. 技术选择和集成困难：

   * 问题：一个企业级应用通常需要集成多种技术，比如不同的数据库访问框架（JDBC、Hibernate、MyBatis）、不同的Web框架（Struts、JSF）、不同的消息队列等等。将这些不同的技术有机地整合在一起，并确保它们协同工作，是一项非常具有挑战性的任务。

4. 资源管理和生命周期管理复杂：

   * 像数据库连接、线程池这样的资源，在使用完毕后必须正确地释放，否则会导致资源泄露，甚至系统崩溃。对象的创建、初始化、销毁等生命周期管理，如果没有框架的帮助，都需要开发者手动控制，容易出错。

## 如何解决这些问题？

1. **控制反转（IoC - Inversion of Control）/ 依赖注入（DI - Dependency Injection）**：

   * 这是Spring最核心的机制。它不再让你的代码自己去创建或查找它所依赖的对象，而是由Spring容器（就像那个“管家”）来负责创建这些对象，并将它们“注入”到需要它们的地方。

2. **面向切面编程（AOP - Aspect-Oriented Programming）：**

   * AOP允许你将那些与核心业务逻辑不直接相关，但又普遍存在于多个模块中的功能（比如日志记录、事务管理、安全检查等）“横向”地抽取出来，独立地管理。
   * 它是对OOP（面向对象编程）的补充和完善。OOP的核心是封装，但有些功能是分散在各个类中的，比如日志记录、事务管理、安全校验等，这些被称为横切关注点 (Cross-cutting Concerns)。

* AOP的作用就是将这些分散的“横切关注点”从业务逻辑代码中剥离出来，模块化到一个可重用的单元中（即“切面”），从而降低代码耦合度，提高可维护性和可重用性。

   * **消除重复代码**：避免了在每个方法中都手动添加日志、事务等代码。

   * **提高模块化和可维护性**：业务逻辑更纯粹，次要功能与核心功能分离，使代码更清晰。

Spring AOP中有哪几种通知（Advice）类型？
- **`@Before` (前置通知):** 在目标方法执行之前执行。
    
- **`@AfterReturning` (后置通知/返回通知):** 在目标方法**正常执行完成并返回结果后**执行。
    
- **`@AfterThrowing` (异常通知):** 在目标方法抛出异常后执行。
    
- **`@After` (最终通知):** 无论目标方法是正常执行完成还是抛出异常，**都会执行**。类似于`finally`代码块。
    
- **`@Around` (环绕通知):** 最强大的通知类型。它集成了前四种通知的功能，可以**完全控制**目标方法的执行。你可以决定方法何时执行、是否执行，甚至修改参数和返回值。你项目中的`Re-Reading Advisor`就是典型的环绕通知。


3. **对各种技术的抽象和集成**：

   * Spring为很多第三方技术（如JDBC、ORM框架、JMS、Web服务等）提供了统一的抽象层，这意味着你可以用相似的方式来使用不同的技术，并且Spring提供了便捷的集成机制。

## 代码示例

**传统方式（没有DI）**：

```Java
// 1. 用户数据仓储接口
public interface UserRepository {
    void save(String user);
}

// 2. 用户数据仓储实现类 A (基于内存)
public class InMemoryUserRepository implements UserRepository {
    @Override
    public void save(String user) {
        System.out.println("Saving " + user + " to in-memory database.");
    }
}

// 3. 用户数据仓储实现类 B (基于数据库)
public class JdbcUserRepository implements UserRepository {
    @Override
    public void save(String user) {
        System.out.println("Saving " + user + " to JDBC database.");
    }
}

// 4. 用户服务类
public class UserService {
    private UserRepository userRepository;

    public UserService() {
        // 问题：这里UserService自己决定使用哪个UserRepository的实现
        // 如果想换成JdbcUserRepository，就必须修改UserService的构造函数
        this.userRepository = new InMemoryUserRepository();
    }

    public void registerUser(String username) {
        System.out.println("Registering user: " + username);
        userRepository.save(username);
    }
}

// 5. 调用
public class Application {
    public static void main(String[] args) {
        UserService userService = new UserService();
        userService.registerUser("Alice");
    }
}

```

# Spring MVC

<mark style="background: #FFF3A3A6;">web框架，提供接口访问、RESTful接口等能力</mark>

**Spring MVC 是 Spring 框架中专门用于构建 Web 应用程序（尤其是基于MVC设计模式的应用程序）的模块。** MVC，顾名思义，代表着 Model（模型）、View（视图）和 Controller（控制器）。

**它主要解决的问题是**：如何在复杂的Web应用中，清晰地分离用户界面、业务逻辑和数据处理，使得代码结构更清晰、更易于维护和扩展。

在一个Web应用中，用户通常会通过浏览器发出请求（比如访问一个网页、提交一个表单），然后服务器处理这个请求，最后向用户返回一个响应（比如展示一个网页、一个成功消息）。MVC模式就是将这个过程中的职责分开：

* **Model (模型)：数据和业务逻辑**

  * **感性认识**：它就像餐厅里的 **“食材”** 和 **“烹饪方法”** 。它代表了应用的数据（比如一个用户对象、一个产品列表）以及处理这些数据的业务规则和操作（比如用户注册、计算订单总价）。Model层是独立于用户界面存在的。

* **View (视图)：用户界面**

  * **感性认识**：它就像餐厅的 **“美味的菜品”** 和 **“漂亮的餐具”** 。负责把Model中的数据展示给用户，通常是HTML页面，也可能是JSON、XML等数据格式。View只负责展示，不处理业务逻辑。

* **Controller (控制器)：请求处理和协调**

  * **感性认识**：它就像餐厅里的 **“服务员”**。它接收用户的请求（点餐），理解用户的意图，然后去调用Model层来处理业务逻辑（告诉厨房做什么菜），最后确定返回给用户什么View（这道菜应该用哪个盘子，怎么摆盘）。

## **Spring MVC 如何工作？（Web请求处理流程）**

1. **客户发出请求（Client Request）**

   -  **感性认识**：你走进餐厅，对服务员说：**“你好，我要点一个宫保鸡丁！”**（用户在浏览器中输入`http://yourdomain.com/order/gongbaojiding`并回车）。

   -  **技术对应**：浏览器发送一个HTTP请求（GET, POST等）到服务器。

 2. `DispatcherServlet`**登场（中央调度器/前台总管）**

   * **感性认识**：你刚喊出要点菜，餐厅的 **“前台总管”**（`DispatcherServlet`）立刻听到了。他是Spring MVC的核心，所有的Web请求都会首先经过他。他不会直接处理点菜，但他知道接下来该找谁。

   * **技术对应**：这是Spring MVC的前端控制器。所有的请求都会先进入`DispatcherServlet`。它负责接收请求，并分发给正确的处理器。

3. `HandlerMapping`**帮忙找对人（菜单和分流指示牌）**

   * **感性认识**：前台总管（`DispatcherServlet`）接到你的“点菜”后，他会查看一本 **“菜单或分流指示牌”**（`HandlerMapping`），上面写着哪种菜（URL）应该由哪个具体的服务员（Controller方法）来处理。

   * **技术对应**：`DispatcherServlet`会查询`HandlerMapping`，根据请求的URL（以及HTTP方法等），找到负责处理这个请求的**Controller类**和它里面具体的**处理方法**。

4. `Controller`**处理请求（具体的服务员）**

   * **感性认识**：菜单指示这个菜由 **“小王服务员”**（`Controller`）负责。小王走到你桌边，微笑着说：“好的，宫保鸡丁！” 然后他会：

     * **理解你的需求**（解析请求参数，比如数量、要不要辣）。

     * **告诉后厨（调用Service层）**：“厨房，帮我做一份宫保鸡丁，要微辣。”

     * **从后厨拿到菜品信息**（通过Service层调用Model层获取数据，比如宫保鸡丁的库存、价格）。

     * **决定用什么盘子装菜**（决定返回给哪个View）。

   * **技术对应**：找到的Controller方法开始执行。它会：

     * 从请求中获取数据（例如URL参数、表单数据）。

     * 调用**Service层**（业务逻辑层，这里会用到Spring核心的IoC/DI来注入Service），让Service层处理具体的业务逻辑和数据操作（查询数据库、更新数据等）。

     * Service层返回数据给Controller。

     * Controller将数据放入一个**Model**对象中，并返回一个**逻辑视图名**（例如`"gongbaojidingDetail"`）。

   * **示例代码（Controller的一部分）**：

```Java
// 这是你的"小王服务员"类
@Controller // 告诉Spring，这是一个Controller
@RequestMapping("/order") // 这是一个处理订单的Controller
public class OrderController {

    // Spring的"大管家"（IoC）会把UserService注入进来
    // 小王服务员不需要关心UserService是怎么被创建的
    @Autowired
    private UserService userService;

    // 这是小王服务员处理“宫保鸡丁”点菜的方法
    @RequestMapping("/gongbaojiding") // 当请求是 /order/gongbaojiding 时，就执行这个方法
    public String getGongBaoJiDingDetail(Model model) {
        // 小王服务员去后厨（调用UserService）获取宫保鸡丁的详细信息
        Dish gongBaoJiDing = userService.getDishDetails("宫保鸡丁");

        // 把菜品信息（Model）放到Model对象里，准备给客人看
        model.addAttribute("dish", gongBaoJiDing);

        // 小王服务员决定用“宫保鸡丁详情页面”的模板来展示
        return "gongbaojidingDetail"; // 这是一个逻辑视图名
    }
}

```

5. `ViewResolver`**确定展示方式（确定用哪套漂亮的餐具）**

  * **感性认识**：小王服务员（Controller）说要用“宫保鸡丁详情页面”的模板来展示。前台总管（`DispatcherServlet`）接到这个指示后，他会找到 **“餐具总管”**（`ViewResolver`），告诉他：“给我一个‘宫保鸡丁详情页面’！”

  * 餐具总管一看，哦，“宫保鸡丁详情页面”对应的是用**中国风的青花瓷盘子**和**精美的雕花筷子**（代表JSP、Thymeleaf、FreeMarker等具体视图技术），他会把这些准备好。

  * **技术对应**：`DispatcherServlet`将Controller返回的逻辑视图名（`"gongbaojidingDetail"`）交给`ViewResolver`。`ViewResolver`会根据配置（比如前缀和后缀），解析出真正的视图资源路径，例如`/WEB-INF/views/gongbaojidingDetail.jsp`。

6.  `View`**渲染输出（上菜，展示给顾客）**

  * **感性认识**：餐具总管（`ViewResolver`）已经找到了合适的餐具模板。后厨（Controller通过Model传递过来的数据）已经把菜品做好了。现在，把**菜品**放到**餐具模板**上，**摆好盘**，然后由`DispatcherServlet`将其**端到你面前**。

  * **技术对应**：`View`（通常是一个JSP文件、Thymeleaf模板等）根据Controller传递过来的`Model`数据，将数据填充到模板中，生成最终的HTML内容。

7. **响应返回给客户（服务员收尾）**

  * **感性认识**：你看到了你点的宫保鸡丁，非常满意！

  * **技术对应**：生成的HTML内容作为HTTP响应返回给浏览器，浏览器解析并显示页面。

# SpringBoot

<mark class="hltr-yellow">**快速启动**/集成项目，不用自己管理Spring配置，不用写XML，不用自己整合各种框架，</mark>

Spring Boot 是 Spring 框架的“脚手架”和“快速开发框架”。 它在Spring框架的基础上，进一步简化了Spring应用的初始化、配置和部署，让你可以“开箱即用”，快速地构建独立的、生产级别的Spring应用程序。

### **解决了什么问题？**

1. **“搭脚手架”很麻烦（繁琐的初始配置）**：

即使是Spring MVC应用，你开始一个新项目时，也需要手动添加一大堆依赖（比如Web模块、数据库模块、JSON处理模块），写一大堆XML配置或者Java配置类（比如配置`DispatcherServlet`、`ViewResolver`、数据源等），才能让项目跑起来。

* **“打包部署”很繁琐（独立运行的挑战）**：

传统的Spring Web应用通常需要打包成WAR文件，然后部署到外部的Tomcat、JBoss等应用服务器上运行。这多了一个部署步骤，也增加了环境依赖。

* **“快速迭代”很困难（开发效率低）**

### **核心特点**

* <mark class="hltr-red">**约定优于配置 (Convention over Configuration)**</mark>：不再需要手动配置，Spring Boot会根据项目依赖和最佳实践提供默认配置。

* **开箱即用 (Out-of-the-box experience)**：极少甚至无需配置就能启动一个功能齐全的Spring应用。

* **独立运行 (Standalone applications)**：直接打包成JAR文件，包含内嵌Web服务器，可以直接运行，无需外部部署。

* **生产级别特性 (Production-ready features)**：提供健康检查、监控（Actuator）、外部化配置等功能，方便部署到生产环境。

# MyBatis
<mark class="hltr-yellow">
Java操作数据库的（持久化）框架，对JDBC的封装</mark>

MyBatis 是一个优秀的持久层框架（Persistence Framework）。

主要作用是：**将程序中的Java对象与数据库中的数据进行映射。** 简单来说，就是帮助你的Java应用方便、高效地与数据库进行交互，执行SQL语句，并将数据库查询结果映射成Java对象，或者将Java对象的数据保存到数据库中。

## 解决什么问题

在没有MyBatis（或者其他持久层框架，如Hibernate、JPA）之前，Java应用与数据库交互主要通过 **JDBC (Java Database Connectivity)** 技术。

1. **“底层JDBC操作的繁琐重复”**：每次数据库操作都需要手动编写大量的JDBC样板代码，包括建立连接、创建Statement、设置参数、执行SQL、处理ResultSet、关闭资源等。这就像每次与数据库交互，你都要**手动操作底层的“数据管道阀门”和“数据提取器”**，既重复又容易出错。

2. **“SQL与Java代码的紧密耦合”**：SQL语句通常硬编码在Java代码中。一旦SQL语句需要调整，或者数据库表结构发生变化，就必须修改Java代码，然后重新编译和部署。这就像你的业务逻辑代码和数据库查询逻&#x8F91;**“混为一谈”**，难以独立维护。

```java
public User findUserById(int id) {
    Connection conn = null;
    PreparedStatement pstmt = null;
    ResultSet rs = null;
    User user = null;
    try {
        conn = DriverManager.getConnection("jdbc:mysql://localhost:3306/test", "root", "password");
        String sql = "SELECT id, name, email FROM users WHERE id = ?";
        pstmt = conn.prepareStatement(sql);
        pstmt.setInt(1, id); // 手动设置参数
        rs = pstmt.executeQuery();
        if (rs.next()) {
            user = new User();
            user.setId(rs.getInt("id")); // 手动从ResultSet取值并映射
            user.setName(rs.getString("name"));
            user.setEmail(rs.getString("email"));
        }
    } catch (SQLException e) {
        e.printStackTrace();
    } finally {
        // 大量的finally块来确保资源关闭
        try { if (rs != null) rs.close(); } catch (SQLException e) { /* ignore */ }
        try { if (pstmt != null) pstmt.close(); } catch (SQLException e) { /* ignore */ }
        try { if (conn != null) conn.close(); } catch (SQLException e) { /* ignore */ }
    }
    return user;
}

```

* **“手动对象-关系映射的枯燥”**：从ResultSet中获取数据并手动将其属性逐一赋值给Java对象，或者将Java对象的属性逐一设置到PreparedStatement的参数中，这个过程非常枯燥且容易出错。这就像你从数据库获取了一堆“原始数据流”，需要**手动将其“塑形”成你的Java对象结构**。

## 解决方案

1. **“SQL外部化与集中管理”**

   * MyBatis 允许你将 SQL 语句定义在独立的 XML 映射文件（或通过注解）中，而不是直接写在 Java 代码里。

   * 这使得 SQL 语句与 Java 代码**解耦**。SQL的修改不再需要触及 Java 代码，提高了可维护性和灵活性。数据库管理员或SQL优化人员可以直接调整SQL，而无需了解Java代码细节。

2. **Mapper 接口与 SQL 映射**

   * MyBatis 的核心是 **Mapper 接口**。你只需定义一个简单的 Java 接口，其中包含你希望执行的数据库操作方法（如`findById(int id)`、`insertUser(User user)`）。MyBatis 会根据这个接口的方法名，自动找到XML文件中对应的 SQL 语句并执行。

```java
public interface UserMapper {
    User findById(int id); // 只需要定义方法签名
}
```

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper
        PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.example.mapper.UserMapper"> <!-- 对应UserMapper接口 -->
    <select id="findById" resultType="com.example.model.User"> <!-- 对应findById方法，指定返回类型 -->
        SELECT id, name, email FROM users WHERE id = #{id} <!-- SQL语句，#{id}表示参数 -->
    </select>
</mapper>
```

* **自动参数绑定与结果映射**

  * 它能够自动将 Java 方法的参数绑定到 SQL 语句的占位符上，并自动将 SQL 查询返回的 ResultSet 数据，根据你定义的映射规则（通常是基于Java Bean的属性名与数据库列名的对应），自动填充到你的 Java 对象中。

    * e.g. 数据库列account\_name映射到Java属性accountName

## 什么是ORM
**对象关系映射** Object-Relational Mapping，用于在**面向对象编程语言**和**关系型数据库**之间建立一种映射关系。

**核心目标**：将应用程序中的“对象模型”（Object Model）与数据库中的“关系模型”（Relational Model）进行自动化地相互转换和同步。

**为了解决**：对象-关系阻抗失配（Object-Relational Impedance Mismatch）问题
1. **数据模型差异**：如何将一个具有复杂继承关系、包含其他对象的 Java 对象，映射到扁平的、由基本类型组成的数据库表中？
2. **操作方式差异**：如何在 Java 代码中直接操作对象，而让框架自动生成并执行 SQL？
3. **数据类型差异**：如何自动地在 Java 类型和数据库类型之间进行转换？
4.  **身份标识差异**：如何同步对象在内存中的状态和数据库中的持久化状态？

**如何解决/核心机制：**
1. **映射定义**：ORM 允许你定义**映射规则**，告诉框架你的 Java 对象（类、属性）如何对应数据库的表格（表、列）。
	- 通常使用XML、注解、约定优于配置等，来定义的
2. **运行时数据转换与 SQL 生成**：通过 ORM 框架操作 Java 对象时（如保存、更新、查询），框架会根据映射元数据，**在运行时自动生成对应的 SQL 语句**。它还会自动将 Java 对象的属性值转换为 SQL 参数，并将数据库查询结果自动映射回 Java 对象。
3. **持久化上下文 (Persistence Context)**：- 许多 ORM 框架（特别是全功能 ORM）会维护一个“持久化上下文”或“一级缓存”。它是一个内存区域，用于管理当前会话中加载和修改的实体对象。框架会跟踪对象状态的变化，并在适当的时候（如事务提交时）将这些变化同步到数据库中，从而减少不必要的数据库操作。

## ORM 和 持久化框架区别
- **持久层框架 (Persistence Framework)** 是一个**更广泛的概念**。
- **ORM (Object-Relational Mapping)** 是一种**特定的技术或方法**，通常用于实现持久化，特别是针对关系型数据库。

MyBatis 是一个**持久层框架**，它执行对象关系映射（**ORM**），但它不是一个典型的、全功能的 ORM 框架（比如 Hibernate 或 JPA），它是“半ORM”。
<mark class="hltr-yellow">区别在于 **“抽象的程度”** 和  **“对 SQL 的控制权”** 。</mark>

**典型的ORM框架：**
- 高度抽象SQL，几乎不需要编写 SQL
- 自动生成SQL
- 不关注数据库表结构和SQL细节
**MyBatis** ：
更准确的定位是 **SQL Mapper（SQL 映射器）** 或 **“半 ORM”** 框架。
- 需要编写SQL，将 SQL 语句写在独立的 XML 文件中（或通过注解），并明确地与 Java 方法关联起来。
- **专注于 SQL 映射**：MyBatis 的主要职责是：
	1. **将 Java 方法的参数** 映射到 SQL 语句的占位符上。
	2. **将 SQL 查询结果集** 映射到 Java 对象（POJO）上。
	3. **管理 JDBC 的底层细节**（如连接的获取与释放、Statement 的创建与关闭等），将你从这些繁琐的样板代码中解放出来。
# SSM 框架

是以下三个框架的首字母缩写：
1. **S**pring (Spring Framework)
2. **S**pring MVC
3. **M**yBatis

SMM 框架是指将 Spring 核心、Spring MVC 和 MyBatis 这三个独立的、功能互补的框架集成在一起，用于构建企业级 Java Web 应用程序的一种常见技术解决方案。

## 为什么会出现 SSM 这种组合？它解决了什么问题？
为了解决在构建复杂 Web 应用时，如何
1. 高效地管理组件？
2. 处理Web请求？
3. 进行数据持久化？
4. 保持代码高内聚、低耦合？
在 SSM 流行之前，Java Web 开发可能面临：
- **组件管理混乱**：没有统一的机制来管理对象（Bean）的创建和依赖关系。
- **Web层职责不清**：Web请求处理逻辑与业务逻辑、数据访问逻辑混杂，导致代码难以维护。
- **数据访问层繁琐**：直接使用 JDBC 编写大量重复的数据库操作代码。
### 1. **Spring (核心)** ：作为 **“应用骨架与组件协调器”**

- **核心作用：** 提供**IoC控制反转/DI依赖注入** 容器，负责管理应用程序中所有组件（包括 Spring MVC 的控制器、业务逻辑服务、MyBatis 的数据访问对象等）的生命周期和依赖关系。
- **解决问题：** 解决了组件间的紧密耦合问题，使得各个模块可以独立开发和测试，提高了系统的可扩展性和可维护性。同时，其 **AOP（面向切面编程）** 能力可以统一处理事务管理、日志记录、权限控制等横向关注点。
- **感性认识**：Spring 就像整个应用程序的 **“中央神经系统”** ，它连接并协调着各个功能模块，确保它们能够顺畅地协同工作，并且能够统一调度一些全局性的“行为模式”（如事务）。
### 2. **Spring MVC**：作为 **“Web请求处理与视图渲染引擎”** 

- **核心作用**：基于 MVC 设计模式，专门处理 Web 请求。它提供 `DispatcherServlet` 作为前端控制器，负责请求的分发，将请求路由到正确的 `Controller`，然后 `Controller` 调用业务逻辑，并将数据模型传递给 `View` 进行渲染。
- **解决问题**：清晰地分离了 Web 应用的职责（请求处理、业务逻辑、界面展示），使得 Web 层代码结构清晰，易于理解和维护。
- **感性认识**：Spring MVC 就像应用程序的 **“Web交互界面管理器”** ，它接收来自用户的各种“指令”（HTTP请求），智能地解析这些指令，并协调内部的“业务处理单元”来响应，最终将处理结果以用户友好的方式“呈现”出来。

### 3. **MyBatis**：作为 **“SQL驱动的数据持久化层”**

- **核心作用**：负责应用程序与数据库之间的数据交互。它允许开发者编写 SQL 语句，并通过 XML 映射文件或注解将 SQL 语句与 Java 方法关联起来，自动完成参数绑定和结果集映射。
- **解决问题**：解决了直接使用 JDBC 时大量的样板代码问题，同时提供了对 SQL 语句的完全控制权（不像一些全自动 ORM 框架），这对于复杂的查询优化和数据库特定功能的使用非常有利。
- **感性认识**：MyBatis 就像应用程序的 **“智能数据存取代理”** 。你只需要告诉它“我需要执行这条 SQL 语句，并把结果映射成这个 Java 对象”，它就会自动帮你处理所有底层与数据库交互的繁琐细节，并直接给你返回处理好的数据对象。

## SSM 框架的工作流程（一个典型的 Web 请求）

让我们以一个用户请求查询商品信息的流程为例，看看 SSM 是如何协同工作的：

1. **用户请求**：用户在浏览器中输入 `http://yourdomain.com/products/123`。
2. **Spring MVC `DispatcherServlet` 拦截**：这个请求首先被 Spring MVC 的 `DispatcherServlet` 拦截。
    - **感性认识**：请求进入了 Web 应用程序的“入口路由器”。
3. **Spring MVC `HandlerMapping` 路由**：`DispatcherServlet` 根据配置（如 `@RequestMapping` 注解），找到负责处理 `/products/{id}` 请求的 `ProductController` 中的相应方法。
    - **感性认识**：路由器根据请求地址，将请求精确地转发给负责处理“商品”相关业务的“控制器”。
4. **Spring MVC `Controller` 处理**：`ProductController` 中的方法接收请求，解析 `id` 参数。
    - **感性认识**：商品控制器接收到“查询ID为123的商品”的指令。
    - **调用业务逻辑**：`ProductController` 会调用业务逻辑层（通常是一个 `ProductService`）。这里的 `ProductService` 是通过 **Spring IoC/DI** 注入到 `ProductController` 中的。
    - **感性认识**：控制器将查询指令传递给“商品业务处理服务”。
5. **Spring Service 层处理**：`ProductService` 接收到查询请求。
    - **调用数据访问层**：`ProductService` 会调用数据访问层（通常是一个 `ProductMapper` 接口，由 MyBatis 生成实现）。这里的 `ProductMapper` 也是通过 **Spring IoC/DI** 注入到 `ProductService` 中的。
    - **感性认识**：商品业务处理服务将数据查询指令传递给“商品数据存取代理”。
6. **MyBatis `Mapper` 接口与 XML 映射**：`ProductMapper` 接口中的 `findById(int id)` 方法被调用。MyBatis 会根据这个方法名，找到对应的 XML 映射文件中定义的 SQL 语句（如 `SELECT * FROM products WHERE id = #{id}`）。
    - **感性认识**：数据存取代理根据“查询商品ID”的指令，查阅其“SQL蓝图仓库”，找到对应的 SQL 语句。
7. **MyBatis 执行 SQL**：MyBatis 负责执行这条 SQL 语句，将 `id` 参数绑定到 SQL 中，并与数据库进行交互。
    - **感性认识**：数据存取代理执行 SQL 指令，与数据库进行实际的数据交换。
8. **MyBatis 结果映射**：数据库返回结果集，MyBatis 自动将结果集中的数据映射到 `Product` Java 对象中。
    - **感性认识**：数据存取代理将数据库返回的原始数据，智能地转换为应用程序能理解的“商品对象”。
9. **数据返回**：`Product` 对象从 MyBatis 返回给 `ProductService`，再从 `ProductService` 返回给 `ProductController`。
10. **Spring MVC 视图渲染**：`ProductController` 将 `Product` 对象放入 `Model` 中，并返回一个逻辑视图名（如 `"productDetail"`）。`DispatcherServlet` 接着通过 `ViewResolver` 找到实际的视图文件（如 `productDetail.jsp` 或 `productDetail.html`），并用 `Model` 中的数据渲染视图。
    - **感性认识**：控制器将“商品对象”交给“Web交互界面管理器”，由它选择合适的“展示模板”，将商品信息呈现给用户。
11. **响应返回**：最终生成的 HTML 页面作为 HTTP 响应返回给用户的浏览器。

## SSM 框架的特点和演进

- **分层清晰**：SSM 组合强制并鼓励了清晰的分层架构（表现层、业务逻辑层、持久层），这对于大型项目的团队协作和维护至关重要。
- **灵活性**：MyBatis 提供了对 SQL 的高度控制，这在某些需要复杂查询优化或特定数据库功能的场景下非常有用。
- **学习曲线**：相较于一些更高级的 ORM 框架（如 JPA/Hibernate），MyBatis 的学习曲线可能更平缓，因为它更接近原生的 SQL 操作。
- **历史地位**：在 Spring Boot 出现之前，SSM 是 Java Web 开发中最主流、最成熟的组合之一。
- <mark class="hltr-red"> **与 Spring Boot 的关系** </mark>：虽然 SSM 组合本身需要一些手动配置，但 **Spring Boot 的出现极大地简化了 SSM 的集成和配置**。现在，你只需要引入 `spring-boot-starter-web` 和 `mybatis-spring-boot-starter` 这样的依赖，Spring Boot 就会自动为你配置好 Spring MVC 和 MyBatis，使得 SSM 的搭建变得异常简单，几乎是开箱即用。因此，可以说 Spring Boot 是 SSM 组合的“加速器”和“简化器”。
